## **2. 核心业务总览**

平台整体业务由 9 条核心流程构成：

1. 用户浏览与下单流程；
2. 商家接单与履约流程；
3. 维修相册生成流程；
4. 独立历史案例上传流程；
5. 订单完成与用户评价流程；
6. 奖励发放流程；
7. 普通分享与推荐成交预留流程；
8. 公开案例发布流程；
9. SEO/GEO 页面生成流程。

整体闭环如下：

```
用户浏览服务/案例/门店
→ 选择服务或预约复杂维修
→ 商家接单
→ 用户到店
→ 商家维修
→ 系统默认维修相册
→ 商家选择性上传图片
→ 用户查看维修过程
→ 商家提交完工
→ 用户评价
→ 用户获得评价奖励
→ 用户选择是否授权公开
→ 系统脱敏
→ 平台审核
→ 生成公开案例 H5
→ 案例页带来新的浏览、咨询、预约和订单

```

# **3. 用户下单流程**

## **3.1 适用服务类型**

用户下单分为两种类型：

### **类型一：标准服务在线支付**

适用于：

- 小保养；
- 电瓶更换；
- 轮胎服务；
- 部分明确套餐的刹车片/刹车盘；
- 部分标准检测或基础服务。

特点：

- 价格明确；
- 可在线支付；
- 履约边界清晰；
- 适合低门槛首次体验。

---

### **类型二：复杂维修预约到店报价**

适用于：

- 刹车系统复杂维修；
- 空调维修；
- 底盘维修；
- 钣喷修复；
- 事故车维修；
- 事故车专修；
- 新能源专项；
- 疑难故障维修。

特点：

- 不做线上最终报价；
- 平台展示价格区间和类似案例；
- 用户先预约；
- 到店检测后由商家报价；
- 平台可记录最终维修项目和价格区间。

---

## **3.2 标准服务在线支付流程**

```
用户进入小程序
→ 浏览首页/服务分类/案例
→ 选择服务项目
→ 查看服务方案详情
→ 查看可预约门店
→ 查看门店案例和评价
→ 选择门店
→ 添加或选择车辆
→ 选择预约时间
→ 确认服务套餐和价格
→ 提交订单
→ 微信支付
→ 支付成功
→ 订单进入待商家接单状态

```

---

## **3.3 复杂维修预约流程**

```
用户进入小程序
→ 浏览事故车/钣喷/底盘/空调/新能源等服务
→ 查看类似维修案例
→ 查看价格区间和影响因素
→ 选择门店
→ 填写问题描述
→ 上传车辆照片，可选
→ 选择预约时间
→ 提交预约单
→ 商家接单
→ 用户到店检测
→ 商家线下报价
→ 用户确认是否维修
→ 商家在订单中记录实际维修项目和价格区间

```

---

## **3.4 用户下单状态**


| 状态    | 说明                 |
| ----- | ------------------ |
| 待支付   | 用户提交标准服务订单但未支付     |
| 待商家接单 | 用户已支付或已提交预约，等待商家处理 |
| 已接单   | 商家确认接单             |
| 待到店   | 用户尚未到店             |
| 已到店   | 商家标记用户到店           |
| 维修中   | 商家开始维修             |
| 待用户确认 | 商家提交完工，等待用户确认      |
| 已完成   | 用户确认或系统自动完成        |
| 售后中   | 用户发起售后             |
| 已关闭   | 超时、取消、拒单等导致关闭      |
| 已退款   | 订单完成退款             |

# **4. 商家接单流程**

## **4.1 商家接单入口**

商家通过小程序中的“商家工作台”进入。

入口路径：

```
我的
→ 商家工作台
→ 今日订单/待接单

```

---

## **4.2 标准订单接单流程**

```
商家收到订单提醒
→ 查看订单详情
→ 查看服务项目
→ 查看预约时间
→ 查看车辆信息
→ 判断是否可履约
→ 接单/拒单

```

### **商家接单后：**

```
接单成功
→ 订单进入待到店
→ 用户收到接单通知
→ 系统默认创建维修相册

```

### **商家拒单后：**

```
商家拒单
→ 填写拒单原因
→ 用户收到通知
→ 系统发起退款或引导用户选择其他门店

```

---

## **4.3 复杂维修预约接单流程**

```
商家收到预约单
→ 查看用户描述和图片
→ 判断是否可接待
→ 接单/拒单/建议改期
→ 接单后联系用户确认到店时间
→ 用户到店检测
→ 商家给出线下报价
→ 用户确认后开始维修

```

---

## **4.4 商家履约流程**

```
用户到店
→ 商家标记已到店
→ 商家开始维修
→ 订单进入维修中
→ 商家可上传维修相册
→ 商家填写维修说明
→ 商家提交完工
→ 用户确认完成
→ 用户评价

```

---

## **4.5 商家可执行操作**


| 操作       | 说明             |
| -------- | -------------- |
| 接单       | 确认可履约          |
| 拒单       | 无法履约时拒绝，需要填写原因 |
| 改约       | 与用户协商更改预约时间    |
| 标记到店     | 用户实际到店后操作      |
| 开始维修     | 进入维修中          |
| 上传图片     | 上传维修相册图片       |
| 编辑说明     | 填写维修过程、配件、建议   |
| 提交完工     | 告知用户维修完成       |
| 发起公开案例申请 | 引导用户授权公开       |
| 处理售后     | 对用户售后进行响应      |


---

# **5. 维修相册生成流程**

## **5.1 相册创建原则**

平台订单一旦生成，系统默认创建维修相册。

但需明确：

> 相册默认创建，不强制上传。  
> 节点为建议，不强制完成。  
> 商家可以跳过任意节点。  
> 案例质量和曝光排序由完整度、清晰度、用户授权、评价等综合决定。

---

## **5.2 订单相册流程**

```
订单创建
→ 系统默认创建空相册
→ 根据服务类型生成建议节点
→ 商家可上传图片
→ 用户可实时查看
→ 商家提交完工
→ 系统生成案例草稿
→ 用户决定是否公开

```

---

## **5.3 用户查看规则**


| 场景          | 图片版本     |
| ----------- | -------- |
| 用户查看自己的订单相册 | 原始上传图    |
| 用户分享自己的维修档案 | 脱敏图      |
| H5 公开案例页    | 脱敏图      |
| 门店公开案例      | 脱敏图      |
| 平台审核查看      | 原图 + 脱敏图 |
| 商家查看本店订单相册  | 原图       |


---

## **5.4 相册为空时的处理**

如果商家未上传任何图片：

- 用户订单仍可正常完成；
- 维修相册显示为空状态；
- 可提示：“商家暂未上传维修过程图片”；
- 不生成高质量公开案例；
- 门店透明度评分不加分；
- 案例曝光权重较低或不可进入平台案例库。

---

# **6. 独立历史案例上传流程**

## **6.1 使用场景**

历史案例用于冷启动和门店专业展示。

适用场景：

1. 商家过往事故车维修案例；
2. 钣喷前后对比案例；
3. 底盘维修案例；
4. 刹车系统案例；
5. 新能源维修案例；
6. 门店擅长车型案例；
7. 门店想展示的代表性维修项目。

---

## **6.2 历史案例流程**

```
商家进入商家工作台
→ 点击历史案例
→ 新建案例
→ 选择服务类型
→ 选择复杂度等级，系统可辅助判断
→ 填写车型/里程/维修项目
→ 填写故障表现和维修方案
→ 上传图片，最多 20 张
→ 系统提示隐私保护责任
→ 商家确认已合法处理隐私
→ 系统自动压缩图片
→ 系统自动脱敏
→ 输出隐私风险指标
→ 高风险案例进入人工审核
→ 审核通过后发布

```

---

## **6.3 历史案例限制**

历史案例不得包含：

1. 未脱敏车牌；
2. VIN 码；
3. 车主姓名；
4. 手机号；
5. 身份证件；
6. 人脸；
7. 微信聊天记录；
8. 支付凭证中的个人信息；
9. 能够识别车主身份的特殊内容；
10. 稀有车型且存在可识别风险的图片。

---

## **6.4 历史案例标识**

所有历史案例必须展示来源标识：

> 商家历史案例，非平台订单案例。

同时可展示：

- 已脱敏；
- 已平台审核；
- 由商家提供；
- 价格仅供参考。

---

# **7. 订单完成与评价流程**

## **7.1 订单完成流程**

```
商家提交完工
→ 用户收到完工提醒
→ 用户查看维修相册
→ 用户确认完成
→ 订单进入已完成
→ 系统提醒用户评价

```

如果用户长时间不确认，可设置自动完成规则。

建议：

- 商家提交完工后 72 小时内用户未确认且未发起售后；
- 系统自动确认完成；
- 但自动完成后仍允许用户评价和申请售后。

---

## **7.2 用户评价流程**

```
订单完成
→ 用户进入评价页
→ 完成 6 维评分
→ 填写文字评价，可选
→ 上传体验图片，可选
→ 选择是否愿意公开评价，可选
→ 提交评价
→ 系统风控检测
→ 平台审核，必要时
→ 奖励进入待发放

```

---

## **7.3 6 维评价**


| 维度     | 含义                |
| ------ | ----------------- |
| 服务态度   | 商家接待、沟通、响应是否良好    |
| 专业程度   | 判断、维修、解释是否专业      |
| 价格透明   | 是否提前说明费用和变动原因     |
| 维修过程透明 | 是否有过程图片、配件说明、维修记录 |
| 完工效果   | 用户对维修结果是否满意       |
| 是否愿意推荐 | 用户是否愿意推荐给他人       |


---

# **8. 奖励发放流程**

## **8.1 奖励触发条件**

用户完成基础 6 维评价后，系统计算评价奖励。

奖励金额受以下因素影响：

1. 服务复杂度；
2. 车价；
3. 订单金额；
4. 是否有有效凭证；
5. 是否存在风控异常；
6. 对应等级奖励封顶。

---

## **8.2 奖励计算流程**

```
用户提交评价
→ 系统识别服务复杂度 L1-L4
→ 系统读取车辆估值或默认车价系数
→ 系统读取订单金额或价格区间
→ 计算奖励金额
→ 检查封顶金额
→ 风控检测
→ 进入待发放/待审核

```

---

## **8.3 奖励复杂度规则**


| 等级       | 基础奖励  | 单笔封顶   |
| -------- | ----- | ------ |
| L1       | 10 元  | 50 元   |
| L2       | 30 元  | 200 元  |
| L3       | 100 元 | 800 元  |
| L4       | 300 元 | 2000 元 |
| L4 保险事故车 | 300 元 | 3000 元 |


---

## **8.4 奖励发放状态**


| 状态  | 说明              |
| --- | --------------- |
| 待计算 | 用户刚提交评价         |
| 待审核 | 金额较高或存在风险       |
| 待发放 | 审核通过，等待发放       |
| 已发放 | 奖励已到账           |
| 已驳回 | 不符合奖励条件         |
| 已冻结 | 存在售后、退款、投诉或风控异常 |


---

# **9. 分享与推荐流程**

## **9.1 MVP 分享原则**

MVP 阶段仅做普通分享，不做浏览收益、点击收益、互动收益。

允许分享：

1. 公开案例；
2. 门店主页；
3. 服务方案；
4. 用户自己的脱敏维修档案。

不允许：

1. 按浏览量返现；
2. 按点击量返现；
3. 按分享次数返现；
4. 多级分销；
5. 拉人头奖励。

---

## **9.2 分享流程**

```
用户进入公开案例/维修档案
→ 点击分享
→ 系统确认使用脱敏图片
→ 生成分享卡片或链接
→ 分享给微信好友/群
→ 被分享人访问
→ 系统记录基础访问数据

```

---

## **9.3 推荐成交奖励预留**

后续可做单层推荐成交奖励：

```
用户 A 分享案例
→ 用户 B 通过分享进入
→ 用户 B 下单并完成维修
→ 系统识别 A 为直接推荐人
→ 订单完成且无售后争议
→ 平台审核
→ 发放推荐成交奖励

```

MVP 阶段只记录，不发放现金收益。

---

# **10. 公开案例发布流程**

## **10.1 平台订单案例发布流程**

```
商家提交完工
→ 系统生成案例草稿
→ 系统自动生成脱敏预览
→ 用户收到公开授权提醒
→ 用户查看脱敏效果
→ 用户可手动打码或裁剪
→ 用户明确同意公开
→ 系统自动脱敏和风险识别
→ 平台审核
→ 审核通过
→ 发布 H5 公开案例页

```

---

## **10.2 用户授权规则**

公开案例必须满足：

1. 用户明确同意；
2. 用户可查看脱敏预览；
3. 用户可选择拒绝；
4. 用户可申请下架；
5. 用户分享时只能分享脱敏版本；
6. 未授权不得公开平台订单案例。

---

## **10.3 审核结果**


| 审核结果       | 含义            |
| ---------- | ------------- |
| 通过         | 可公开展示         |
| 驳回         | 不允许公开         |
| 要求修改       | 需补充或重新脱敏      |
| 仅门店主页展示    | 不进入平台案例库      |
| 可进入平台案例库   | 可被平台推荐        |
| 可进入搜索引擎公开页 | 可生成 H5 SEO 页面 |
| 推荐为优质案例    | 增加曝光权重        |


---

# **11. SEO/GEO 页面生成流程**

## **11.1 页面类型**

MVP 阶段优先生成：

1. 公开案例详情页；
2. 门店公开主页；
3. 服务项目页基础版；
4. 城市服务页基础版；
5. sitemap；
6. Schema 结构化数据。

---

## **11.2 公开案例页面生成流程**

```
案例审核通过
→ 生成案例标题
→ 生成 AI 可引用摘要
→ 生成关键信息表
→ 生成维修过程内容
→ 生成 FAQ
→ 绑定门店信息
→ 绑定相关案例
→ 生成 URL slug
→ 输出 Schema
→ 加入 sitemap
→ 发布 H5 页面

```

---

## **11.3 AI 可引用摘要规则**

每个公开案例页顶部应有标准摘要：

```
该案例为【车型】在【里程】时进行【维修项目】的维修记录。
检查发现【主要问题】，门店采用【维修方案】完成处理。
案例包含维修前、拆检、配件、维修后等过程图片，已进行隐私脱敏。
价格展示为参考区间，实际费用会因车型、配件品牌、损伤程度和维修方案不同而变化。

```

---

## **11.4 Schema 预留**

页面需预留结构化数据：

- LocalBusiness；
- Service；
- Review；
- FAQPage；
- Article；
- HowTo；
- BreadcrumbList；
- ImageObject。

# **12. 异常流程**

## **12.1 用户未支付**

```
用户提交订单
→ 未支付
→ 订单进入待支付
→ 超过有效期自动关闭

```

建议未支付订单有效期：30 分钟。

---

## **12.2 商家拒单**

```
商家拒单
→ 用户收到通知
→ 标准支付订单自动退款
→ 用户可选择其他门店

```

---

## **12.3 用户取消**


| 订单阶段    | 处理规则        |
| ------- | ----------- |
| 未支付     | 直接关闭        |
| 已支付未接单  | 可取消并退款      |
| 已接单未到店  | 可申请取消，按规则退款 |
| 已到店/维修中 | 不支持直接取消，需协商 |
| 已完成     | 走售后流程       |


---

## **12.4 支付失败**

```
支付失败
→ 订单仍为待支付
→ 用户可重新支付
→ 超时关闭

```

---

## **12.5 支付回调异常**

```
微信支付成功
→ 平台未收到回调
→ 系统主动查询支付状态
→ 确认成功后更新订单
→ 记录异常日志

```

---

## **12.6 用户不同意公开**

```
用户查看脱敏预览
→ 选择不同意公开
→ 案例不生成公开页
→ 相册仅作为用户维修档案和订单履约记录存在

```
