# **《消息通知PRD》**

---

## **1. 文档信息**


| 项目     | 内容                                    |
| ------ | ------------------------------------- |
| 文档名称   | 消息通知PRD                               |
| 当前版本   | V1.0                                  |
| 适用端    | 用户端小程序、商家工作台、平台运营工作台、微信服务通知           |
| 核心对象   | 用户、商家、门店员工、平台运营人员                     |
| 核心目标   | 在订单、审核、退款、奖励、售后等关键节点及时通知相关角色，减少履约漏斗损失 |
| MVP 原则 | 优先使用微信服务通知和小程序内消息，不做短信和 App Push      |


---

# **2. 模块定位**

消息通知模块用于保障平台核心流程顺畅运行。

主要解决：

1. 用户不知道订单状态变化；
2. 商家错过接单和预约；
3. 平台审核任务积压；
4. 退款、售后、奖励状态不透明；
5. 案例公开、下架申请结果无法及时告知。

---

| 邮件通知    | 企业邮箱              | MVP 不做   |

# **9. 通知规则**

## **9.1 去重规则**

同一业务对象短时间内不重复发送相同通知。

示例：

```
同一订单 5 分钟内重复上传多张维修图片，只发送一次“维修进度更新”。

```

---

## **9.2 频率控制**


| 对象   | 限制               |
| ---- | ---------------- |
| 单用户  | 每日通知数量限制，运营配置    |
| 单订单  | 相同类型通知间隔限制       |
| 单商家  | 高频新订单不合并，其他通知可合并 |
| 平台运营 | 待办类可聚合展示         |


---

## **9.3 失败重试**

通知发送失败：

1. 记录失败原因；
2. 小程序内消息仍保留；
3. 微信服务通知失败可重试；
4. 超过重试次数后标记失败；
5. 不影响主业务流程。

---

# **10. 通知状态**


| 状态   | 说明           |
| ---- | ------------ |
| 待发送  | 已生成消息，未发送    |
| 已发送  | 已调用微信/站内消息成功 |
| 发送失败 | 触达失败         |
| 已读   | 用户打开或点击      |
| 已过期  | 超过有效期        |
| 已撤回  | 平台撤回，MVP 可不做 |

# **12. P0 验收标准**

必须满足：

1. 用户可接收订单状态通知；
2. 用户可接收支付和退款通知；
3. 用户可接收评价奖励通知；
4. 商家可收到新订单提醒；
5. 商家可收到审核结果通知；
6. 平台工作台可展示待办数量；
7. 小程序内消息中心可查看历史消息；
8. 微信服务通知可跳转到对应详情页；
9. 消息发送失败不影响主流程；
10. 所有通知需有发送记录；
11. 同类通知具备基础去重；
12. 用户敏感信息不得出现在通知标题中。
