## **3. 接口基础规范**

## **3.1 协议**

所有业务接口默认使用：

```
HTTPS + JSON

```

不允许生产环境使用 HTTP 明文传输。

---

## **3.2 URL 版本规范**

接口路径统一携带版本号：

```
/api/v1/模块/资源

```

示例：

```
GET    /api/v1/services
GET    /api/v1/services/{service_id}
POST   /api/v1/orders
GET    /api/v1/orders/{order_id}
POST   /api/v1/orders/{order_id}/pay

```

---

## **3.3 请求方法规范**


| 方法     | 使用场景         |
| ------ | ------------ |
| GET    | 查询列表、查询详情    |
| POST   | 创建资源、执行动作    |
| PUT    | 全量更新，MVP 可少用 |
| PATCH  | 局部更新         |
| DELETE | 删除或逻辑删除，需谨慎  |


原则：

- 查询用 GET；
- 创建用 POST；
- 状态流转动作优先用 POST；
- 审核、接单、取消、退款等动作不建议用 PUT，而用语义化动作接口。

示例：

```
POST /api/v1/orders/{order_id}/accept
POST /api/v1/orders/{order_id}/reject
POST /api/v1/orders/{order_id}/cancel
POST /api/v1/refunds/{refund_id}/approve
POST /api/v1/cases/{case_id}/audit

```

# **4. 统一请求头**

所有需要鉴权的接口应携带：

```
Authorization: Bearer {token}
Content-Type: application/json
X-Request-Id: {request_id}
X-Client-Type: weapp_user / merchant / admin / h5
X-App-Version: 1.0.0

```

## **4.1 Header 字段说明**


| 字段            | 是否必填  | 说明                  |
| ------------- | ----- | ------------------- |
| Authorization | 是     | 登录态 token           |
| Content-Type  | 是     | 默认 application/json |
| X-Request-Id  | 建议必填  | 请求唯一 ID，用于链路追踪      |
| X-Client-Type | 是     | 客户端类型               |
| X-App-Version | 建议必填  | 小程序或前端版本            |
| X-Timestamp   | 回调类必填 | 时间戳                 |
| X-Signature   | 回调类必填 | 签名                  |


---

# **5. 统一响应结构**

所有接口响应统一格式：

```
{
  "code": 0,
  "message": "success",
  "data": {},
  "request_id": "req_202501010001"
}

```

## **5.1 字段说明**


| 字段         | 说明           |
| ---------- | ------------ |
| code       | 业务状态码，0 表示成功 |
| message    | 给开发和前端使用的提示  |
| data       | 业务数据         |
| request_id | 请求追踪 ID      |


---

## **5.2 成功响应示例**

```
{
  "code": 0,
  "message": "success",
  "data": {
    "order_id": "ord_123456",
    "order_status": "WAIT_PAY"
  },
  "request_id": "req_abc123"
}

```

---

## **5.3 失败响应示例**

```
{
  "code": 400101,
  "message": "订单不存在",
  "data": null,
  "request_id": "req_abc123"
}

```

# **6. 错误码规范**

## **6.1 错误码分段**


| 错误码范围         | 类型         |
| ------------- | ---------- |
| 0             | 成功         |
| 100000-199999 | 通用错误       |
| 200000-299999 | 用户与登录错误    |
| 300000-399999 | 商家与门店错误    |
| 400000-499999 | 订单交易错误     |
| 500000-599999 | 支付退款错误     |
| 600000-699999 | 相册与上传错误    |
| 700000-799999 | 案例与内容错误    |
| 800000-899999 | 审核、风控、权限错误 |
| 900000-999999 | 第三方服务错误    |


---

## **6.2 通用错误码**


| code   | message | 说明            |
| ------ | ------- | ------------- |
| 0      | success | 成功            |
| 100001 | 参数错误    | 请求参数缺失或格式错误   |
| 100002 | 未授权     | 未登录或 token 失效 |
| 100003 | 无权限     | 当前角色无操作权限     |
| 100004 | 资源不存在   | 查询对象不存在       |
| 100005 | 请求过于频繁  | 触发限流          |
| 100006 | 系统繁忙    | 服务异常或降级       |
| 100007 | 幂等冲突    | 重复请求或状态不允许    |

# **7. 分页规范**

列表接口统一使用：

```
page
page_size

```

请求示例：

```
GET /api/v1/orders?page=1&page_size=20

```

响应示例：

```
{
  "code": 0,
  "message": "success",
  "data": {
    "list": [],
    "pagination": {
      "page": 1,
      "page_size": 20,
      "total": 120,
      "has_more": true
    }
  }
}

```

## **7.1 分页要求**


| 项目           | 规则              |
| ------------ | --------------- |
| 默认 page      | 1               |
| 默认 page_size | 20              |
| 最大 page_size | 100             |
| 排序           | 默认按创建时间倒序       |
| 大数据列表        | 后续可升级 cursor 分页 |

# **8. 鉴权规范**

## **8.1 登录态分类**


| 端     | 登录方式        | Token 类型           |
| ----- | ----------- | ------------------ |
| 用户小程序 | 微信登录        | user_token         |
| 商家端   | 手机号/账号/微信绑定 | merchant_token     |
| 运营后台  | 平台账号密码/企业微信 | admin_token        |
| H5公开站 | 默认匿名访问      | public             |
| 支付回调  | 微信签名验证      | callback_signature |


---

## **8.2 权限校验原则**

接口应同时校验：

1. 是否登录；
2. 当前角色是否允许访问；
3. 是否拥有资源权限；
4. 当前资源状态是否允许操作。

示例：

```
商家只能查看自己门店的订单；
用户只能查看自己的订单和相册；
运营人员只能访问其权限范围内的审核任务；
未授权公开的案例不能被 H5 API 返回。

```

---

# **9. 幂等规范**

以下接口必须支持幂等：


| 接口类型 | 幂等依据                   |
| ---- | ---------------------- |
| 创建订单 | client_request_id      |
| 发起支付 | order_id + pay_scene   |
| 支付回调 | transaction_id         |
| 发起退款 | refund_request_id      |
| 退款回调 | refund_id              |
| 商家接单 | order_id + action      |
| 提交完工 | order_id + action      |
| 提交评价 | order_id + user_id     |
| 领取奖励 | reward_id              |
| 审核操作 | audit_task_id + action |


## **9.1 幂等字段**

建议客户端请求重要动作时携带：

```
{
  "client_request_id": "cli_202501010001"
}

```

后端应记录：

```
client_request_id + user_id + action_type

```

避免重复创建、重复支付、重复退款、重复奖励。

---

# **10. 状态流转接口规范**

所有状态流转接口必须校验当前状态。

示例：

```
订单只有 WAIT_ACCEPT 状态才能接单；
订单只有 ACCEPTED 或 IN_SERVICE 状态才能提交完工；
订单只有 WAIT_CONFIRM 状态才能用户确认完成；
已取消订单不能再次支付；
已退款订单不能发放奖励。

```

错误示例：

```
{
  "code": 400201,
  "message": "当前订单状态不允许接单",
  "data": {
    "current_status": "CANCELLED"
  }
}

```

# **11. 回调接口规范**

## **11.1 支付回调**

支付回调必须满足：

1. 校验微信支付签名；
2. 校验订单号；
3. 校验支付金额；
4. 防重复处理；
5. 更新支付单状态；
6. 更新订单状态；
7. 写入支付日志；
8. 触发消息通知；
9. 异常进入对账或人工处理。

---

## **11.2 退款回调**

退款回调必须满足：

1. 校验签名；
2. 校验退款单号；
3. 校验退款金额；
4. 防重复处理；
5. 更新退款单状态；
6. 必要时更新订单售后状态；
7. 冻结或撤销关联奖励；
8. 写入退款日志。

---

# **12. 文件上传接口规范**

上传流程建议采用：

```
客户端申请上传凭证
→ 客户端直传对象存储
→ 回调或客户端确认上传完成
→ 后端生成素材记录
→ 进入相册节点或脱敏流程

```

## **12.1 上传限制**


| 类型     | 限制              |
| ------ | --------------- |
| 图片格式   | jpg/png/webp    |
| 视频格式   | mp4/mov         |
| 单张图片大小 | 建议不超过 10MB      |
| 单个视频大小 | MVP 建议不超过 200MB |
| 图片数量   | 按相册节点限制         |
| 视频数量   | MVP 可限制每节点 1 个  |

# **13. 安全规范**

接口必须遵守：

1. 不返回用户手机号明文，除非有明确业务需要；
2. 不返回车牌、VIN 明文到 H5；
3. 后台接口按角色权限过滤；
4. 支付、退款接口必须记录操作日志；
5. 审核、冻结、驳回等高风险操作必须记录操作人；
6. H5 接口只返回已公开、已脱敏、允许展示的内容；
7. 防止越权访问订单、相册、案例；
8. 上传接口防止恶意文件类型；
9. 所有请求记录 request_id；
10. 异常接口不得直接暴露数据库错误。

---

# **14. P0 验收标准**

必须满足：

1. 所有接口使用统一响应结构；
2. 所有鉴权接口必须校验 token；
3. 用户、商家、平台权限隔离；
4. 订单、支付、退款、奖励类接口支持幂等；
5. 支付和退款回调支持签名校验；
6. 上传接口支持素材归属校验；
7. H5 接口不返回未授权、未脱敏内容；
8. 后台审核接口必须记录操作日志；
9. 所有核心接口可通过 request_id 追踪；
10. 错误码有明确归类；
11. 分页接口结构统一；
12. 生产环境仅允许 HTTPS。
