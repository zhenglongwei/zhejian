## **2. 总体策略**

系统可以先单体，后模块化，再逐步微服务化。

早期建议：

```
一个主业务后端
一个媒体处理服务
一个 H5内容生成服务
一个后台管理系统

```

后期再拆微服务。

## **9. 订单状态机**

```
CREATED             已创建
WAIT_PAY            待支付
PAID                已支付
WAIT_ACCEPT         待商家接单
ACCEPTED            商家已接单
ARRIVED             用户已到店
INSPECTING          检查中
REPAIRING           维修中
WAIT_CONFIRM        待确认完工
COMPLETED           已完成
CANCELLED           已取消
REFUNDING           退款中
REFUNDED            已退款
AFTER_SALE          售后中
CLOSED              已关闭

```

状态流转：

```
CREATED
→ WAIT_PAY
→ PAID
→ WAIT_ACCEPT
→ ACCEPTED
→ ARRIVED
→ INSPECTING
→ REPAIRING
→ WAIT_CONFIRM
→ COMPLETED

```

异常流转：

```
WAIT_PAY → CANCELLED
PAID → REFUNDING → REFUNDED
COMPLETED → AFTER_SALE

```

## **14. 幂等设计**

以下接口必须支持幂等：

1. 创建支付单；
2. 支付回调；
3. 退款申请；
4. 退款回调；
5. 上传回调；
6. 订单状态流转；
7. 完工确认；
8. 案例发布；
9. 奖励发放；
10. 消息发送。

幂等键建议：

```
业务类型 + 外部单号 + 回调流水号

```

示例：

```
PAY_CALLBACK:wx_transaction_id
REFUND_CALLBACK:wx_refund_id
UPLOAD_CALLBACK:object_key

```

## **15. 事务边界**

强事务：

1. 创建订单；
2. 支付成功更新支付单和订单；
3. 退款成功更新退款单和订单；
4. 用户评价提交；
5. 商家完工确认。

最终一致：

1. 消息通知；
2. 搜索索引；
3. sitemap 更新；
4. 案例生成；
5. 图片处理；
6. 评价统计；
7. 奖励收益统计。

## **17. 安全设计**

1. 所有接口 HTTPS；
2. 用户端、商家端、后台 Token 隔离；
3. 后台接口必须 RBAC 校验；
4. 支付回调必须验签；
5. 上传回调必须验签；
6. 敏感字段加密或脱敏；
7. 防止重复提交；
8. API 限流；
9. 文件类型校验；
10. 操作日志不可篡改；
11. 密钥使用环境变量或密钥管理系统；
12. SQL 参数化查询；
13. 防止越权访问订单、门店和用户数据。
