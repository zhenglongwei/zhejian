## **2. 验收目标**

本标准用于判断平台是否具备上线条件，重点确认：

1. V2.0 核心业务链路完整（浏览、咨询、相册、案例、GEO）；
2. 咨询线索与服务相册链路正确；
3. 服务相册上传、脱敏、公开授权与案例链路可控；
4. **Phase 1**：服务相册六阶段记录与公开授权留痕可验证；**Phase 2**：配件确认留痕（L2+）可验证（见 §5.0c）；
5. 运营审核能力可用；
6. 用户隐私和权限安全达标；
7. H5 符合公开展示与 SEO 基础要求；
8. 测试、部署、回滚、监控准备完成。

---

## **3. 验收结论类型**


| 结论    | 说明                 |
| ----- | ------------------ |
| 通过    | 满足上线标准，可发布         |
| 有条件通过 | 存在非阻塞问题，明确责任人和修复时间 |
| 不通过   | 存在P0/P1阻塞问题，不允许上线  |

## **4. 缺陷准出标准**

上线前缺陷必须满足：


| 缺陷等级 | 准出要求                    |
| ---- | ----------------------- |
| P0   | 必须为 0                   |
| P1   | 必须为 0，特殊情况需产品、技术、业务共同确认 |
| P2   | 可存在，但需记录延期计划            |
| P3   | 可存在，不阻塞上线               |


以下问题一律视为 P0 阻塞：

1. 咨询线索状态错乱或越权访问；
2. 服务相册权限隔离失败（用户可查看他人相册）；
3. 后台权限越权；
4. H5 泄露车牌、VIN、手机号、人脸；
5. 未授权案例被公开；
6. 原图可被公网访问；
7. 无法回滚。

**Phase 2 额外 P0 阻塞**（启用配件车主确认后适用，见 §5.0c）：L2+ 存在待确认配件时商家仍可标记完工。

> **口径**：Phase 1 产品定义见 `docs/04_维修过程相册/00_Phase1_服务相册产品口径.md`；Phase 1 **不要求**配件/方案车主确认。

**以下 V1.0 项不纳入 V2.0 验收**：支付失败、订单状态错乱、退款金额错误等。

## **5. 业务链路验收标准**

## **5.0 咨询与预约链路**（V2.0 P0）

必须满足：

1. 用户可完成微信登录；
2. 用户可浏览服务、门店、案例；
3. 用户可提交咨询/预约（含车辆、联系方式、意向门店）；
4. 咨询记录仅本人可见；
5. 商家可查看本门店线索并更新状态；
6. 商家不可查看其他门店线索；
7. 关闭/暂停门店不可提交新咨询（或明确提示）；
8. 咨询提交有防重复与必填校验。

---

## **5.0b 服务相册链路**（辙见 · Phase 1 P0）

必须满足：

1. 商家可独立创建 **服务相册**（不依赖平台订单；门店 + 服务 + 可选手机号）；
2. 六阶段节点可上传、保存与查看；
3. 商家可录入配件标准类型与价格（**不要求**发起车主配件确认、**不阻断**完工）；
4. 用户仅可查看与自己关联的服务相册；
5. 用户端 **无**「待确认配件/方案」主路径（`PendingConfirmList` 等非主路径引用，见组件 API §3.5）；
6. 相册 `completed` 后，用户可发起/撤回 **公开授权**（留痕 + 脱敏预览）；
7. 未授权相册不可生成 H5 公开页；
8. 脱敏与展示审核通过后方可公开；
9. 公开案例 Tag 符合 Phase 1：**已授权 · 已脱敏 · 已审核**（不得出现「平台订单案例」）。

---

## **5.0c 配件车主确认链路**（Phase 2 · **非 Phase 1 验收**）

<details>
<summary>Phase 2 启用后再验收（当前勿按 P0 执行）</summary>

必须满足：

1. L2+ 非原厂/待定配件须车主确认后才可标记完工；
2. 用户端待确认配件/方案入口可用，确认留痕完整；
3. 方案变更后原确认失效并须重新确认；
4. 商家不可代车主确认配件或方案。

</details>

## **5.1 用户下单支付链路**（V1.0 遗留 — V2.0 **不验收**）

<details>
<summary>归档</summary>

必须满足：

1. 用户可完成微信登录；
2. 用户可浏览服务；
3. 用户可查看服务详情；
4. 用户可创建订单；
5. 订单金额计算正确；
6. 用户可发起微信支付；
7. 支付成功后订单状态正确推进；
8. 支付取消后订单保持待支付；
9. 支付超时后订单可关闭或重新支付；
10. 用户只能查看自己的订单。

</details>

---

## **5.2 商家接单履约链路**（V1.0 遗留 — V2.0 **不验收**）

<details>
<summary>归档</summary>

必须满足：

1. 审核通过的商家可登录；
2. 商家只能查看本门店订单；
3. 商家可接单；
4. 商家可拒单；
5. 拒单后触发取消和退款流程；
6. 商家可开始服务；
7. 商家可上传维修图片；
8. 商家可提交完工；
9. 履约节点状态正确；
10. 商家操作有日志。

</details>

---

## **5.3 订单完成与评价链路**（V1.0 遗留 — V2.0 **不验收**）

<details>
<summary>归档</summary>

必须满足：

1. 已完成订单可评价；
2. 未完成订单不可评价；
3. 已取消订单不可评价；
4. 全额退款订单不可评价或不发奖励；
5. 评价提交后进入审核；
6. 审核通过后可展示；
7. 审核驳回后不展示；
8. 奖励状态正确生成；
9. 退款中评价奖励冻结；
10. 全额退款后奖励撤销。

</details>

---

## **5.4 退款售后链路**（V1.0 遗留 — V2.0 **不验收**）

<details>
<summary>归档</summary>

必须满足：

1. 已支付订单可按规则退款；
2. 未支付订单取消不退款；
3. 商家拒单可触发退款；
4. 用户取消已支付未接单订单可退款；
5. 后台可审核退款；
6. 退款金额不得超过可退金额；
7. 支持全额退款；
8. 支持部分退款，若MVP开启；
9. 退款回调可正确更新状态；
10. 重复退款回调幂等。

</details>

## **6. 相册与公开案例验收标准**

## **6.1 上传验收**

必须满足：

1. 商家可上传图片；
2. 图片绑定**服务相册**或案例；
3. 上传前校验身份；
4. 上传后校验归属；
5. 非法格式不可上传；
6. 超大文件不可上传；
7. 上传失败有提示；
8. 上传记录可追踪。

---

## **6.2 脱敏验收**

必须满足：

1. 车牌可识别并脱敏；
2. 人脸可识别并脱敏；
3. VIN、手机号等敏感信息不可公开；
4. 脱敏失败进入人工处理；
5. 原图不得公开访问；
6. 公开页只能使用脱敏图。

---

## **6.3 公开案例验收**

必须满足：

1. 案例发布前必须审核；
2. 未审核案例不可公开；
3. 审核驳回案例不可公开；
4. 用户未授权案例不可公开；
5. 用户撤回授权后案例下线；
6. 争议/未授权服务相册案例不可公开；
7. 后台可下线案例；
8. 下线后H5不再展示；
9. SEO字段可配置或生成；
10. 页面不泄露隐私。

---

## **7. H5与SEO验收标准**

H5公开页面必须满足：

1. 公开案例页可正常访问；
2. 门店页可正常访问；
3. 服务项目页可正常访问；
4. 移动端展示正常；
5. title 不为空；
6. description 不为空；
7. H1 唯一；
8. canonical 存在；
9. 图片 alt 不为空；
10. sitemap 只包含可收录页面；
11. 未发布页面不进入 sitemap；
12. 已下线页面不进入 sitemap；
13. 页面不展示敏感个人信息；
14. 页面状态码符合规范；
15. 页面访问埋点可采集。

---

## **8. 运营后台验收标准**

后台必须满足：

1. 支持账号登录；
2. 支持角色权限；
3. 支持商家审核；
4. 支持门店审核；
5. 支持服务审核；
6. 支持订单查询；
7. 支持退款审核；
8. 支持评价审核；
9. 支持公开案例审核；
10. 支持案例下线；
11. 支持敏感信息脱敏展示；
12. 支持操作日志；
13. 支持按条件筛选；
14. 支持异常数据查看；
15. 高危操作需二次确认或日志记录。

---

## **9. 接口与技术验收标准**

接口必须满足：

1. 所有核心接口需要鉴权；
2. 所有写接口需要参数校验；
3. 关键状态流转需校验当前状态；
4. 支付回调验签；
5. 退款回调验签；
6. 支付、退款、消息、上传支持幂等；
7. 接口返回统一结构；
8. 错误码明确；
9. 请求可通过 request_id 追踪；
10. 日志不输出敏感明文；
11. 数据库关键字段有约束；
12. 文件访问权限正确；
13. 定时任务可监控；
14. 异常有告警。

---

## **10. 安全与隐私验收标准**

必须满足：

1. 首次登录展示隐私协议；
2. 用户授权记录可追溯；
3. 公开案例授权独立记录；
4. 用户拒绝授权不得公开；
5. 用户撤回授权后可处理；
6. 手机号默认脱敏展示；
7. 车牌默认脱敏展示；
8. VIN不公开展示；
9. 原图私有存储；
10. H5只展示脱敏图；
11. 后台查看敏感信息有权限控制；
12. 后台查看明文有日志；
13. 数据导出需权限控制；
14. 埋点不得采集敏感明文；
15. 第三方服务用途在隐私协议中说明。

---

## **11. 性能验收标准**

MVP 阶段建议标准：


| 指标     | 验收标准                    |
| ------ | ----------------------- |
| 核心接口响应 | P95 小于 1000ms，特殊上传/支付除外 |
| 首页加载   | 主要内容 3s 内可见             |
| 服务详情页  | 3s 内可用                  |
| H5案例页  | 首屏 3s 内可见               |
| 后台列表页  | 常规查询 2s 内返回             |
| 图片上传   | 能显示进度或明确状态              |
| 支付回调处理 | 3s 内响应微信                |
| 退款回调处理 | 3s 内响应微信                |


---

## **12. 兼容性验收标准**

必须覆盖：

1. 微信小程序 iOS；
2. 微信小程序 Android；
3. 常见屏幕尺寸；
4. 微信内置浏览器H5；
5. iOS Safari，H5；
6. Android Chrome，H5；
7. 后台 Chrome 浏览器；
8. 弱网场景；
9. 页面刷新和返回；
10. 支付中断后恢复。

---

## **13. 数据验收标准**

必须满足：

1. 订单数据准确；
2. 支付数据准确；
3. 退款数据准确；
4. 评价与订单关联准确；
5. 奖励与评价、订单关联准确；
6. 相册与订单、案例关联准确；
7. 审核状态准确；
8. 用户授权记录准确；
9. 操作日志完整；
10. 埋点核心事件可查询。

---

## **14. 上线准备验收标准**

上线前必须确认：

1. 生产环境已部署；
2. 生产数据库已初始化；
3. 生产配置已确认；
4. 微信支付配置正确；
5. 小程序配置正确；
6. 对象存储配置正确；
7. 域名和证书可用；
8. 后台账号已创建；
9. 监控告警已配置；
10. 日志可查看；
11. 备份策略已启用；
12. 回滚方案已验证；
13. 客服和运营已培训；
14. 上线公告或通知准备完成。

---

## **15. 验收签署**

建议上线前由以下角色确认：


| 角色       | 验收内容           |
| -------- | -------------- |
| 产品负责人    | 功能范围、业务流程、体验   |
| 技术负责人    | 架构、接口、部署、回滚    |
| 测试负责人    | 测试结果、缺陷状态      |
| 运营负责人    | 后台审核、内容运营、客服流程 |
| 财务/结算负责人 | 支付、退款、结算规则     |
| 合规负责人，若有 | 隐私、授权、安全、内容合规  |
