## 核心产品模型：维修案例只有两种来源

消费者看到的案例/相册 **样式与内容结构统一**；差异仅体现在 **来源标识与合规说明**。

| 来源类型 | 说明 | 商家侧产生方式 | 用户端标识示例 |
|---|---|---|---|
| **平台成交案例** | 有平台订单号，履约过程中产生 | 订单详情 → 维修相册（随订单创建） | 平台订单案例 · 已脱敏 · 已审核 |
| **非平台成交案例** | 无平台订单，用于冷启动与内容沉淀 | 商家「新建相册 / 历史案例 / 独立创建」等入口（**同一类**） | 商家历史案例 · 已脱敏 · 价格仅供参考 |

说明：

- 文档中的「历史案例」「独立相册」「独立线下相册」在 **消费者呈现层属于同一类**（非平台成交），不要在 UI 上拆成多套样式。
- 「历史案例上传」不是独立于维修相册的另一条线，而是 **维修相册模块中「非平台创建」流程** 的冷启动用法。
- 开发时应先建设 **维修相册共用层**，再分叉两种创建入口，而不是先做互不相干的页面。

## 维修相册模块：分版本交付（勿一次做全）

| 版本 | 范围 | 阶段 |
|---|---|---|
| **相册 V1** | 共用数据模型、节点模板、图片上传、`AlbumNode`/`CaseCard`/案例详情；商家 **非平台创建** + 提交审核；用户 **案例列表/详情**（统一展示 + 来源 Tag） | V0.1 ✅ |
| **相册 V2** | 订单关联相册（自动/手动创建）、商家在订单中上传、用户 **订单内** 查看相册、`Timeline`/`PrivacyBanner`；**授权公开预览（pre-mask + 脱敏工作台）** | V0.5 ✅（**D8 + D9**） |
| **相册 V3** | 公开案例上架、H5 页、平台审核闭环展示 | V1.0 🟡（mock 授权→发布 ✅；H5 静态骨架 ✅；正式 SEO/联调 🔲） |

相册 V1 **不等于** 完整维修相册；订单链、用户授权公开业务等明确后置。

### 图片脱敏工具（独立能力，与相册版本正交）

> **主文档**：`docs/04_维修过程相册/08_图片脱敏工具PRD.md`  
> 商家端与车主端 **同一脱敏工作台**；**确认脱敏结果后** 才可提交审核/授权公开/公示。未来可独立部署官网工具。

| 版本 | 范围 | 阶段 |
|---|---|---|
| **脱敏 V1** | 任务模型 + API；`DesensitizeWorkbench`；一键脱敏；**商家历史案例** 必经确认闸门 | **V0.1+ ✅**（mock 引擎，见 §V0.1+） |
| **脱敏 V1.5** | **`order_pre_mask` 预脱敏任务**（统一 `desensitize_tasks`）；用户 **订单授权预览** `pages/desensitize/preview` | **V0.5 ✅**（D8；mock 引擎） |
| **脱敏 V2** | 手动打码；运营原图/脱敏对比；真实脱敏 API 联调 | V1.0 |
| **脱敏 V3** | 独立官网 / H5 开放工具页；导出 | V1.0+ |

V0.1 起 `desensitize-mock` 仅作 **URL 占位引擎**；业务须经 `services/desensitize.js` + 工作台确认闸门。联调后替换为真实脱敏服务 API。
