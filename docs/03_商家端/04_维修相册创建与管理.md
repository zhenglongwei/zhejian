## **2. 模块定位**

服务相册是平台 **P0 核心能力**，Phase 1 定位 **留档为主**：

1. 记录维修六阶段过程、配件类型、参考价与凭证图（**争议证据**）；
2. 可选关联车主，邀请查看私密留档（**只读，无需签字**）；
3. 车主 **授权公开** 后，脱敏沉淀案例与 GEO 内容（Phase 1 唯一车主主动确认环节）；
4. 提升门店透明度与咨询转化。

**Phase 2** 再启用：配件/方案车主确认、L2+ 阻断完工（见 `10_配件告知确认规则.md`）。

**配件验真链路（2026-06-24 定稿）**：阶段三报价表 OCR → `planParts` → 阶段四逐件上传 + 编码 OCR → 车主 **配件验真** 页对照（见 §13–§14、`07_维修相册查看页` §17.5）。

## **5. 六阶段编辑（Phase 1）**

按 `02_相册模板与节点规则.md` 分阶段上传：

1. 接车登记  
2. 检测诊断  
3. 维修方案与 **固定参考报价**（门店填写 `planAmount`；**上传详细报价表图**，OCR 生成 `planParts` 目录，见 §13）  
4. **配件告知**（**按 `planParts` 逐项**上传凭证图 + 编码 OCR；标准类型 + 价格 — **录入留档**，见 §14）  
5. **施工记录**（施工工单分槽 + **旧件留痕** + 通用过程图；见 §15）  
6. 完工交付  

**Phase 1 无阻断规则**：商家可在配件未「车主确认」时标记完工。  
**Phase 2**：L2+ 非原厂件未确认不可完工。

## **8. 邀请车主与公开案例**

### **8.1 邀请查看（私密）**

- 已填手机号：车主可在「我的 → 我的服务相册」查看；  
- 可分享小程序路径 / 码（P1）；  
- **仅查看留档**，不要求确认配件或方案。

### **8.2 授权公开（公开案例）**

- 须档案 `completed` 且至少 1 张有效图片；  
- **车主**在 user 端发起授权（勾选 + 留痕；Phase 2 电子签名）；  
- 脱敏工作台 → 平台展示审核 → 发布 H5 / 案例库；  
- Tag：**已授权 · 已脱敏 · 已审核**（按授权档位展示字段）。

商家 **不可** 代车主授权公开。

## **10. P0 验收标准（Phase 1）**

1. 商家可新建服务相册（门店 + 服务 + **车主扫码关联手机号**）；

## **13. 阶段三 · 方案报价表与 OCR 确认台（B-PLAN · 2026-06-24）**

> **真源字段**：`02_相册模板与节点规则.md` §3.5 · 接口见 `10/04_接口规范.md`（待 B-ALB-09 补全）。

### **13.1 商家上传**

| 步骤 | UI | 规则 |
| --- | --- | --- |
| 1 | 阶段三编辑页 | 填写方案说明 + **`planAmount`** |
| 2 | **上传报价表** | 1～N 张图（门店 DMS/手写报价单/系统截图） |
| 3 | 触发识别 | 异步 `plan_quote_ocr`：先 OCR，失败时多模态 LLM 兜底（手写报价表） |
| 4 | **确认台** | 表格预览：名称 / 类型 / 品牌 / 编码 / 数量 / 单价 |
| 5 | 锁定 | 「确认方案配件目录」→ `planPartsLockedAt` |

**确认台交互**

- 每行可 **编辑/删除/新增**（OCR 草稿不可直接对外展示）。  
- 页脚 Compliance：`识别结果仅供参考，请以你确认的报价表为准；确认后将锁定方案配件目录，供阶段四录入与车主验真参考。`  
- 锁定后：阶段四 **清单向导** 只读引用 `planParts`；修改方案目录须 **解锁** 并记录版本（P1：`planPartsRevision++`）。

### **13.2 OCR 失败降级**

- 无 OCR 结果：允许 **纯手工** 维护 `planParts` 表格（与确认台同 UI）。  
- L3/L4：可配置 **必须有报价表图** 才允许锁定（Phase 2 与阻断规则一并启用）。

### **13.3 与 `planAmount` 校验**

- OCR 行金额合计 ≠ `planAmount` → 黄条：「请核对报价表与方案总额是否一致」。  
- **不自动修改** `planAmount`。

## **14. 阶段四 · 配件清单向导与编码 OCR（B-PART · 2026-06-24）**

### **14.1 清单式向导（非自由表单）**

```text
方案配件目录（3 项）                    进度 2/3
┌────────────────────────────────────────────┐
│ ✓ 前刹车片 · 品牌件 · ×2                  │
│   编码 341168XXXX · 已上传 2 张            │
├────────────────────────────────────────────┤
│ ○ 后刹车片 · 品牌件 · ×2                  │
│   [ 上传包装/编码/实物图 ]                 │
│   识别编码：（可编辑）                      │
└────────────────────────────────────────────┘
施工中有增项 → 回阶段三上传「经客户确认的增项/报价单」并更新 planParts（见 11_施工增项留痕SOP）
```

| 项 | 规则 |
| --- | --- |
| 进度 | `已完成 / planParts.length` |
| 每行必做 | **标准类型** + **≥1 凭证图**（保存本项时校验） |
| 每行选填 | **品牌、编码** — 商家视配件重要程度填写；**机油/油液/冷却液与刹车片等同一表单**，不做「油液简化登记」 |
| 配件 OCR | 多图顺序识别编码候选 → 商家选择或单候选自动填入（可编辑） |
| 确认 | 单行「保存本项」→ 写入 `partsJson` |
| 增项 | Phase 1 **不立项** 独立表单；走 **`11_施工增项留痕SOP.md`**：阶段三增项单 → 更新 `planParts` → 阶段四登记。数据层 `source: extra` 保留供验真分组 |
| L3+ 建议 | 主要更换件 **建议** 填写编码或编码图（非 Phase 1 阻断） |

### **14.3 配件类型锁定与车主验真指引（定稿 · 2026-07-03）**

**验真目的（车主）**：根据编码自行查询，核对 **真实配件类型** 是否与相册登记一致；**不是** 车主对照方案与相册类型。

**商家阶段四**：

| 场景 | UI / 规则 |
| --- | --- |
| 报价识别 **含** `partType` | 类型 **只读**；点击修改 → 弹窗：与车主沟通 + 方案节点上传最新报价单 |
| 识别 **无** 类型 / 手工选择 | 类型可选；展示提示：须与报价单一致，变更须沟通车主并更新报价 |
| 区块顶部 | **车主验真指引**（复制编码 → 查询 → 对比登记类型） |

详见 `07_维修相册查看页` §17.5.4b · `constants/part-verify-copy.js`。

### **14.4 登记分级（定稿 · 不做 · 2026-06-25）**

不做「主要件 / 油液 / 辅料」商家可见分类；见 `04/02` §3.6.3 · `00_开发计划` §7.6.13。

### **14.5 商家端待办（P1）**

| 能力 | 时机 | 文档 |
| --- | --- | --- |

## **15. 阶段五 · 旧件留痕（定稿 · 2026-07-05）**

> **与阶段四分工**：阶段四 = **新件凭证**（逐行登记 + 编码 OCR）；阶段五 = **旧件/拆下留痕**（单独入口，不与通用过程图、施工工单混桶）。

### **15.1 商家上传**

| 项 | 规则 |
| --- | --- |
| 入口 | 阶段五编辑页 **「旧件留痕」** 独立区块（与 `album-evidence-slots` 施工工单槽、通用过程图 **并列**） |
| 数量 | 批量上传，建议 1～9 张（旧件单拍、新旧同框、拆下过程均可） |
| 关联方案行 | **可选**：每张图可下拉关联阶段四已登记配件（`planPartId`）；**不强制**，未关联的图仍有效 |
| 候选项 | 关联下拉 **自动** 来自本相册 `planParts` / 已保存 `parts`，商家无需在阶段四预操作 |
| Phase 1 | **不阻断** 完工；缺旧件留痕仅影响车主检查/AI 提示权重 |

**不做**

- 旧件 **不做** 第二套配件向导（不要求逐行名称/类型/编码 + 旧件图必填）  
- 阶段四 **不上传** 旧件图、不要求提前勾选「需旧件」

### **15.2 数据结构（待 B-EVID-06）**

`evidenceItems[]` 或等价字段，与单据分槽并列：

| 字段 | 说明 |
| --- | --- |
| `id` | `old_part_trace`（相册级一条或多条图组，实现可一张一项或一组一项） |
| `category` | `part` 或专用 `old_part` |
| `stageId` | `stage_5` |
| `images[]` | 旧件/拆下/同框图 URL |
| `planPartId` | **可选**；关联阶段四配件行 |
| `linkKey` | 与 `planPartId` 同义，验真/检查对齐用 |

通用 `nodes[stage_5].images` 仅保留 **非旧件、非工单** 的施工过程图；车主检查「旧件/过程留痕」改读本结构，**不再** 用整段 stage_5 过程图启发式。

### **15.3 车主端与 AI**

| 消费方 | 规则 |
| --- | --- |
| 完整性 Tab | 独立项「旧件留痕」✓/× + 缩略图 |
| 检查方法 | 有图即可说明如何对照；有 `planPartId` 时可写行级「换件项 ↔ 旧件图 ↔ 新件凭证 ↔ 工单」 |
| AI 分析 | 优先行级（三源均有 `planPartId`）；无关联时做工单/方案 vs 旧件图集 **弱匹配** |

### **15.4 实操提示（商家文案）**

> 更换类项目建议上传旧件或拆下照片；可多张合框。需要时可点选对应配件名称，便于车主核对与系统分析；不选也可保存。

油液、钣金、以修代换等 **不要求** 旧件实体图；过程/损伤图仍走通用过程上传。
