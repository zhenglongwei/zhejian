# **2. 模块定位**

订单履约模块是商家经营闭环的核心模块。

它承担：

1. 商家查看新订单；
2. 接单或拒单；
3. 与用户沟通预约时间；
4. 标记用户到店；
5. 开始维修；
6. 创建维修相册；
7. 上传维修过程图片；
8. 提交完工；
9. 发起用户确认；
10. 处理售后和异常。

---

# **3. 订单类型**

## **3.1 标准在线支付订单**

适用于：

- 一口价服务；
- 套餐价服务；
- 可明确报价的标准服务。

特点：

- 用户下单时已支付；
- 商家接单后进入待到店；
- 拒单需触发退款；
- 完工后进入用户确认或自动完成。

---

## **3.2 预约到店订单**

适用于：

- 价格区间服务；
- 到店检测后报价服务；
- 事故车维修；
- 复杂维修项目。

特点：

- 用户先提交预约；
- 可不在线支付最终维修费用；
- 商家可建议改期；
- 到店后确认维修方案和费用；
- 平台可记录最终成交信息，MVP 可先不强制支付闭环。

---

# **4. 订单状态流转**

## **4.1 主流程**

```
待接单
→ 已接单/待到店
→ 已到店
→ 维修中
→ 待完工
→ 已完工/待用户确认
→ 已完成

```

---

## **4.2 异常流程**

```
待接单
→ 已拒单
→ 已关闭/退款中/已退款

已接单
→ 用户取消
→ 已关闭/退款中/已退款

维修中
→ 异常处理中
→ 继续维修/取消/平台介入

已完工
→ 用户申请售后
→ 售后处理中
→ 售后完成

```

# **8. 接单与拒单**

## **8.1 接单**

商家点击“接单”后：

1. 系统校验门店状态；
2. 校验服务方案是否仍有效；
3. 校验员工权限；
4. 订单状态变为“已接单/待到店”；
5. 用户收到接单通知；
6. 商家可开始准备履约；
7. 系统生成相册草稿或允许创建相册。

---

## **8.2 拒单**

商家拒单必须填写原因。

拒单原因包括：

1. 当前时间无法接待；
2. 服务项目不匹配；
3. 用户车型不适用；
4. 门店临时停业；
5. 资质或能力不支持；
6. 其他原因。

拒单后：

- 在线支付订单进入退款流程；
- 预约订单关闭；
- 用户收到拒单原因；
- 平台记录拒单日志；
- 拒单率进入数据看板和运营监控。

---

# **9. 修改预约时间**

## **9.1 适用场景**

适用于：

1. 用户临时改期；
2. 门店接待能力不足；
3. 事故车需拖车安排；
4. 商家建议更合理的到店时间。

---

## **9.2 操作流程**

```
商家点击修改预约时间
→ 选择新日期和时间段
→ 填写改期原因
→ 提交给用户确认
→ 用户确认后生效

```

MVP 可简化为商家与用户沟通后直接修改，但需保留修改日志。

---

# **10. 到店与维修**

## **10.1 标记用户到店**

商家点击“标记到店”后：

- 订单状态变为“已到店”；
- 记录到店时间；
- 用户端同步显示；
- 可进入“开始维修”。

---

## **10.2 开始维修**

商家点击“开始维修”后：

- 订单状态变为“维修中”；
- 记录维修开始时间；
- 系统提示创建或完善维修相册；
- 根据服务项目生成建议相册节点。

---

# **11. 维修相册**

订单履约过程中，商家可创建维修相册。

入口：

1. 订单详情页；
2. 维修中订单卡片；
3. 待完工订单；
4. 工作台快捷入口。

相册来源：

- 订单自动关联相册；
- 商家手动创建相册；
- 独立相册后续关联订单，MVP 可选。

---

# **12. 完工确认**

## **12.1 商家提交完工**

商家提交完工时建议填写：


| 字段       | 是否必填     | 说明         |
| -------- | -------- | ---------- |
| 维修说明     | 建议必填     | 本次维修做了什么   |
| 配件说明     | 可选       | 更换配件、品牌、规格 |
| 价格说明     | 复杂维修建议必填 | 最终费用或区间说明  |
| 质保说明     | 可选       | 门店承诺或平台规则  |
| 完工图片     | 可选       | 可从相册选择     |
| 是否生成案例草稿 | 可选       | 用于内容沉淀     |


---

## **12.2 空相册提交**

允许商家不上传图片直接完工，但页面提示：

```
上传维修过程图片有助于用户理解维修过程，也能提升门店透明度和案例曝光。

```

---

## **12.3 发起用户确认**

商家提交完工后：

1. 订单进入“待用户确认”；
2. 用户收到完工通知；
3. 用户可确认完成；
4. 用户可发起异议；
5. 超过平台设定时间后可自动完成，MVP 可配置为 72 小时。

---

# **13. 用户确认与评价**

用户确认完成后：

- 订单状态变为“已完成”；
- 用户可评价；
- 评价进入评价管理；
- 订单可沉淀为真实订单案例；
- 数据看板统计完成订单。

---

# **14. 售后处理**

## **14.1 售后入口**

用户可在订单详情页发起售后。

商家可在：

```
商家工作台 → 订单管理 → 售后/异常

```

查看。

---

## **14.2 售后类型**


| 类型     | 说明          |
| ------ | ----------- |
| 未按约到店  | 用户或商家爽约     |
| 服务不符   | 实际服务与订单不一致  |
| 价格争议   | 到店报价或额外收费争议 |
| 维修质量问题 | 用户认为维修结果有问题 |
| 退款问题   | 支付订单退款争议    |
| 隐私问题   | 图片、案例涉及隐私   |
| 其他     | 平台人工介入      |


---

## **14.3 售后处理状态**

```
待处理
→ 商家处理中
→ 平台介入
→ 已解决
→ 已关闭

```

# **18. P0 验收标准**

必须满足：

1. 商家可查看新订单；
2. 商家可接单；
3. 商家可拒单并填写原因；
4. 商家可修改预约时间；
5. 商家可标记用户到店；
6. 商家可开始维修；
7. 商家可创建或进入维修相册；
8. 商家可提交完工；
9. 商家可发起用户确认；
10. 用户可确认完成；
11. 售后/异常订单可被识别；
12. 关键履约操作必须记录日志；
13. 员工权限受门店和角色限制。
