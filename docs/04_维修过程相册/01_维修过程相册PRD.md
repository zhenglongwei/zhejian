# **2. 产品定位**

维修过程相册不是普通图片上传功能，而是平台的核心基础设施。

它同时承担四个角色：

1. **用户维修档案**

让车主看到维修过程，保留车辆维修记录。
2. **商家专业展示工具**  
帮助商家把维修过程变成可信案例。
3. **平台内容生产工具**  
为公开案例、门店主页、服务页、车型页提供内容源。
4. **SEO/GEO/AI 数据基础**  
为搜索引擎和 AI 系统提供可理解、可引用、可结构化的汽修案例数据。

# **3. 设计原则**

## **3.1 默认创建，不强制上传**

平台订单生成后，系统自动创建一个空相册。

但：

- 不强制商家上传；
- 不强制完成节点；
- 不因相册为空阻断订单；
- 通过曝光、案例转化、门店透明度等机制鼓励商家上传。

---

## **3.2 节点建议，不做硬性校验**

系统根据服务类型生成建议节点。

商家可以：

- 上传全部节点；
- 上传部分节点；
- 跳过任意节点；
- 只上传完工图；
- 不上传。

节点完整度只影响：

1. 案例质量评分；
2. 门店透明度评分；
3. 公开案例推荐权重；
4. 用户信任感；
5. 后续搜索展示效果。

---

## **3.3 用户可实时查看**

商家上传图片后，用户在订单相册中可实时查看。

但：

- 用户查看的是自己的订单档案；
- 不等于同意公开；
- 未经用户明确授权，不得公开展示。

---

## **3.4 公开必须授权和脱敏**

平台订单相册转公开案例必须满足：

1. 用户明确同意；
2. 经 **统一图片脱敏工具**（`08_图片脱敏工具PRD.md`）完成：一键脱敏 → 预览 → 确认脱敏结果（与商家端同一工作台）；
3. 用户可手动打码/裁剪（脱敏工具 P1）；
4. 平台审核；
5. 公开页面只使用已确认脱敏图。

---

## **3.5 历史案例单独处理**

商家可上传历史案例，但必须：

1. 明确标记为商家历史案例；
2. 经 **统一图片脱敏工具** 完成脱敏预览与 **确认脱敏结果**（商家对隐私与合法性承担责任）；
3. 平台审核；
4. 高风险人工复核；
5. 支持投诉后快速下架。

# **4. 用户角色与使用场景**

## **4.1 车主用户**

使用场景：

1. 查看自己订单的维修过程；
2. 了解更换了什么配件；
3. 查看维修前后对比；
4. 保存维修档案；
5. 维修完成后评价；
6. 查看脱敏预览；
7. 决定是否公开案例；
8. 分享脱敏后的维修档案。

---

## **4.2 商家主账号/员工**

使用场景：

1. 订单履约中上传图片；
2. 给用户展示维修过程；
3. 生成公开案例草稿；
4. 上传历史案例；
5. 展示门店专业能力；
6. 提升门店主页内容质量；
7. 提升案例曝光和咨询转化。

---

## **4.3 平台审核人员**

使用场景：

1. 查看待公开案例；
2. 查看原图和脱敏图；
3. 检查隐私风险；
4. 检查内容真实性；
5. 审核是否可公开；
6. 审核是否可进入案例库；
7. 下架违规案例。

---

# **5. 相册类型**

## **5.1 订单相册**

由平台订单自动创建。

特点：

- 关联用户；
- 关联车辆；
- 关联门店；
- 关联服务项目；
- 用户可查看；
- 用户决定是否公开；
- 可信等级较高。

---

## **5.2 历史案例相册**

由商家主动创建。

特点：

- 不关联平台订单；
- 可关联门店；
- 可填写车型、服务、维修方案；
- 用于冷启动；
- 商家承担隐私合法性责任；
- 需平台审核；
- 可信等级低于平台订单案例。

---

## **5.3 独立线下相册，后续**

用于用户未通过平台下单但商家希望记录的线下订单。

MVP 可暂不完整支持，或归入历史案例流程。

后续可扩展：

- 商家创建线下维修记录；
- 用户扫码绑定；
- 用户授权后转为平台可验证案例。

---

# **7. 相册创建规则**

## **7.1 订单相册创建时机**

当订单创建成功时：

- 标准服务订单：提交订单后即创建；
- 支付订单：支付前可创建空相册，支付成功后正式有效；
- 复杂维修预约：预约提交后创建；
- 商家拒单或订单关闭：相册随订单关闭，不公开。

# **9. 上传规则**

## **9.1 图片数量**


| 项目       | 规则                |
| -------- | ----------------- |
| 单个相册图片上限 | 20 张              |
| 单次上传     | 建议最多 9 张          |
| 单张大小     | 建议前端压缩到 1-2MB 内   |
| 图片格式     | jpg/jpeg/png/webp |
| 视频       | MVP 不支持           |


---

## **9.2 图片压缩**

必须做压缩。

建议：

1. 小程序端先压缩；
2. 服务端二次压缩；
3. 公开展示图控制在合理尺寸；
4. 不影响用户查看关键细节；
5. 原图可在生成压缩图和脱敏图后删除。

---

## **9.3 原图保存策略**

考虑成本限制：

- 用户订单内可展示未公开原图或压缩原图；
- 平台可不长期保留高清原图；
- 原图生成压缩图后可删除；
- 若进入公开流程，必须另存脱敏图；
- 公开页面永远不使用原图。

建议保存：

1. 压缩原图；
2. 脱敏图；
3. 缩略图。

# **10. 脱敏规则**

## **10.1 必须识别和处理的敏感信息**

1. 车牌号；
2. VIN 码；
3. 发动机号；
4. 手机号；
5. 姓名；
6. 身份证号；
7. 人脸；
8. 微信号；
9. 地址；
10. 支付码；
11. 维修单据上的个人信息；
12. 保险定损单上的个人信息。

---

## **10.2 自动脱敏方式**

支持：

1. 识别车牌并打码；
2. 识别人脸并打码；
3. OCR 识别文字敏感信息；
4. 对疑似 VIN、手机号、姓名区域打码；
5. 对风险图片打标；
6. 输出风险等级。

---

## **10.3 风险等级**


| 风险等级 | 说明               | 处理            |
| ---- | ---------------- | ------------- |
| 低风险  | 未发现明显隐私          | 可进入审核         |
| 中风险  | 存在疑似隐私           | 需人工抽检或商家/用户确认 |
| 高风险  | 明显车牌、人脸、VIN、单据隐私 | 必须人工审核        |
| 禁止公开 | 无法有效脱敏或高度可识别     | 不允许公开         |


---

## **10.4 用户手动脱敏**

用户授权公开前，可提供：

1. 手动打码；
2. 裁剪；
3. 删除某张图片；
4. 选择不公开。

适用场景：

- 用户认为自动脱敏不足；
- 用户不希望某些维修细节公开；
- 图片包含特殊车辆外观或私人物品。

---

# **11. 用户授权公开流程**

## **11.1 授权触发时机**

推荐在商家提交完工后触发：

```
商家提交完工
→ 后台创建 **`order_pre_mask` 脱敏任务**（`desensitize_tasks`，不对用户展示 UI，见 03_订单相册流程 §6.4.1）
→ 系统生成案例草稿
→ 用户收到微信服务通知（可选）
→ 用户进入维修相册 → 确认公开
→ 基于 pre-mask 创建 **`order_authorize`** 任务，直接加载预脱敏预览（零等待）
→ 用户核对 / 手动打码（可选）→ 确认并公开

```

---

## **11.2 授权页内容**

授权页应明确说明：

1. 公开后会展示脱敏后的维修案例；
2. 不展示车牌、人脸、VIN、手机号等隐私；
3. 不默认展示实际成交价；
4. 仅展示价格区间和影响价格因素；
5. 用户可获得公开案例奖励或额外奖励，若配置；
6. 用户可以拒绝公开；
7. 用户后续可申请下架；
8. 用户可先预览脱敏效果。

---

## **11.3 授权选项**

用户可选择：

1. 同意公开；
2. 不同意公开；
3. 删除部分图片后公开；
4. 手动脱敏后公开；
5. 暂不处理。

---

## **11.4 授权记录**

必须记录：

1. 用户 ID；
2. 订单 ID；
3. 相册 ID；
4. 授权结果；
5. 授权时间；
6. 授权协议版本；
7. 授权时展示的脱敏预览版本；
8. 用户是否手动修改；
9. IP 和设备信息。

---

# **13. 历史案例上传流程**

## **13.1 商家操作流程**

```
商家工作台
→ 历史案例
→ 新建案例
→ 选择门店
→ 选择服务项目
→ 填写车型信息
→ 填写维修问题
→ 填写维修方案
→ 填写价格区间
→ 上传图片
→ 确认隐私责任
→ 提交审核

```

---

## **13.2 商家隐私责任确认文案**

建议文案：

```
请确认你上传的历史案例已取得必要授权，或已确保图片和文字内容不会暴露客户身份、车牌、VIN、手机号、人脸、支付信息等隐私信息。

平台会对图片进行自动脱敏和审核，但商家仍需对上传内容的合法性、真实性和不侵犯客户隐私承担责任。

如平台发现隐私风险、侵权投诉或内容不实，有权立即下架案例，并视情况限制门店案例发布权限。

```

---

## **13.3 历史案例审核要求**

冷启动阶段建议：

> 每个历史案例都人工审核。

审核重点：

1. 是否存在隐私泄露；
2. 是否为稀有车型且可识别车主；
3. 图片是否清晰；
4. 内容是否真实可信；
5. 维修描述是否夸大；
6. 是否适合公开；

# **18. 验收标准**

## **18.1 P0 功能验收**

必须满足：

1. 订单创建后自动生成相册；
2. 商家可上传最多 20 张图片；
3. 图片可压缩；
4. 用户可实时查看；
5. 用户授权前不公开；
6. 自动生成脱敏图；
7. 用户可查看脱敏预览；
8. 用户可同意或拒绝公开；
9. 平台可审核案例；
10. H5 公开页面只使用脱敏图；
11. 历史案例可上传；
12. 历史案例需商家确认隐私责任；
13. 冷启动历史案例可人工审核；
14. 用户分享使用脱敏图。

---

## **18.2 隐私验收**

必须满足：

1. 车牌可识别区域必须打码；
2. VIN 必须打码；
3. 人脸必须打码；
4. 手机号必须打码；
5. 单据个人信息必须打码；
6. 高风险图片不得自动公开；
7. 平台订单案例必须有用户授权记录；
8. 用户可申请下架；
9. 历史案例发现侵权风险可快速撤回。

---

## **18.3 业务验收**

MVP 上线后应能验证：

1. 商家是否愿意上传图片；
2. 用户是否查看相册；
3. 用户是否愿意公开；
4. 案例是否能形成门店资产；
5. 案例页是否能带来咨询；
6. 相册完整度是否影响转化。
