## 1. 一句话定位

**私域是信任账本（全量留存）；公域是合规知识产品（精选文案 + 脱敏关键帧）。**  
文案是**商家的服务内容**（如同电商商品详情），车主**不能改文案**，只能**发布评价**并决定是否上公开网站。  
AI：**域内**做分类/教练/选图候选；**外网大模型只吃已 scrub 的文字**；**图片永不进入外部大模型**。

## 2. 三层数据架构

```text
层级 A · 私域留档（车主 / 门店）
  六阶段全量图文 + 单据图 + planAmount 等
  仅鉴权可见；搜索引擎与 AI 爬虫不可抓取

层级 B · 候选公开池（平台内）
  阶段策略 + 隐私闸门 + 域内 CV/规则选图
  商家完工确认的「案例正文草稿」+ 脱敏配图清单
  车主未点「发布到公开网站」前 = 不上 H5 / GEO

层级 C · 公域知识包（H5 / GEO / 外发长文）
  CaseSnapshot（事实冻结，含商家正文与配图 URL）
  + 车主评价（若纳入公开）
  + GeoEnrichment（标题润色 / FAQ 等，可改，不改 snapshot 事实）
```

| 方案层 | 辙见落点 |
| --- | --- |
| 私域全量 | live 服务相册 |
| 候选公开池 | PublicView 候选 + `merchant_case_draft`（见 §6） |
| 公域事实 | `CaseSnapshot` / `publicView` |
| 公域文采 | `GeoEnrichment`（仅脱敏文本/已公开字段） |

## 3. 合规铁律（不可破）

1. **未点「发布到公开网站」前**  
   - 禁止：原图 / 未脱敏图 → **任何外部多模态大模型**。  
   - 允许：平台自有服务器上的传统 CV、OCR 隐私检测、规则教练、文字-only LLM（输入白名单见 §5）。

2. **已授权 / 已发布后**  
   - 车主同意的是：展示**商家精选脱敏案例正文 + 脱敏关键帧**（及车主评价，若展示）。  
   - **不**同意：将全部原始照片交给第三方 AI 分析或训练。  
   - 外网 LLM（含多模态）**只许**使用脱敏后媒体 + scrub 文本；生产默认保持 Vision 关闭，直至合同具备零训练留存（ZDR）等条款。

3. **双向网关**  
   - 出域：脱敏网关（去 EXIF、打码、字段白名单）。  
   - 入域：生成结果进草稿或 enrichment，**禁止回写** snapshot 已冻结事实。  
   - 审计：任务 ID、输入字段哈希、模型商、时间戳可查。

4. **合同层（采购）**  
   - Zero Data Retention、短时缓存销毁、安全审计接口；不具备则多模态默认关闭。

## 4. 公开 vs 私人分享：风险拦截差异

| 场景 | 高风险图（车牌 / 人脸 / 钥匙 / 导航屏 / 单据金额等） | 金额文案 |
| --- | --- | --- |
| **发布到公开网站 / H5·GEO** | **硬拦**：不得进入公开配图包 | **不写任何金额数字**；不写「费用以门店为准」类套话 |
| **复制去社交媒体**（基于公开稿） | 同公开包（默认只用可公开配图） | **不写金额** |
| **发给微信 / 朋友圈** | **不因「公开硬拦」而阻止分享**；仍建议走脱敏图链路，但不套用公开级熔断 UX | **不写金额** |

说明：硬拦管的是「上网给不特定公众看」；朋友圈为车主私域传播，平台不套用公开硬拦拦死分享，但**文案生成仍禁止输出金额**。

## 5. 金额策略（公域藏价）

| 场景 | 是否展示 `planAmount` / 价税数字 |
| --- | --- |
| 商家编辑 / 私密相册（车主·门店） | ✅ 可见（私域知情权） |
| 公开案例 H5 / Feed / GEO | ❌ 不展示 |
| 商家完工示例稿 / 车主复制长文 / 朋友圈文案 | ❌ 不写金额；相关段落改为旧件确认、责任边界、方案要点等，**整段省略价格** |
| 传给外部 LLM 的输入 | ❌ strip 一切金额字段后再调用 |

废止公开侧「价格诚实展示成交价」作为公域默认（原 Phase1 §10 公开行以本文为准）。服务**商品**四型价格展示不变。

## 6. 文案归属与角色分工

| 内容 | 归属 | 车主能否改 |
| --- | --- | --- |
| 案例标题、各节正文、配图位置与选用方案（商家确认版） | **商家**（服务内容 ≈ 商品详情） | **否**（只读预览） |
| 评价 / 反馈 | **车主** | 仅此可写 |
| 是否发布到公开网站 | **车主** | 决定权 |
| 公开配图增删 | **车主可删不可加**（规则见 §8） | 仅可从公开配图包移除；不可自行加图 |

**产品类比**：电商平台不让顾客改商品详情；顾客可评价、可决定是否把购物晒到公开社区。

商家若需改正文：须在相册锁定规则允许的窗口内（如未发布或车主撤回后），由商家编辑后再生成/更新草稿；**禁止**在车主预览页开放正文编辑。

## 7. 端到端主路径

```text
① 商家制作相册
   · 相册教练规则引擎（§9）：按服务类型提示「要拍 / 别拍 / 备注怎么写」
   · 不把原图送给外部大模型

② 商家准备完工（一稿定稿）
   · 完整度检查清单（规则引擎输出，默认不因缺节点 alone 硬拦完工，与 Phase1 一致）
   · 点「完工」→ 案例预览页：按节点文字+公开配图**规则拼稿**（同步，不调大模型）
   · 可选「AI 润色」：仅传白名单文字（§7.1）；**图片不进入大模型**；润色后 media[] 原样按 sectionKey 挂回
   · 商家可手改正文、删减配图 → 「确认并完工」= 最终案例稿
   · 写入「商家案例草稿」confirmedAt → 真正 completed → 触发 pre-mask / 闸门 A

③ 车主端
   · 严格只读呈现商家确认稿（文案不可改）
   · 可发布评价；公开配图可删不可加（§8）
   · 确认「发布到公开网站」→ 闸门 B → H5/GEO（正文真源 = 确认稿，不模板整篇重建）
   · 发给微信/朋友圈 / 复制社交媒体：与商家自媒体导出同源确认稿；文案不写金额；平台不代发

④ 过审后
   · Snapshot 冻结商家确认正文 + 确认后的公开配图 +（若纳入）评价摘要
   · Enrichment 可对 scrub 后文本做 GEO 润色；不改 snapshot 事实
```

### 7.1 完工 LLM 输入白名单（仅「AI 润色」主动调用）

**允许**：当前预览草稿章节正文、节点 ID/标题、节点备注与说明、服务名、车型（脱敏粒度）、城市/门店名、配件**名称**（无编码、无供应商）、规则引擎给出的 `geo_angle` 提示。

**禁止**：任意图片或图片 URL、EXIF、原图 OCR 全文（若含 PII/金额）、`planAmount` 及一切价税数字、完整工单/结算单图。

**输出约束**：结构化案例稿 sections；**不得出现具体费用**；命中金额则打回或自动抹除后再给商家预览；**不得改写 media 列表**。

### 7.2 配图与 LLM 隔离

```text
节点文字 ──►（可选）外部 LLM 润色 ──► sections 正文
本册候选图 ──► 域内选图 ──►（确认完工后）脱敏网关 ──► media[] 按 sectionKey 嵌入
                    ▲
              图片绝不进入 LLM
```

## 8. 公开配图：可删不可加 + 高风险硬拦

| 动作 | 规则 |
| --- | --- |
| 系统初选 | 域内规则/CV 给出约 **5～8** 张关键帧嵌入商家草稿（软上限可配置，默认优于「过闸全进 ≤30」） |
| 商家确认完工前 | 可调整配图（仍受阶段策略与闸门约束；高风险图**不得**进入公开配图包） |
| 车主删 | 可从公开配图包移除（降低顾虑、保留控制感） |
| 车主加 | **不可加**：公开配图范围以商家确认稿为准，车主不能从本册或手机另增图片上公开网 |
| 高风险（商家/系统侧） | **硬拦**：车牌、人脸、钥匙、车机导航、单据金额姓名等不得进入公开配图包（清单可配置） |
| 仅留档阶段 | stage_1 / stage_3 等 `always_private`：**不可**进入公开包 |

车主预览 = WYSIWYG：**商家正文（只读）+ 当前公开配图包（可删不可加）+ 评价区**；不是全量私域相册墙。

说明：配图「加什么」由商家在完工稿中决定（类商品主图）；车主可删减不愿公开的帧，但不可扩权加图，避免绕过商家留档与闸门。

## 9. 相册教练规则引擎（重点）

### 9.1 目标

在**源头**帮助修理厂做出「基本完整、可公开、可 GEO」的相册：

1. **别拍什么**（隐私与无关物）  
2. **要拍什么**（完整度与信任素材）  
3. **备注怎么写**（短句结构，便于完工 LLM）  
4. 按**服务类型**匹配不同规则包（可运营配置）

**不做**：用外部大模型看原图做教练（合规禁止未授权/全量外传）。

### 9.2 与现有能力关系

| 已有 | 规则引擎中的位置 |
| --- | --- |
| `02_相册模板与节点规则.md` 分服务模板 | 规则包的**业务真源之一**；引擎输出更偏「当下提示」 |
| `14` STAGE 拍摄一行 hint | 并入 `shoot_prefer` / `shoot_avoid` 的 UI 呈现 |
| publicGate / always_private | **硬过滤**；教练是**软引导**，不替代闸门 |
| 文案体检（PII/绝对化） | 备注与完工稿的**出口检查** |

### 9.3 引擎输入 / 输出

**输入**

```text
serviceTypeId | serviceCategory   # 服务类型（保养、刹车、底盘异响、钣喷…）
stageId                           # stage_1 … stage_6
albumSnapshot                     # 各阶段已有图数量、备注是否为空、gate 结果摘要
merchantTier / checklistMode      # 可选：提示强度
```

**输出（对商家端）**

```text
coachCards[]          # 当前阶段常驻卡片：要拍 / 别拍 / 备注示例
uploadInlineHints[]   # 点「添加照片」时的一行提示
notePlaceholder       # 备注输入框占位与结构提示
completenessReport    # 完工前检查清单与缺口（建议项 / 强烈建议）
geoAngleHints[]       # 建议强化的 GEO 维度：standard_5s | avoid_pitfall | liability
draftPromptHints      # 传给完工 LLM 的角度提示（仍无图）
```

### 9.4 规则包数据模型（配置态）

建议独立配置表或 JSON 包（运营可改，发版可热更新）：

```text
RulePack {
  id
  serviceTypeIds[]          # 匹配的服务类型；含 "*" 通用兜底
  priority                  # 多包命中时取高优 + merge 策略
  version
  stages: {
    [stageId]: StageRules
  }
  completeChecklist[]       # 跨阶段完工项
  geoPyramidHint            # 可选：塔尖/塔身/塔基建议标签
}

StageRules {
  shoot_avoid[]    # { code, title, detail, severity: info|warn }
  shoot_prefer[]   # { code, title, detail, strength: tip|strong }  # strong=完工清单强调
  note_hints[]     # { title, example, bullets[] }  # 怎么写
  geo_angle[]      # standard_5s | avoid_pitfall | liability
  public_policy    # 继承全局 always_private | gate（只读展示用）
}
```

**Merge 策略**：`serviceType` 专用包覆盖同 `code` 的通用包；`shoot_*` 数组合并去重；`note_hints` 专用优先。

### 9.5 通用规则（所有服务类型兜底）

#### 9.5.1 别拍（shoot_avoid）

| code | 提示摘要 |
| --- | --- |
| `plate_full` | 避免清晰车牌入镜；需整车时注意角度或仅作留档 |
| `face` | 避免人脸、工牌特写入镜 |
| `key` | 避免车钥匙、遥控入镜 |
| `cabin_privacy` | 避免特殊内饰、私人物品特写 |
| `nav_screen` | 避免车机导航/行踪信息清晰入镜 |
| `other_vehicle` | 避免背景其他客户车辆可辨识 |
| `doc_amount` | 工单/结算单上的金额、姓名；单据图仅留档、不进公开 |

#### 9.5.2 要拍（shoot_prefer）— 按阶段通用底稿

| 阶段 | 建议拍 | 备注怎么写（note_hints 要点） |
| --- | --- | --- |
| stage_1 接车 | 到店里程、约定外观点（知悉仅留档） | 到店诉求一句话；异常外观点 |
| stage_2 检测 | 故障点近景、检测数据/设备读数 | 现象 + 检测手段 + 结论（排除了什么） |
| stage_3 方案 | **图仅留档**；字段填方案与配件名 | 方案要点、为何不采用过度维修；**金额只填字段不进公开稿** |
| stage_4 配件 | 零件/包装特写、旧件 | 配件名称与品质说明（无供应商编码可公开） |
| stage_5 施工 | 防护/工具摆放、扭矩打卡、关键工序 | 力矩标准、防松标记、5S 要点 |
| stage_6 交付 | 修复局部、机舱清洁复查、旧件交接 | 质检项、旧件是否交还车主 |

#### 9.5.3 GEO 三维（写入 geo_angle，供完工 LLM）

| 维度 | 编码 | 内容取向 |
| --- | --- | --- |
| 1 · 5S 与标准化 | `standard_5s` | 防护垫、工具车、扭矩、清洁复查；打消「乱丢工具/少螺丝」顾虑 |
| 2 · 避坑与科普 | `avoid_pitfall` | 现象 + 实测数据 + 方案对比；反对过度维修叙事 |
| 3 · 责任边界 | `liability` | 自带件核验、防伪、确认单；纠纷边界清晰 |

**金字塔配比（运营教育，非系统强制月更 10 篇）**：塔尖技术权威 20% / 塔身信任构建 50% / 塔基科普避坑 30%。规则包可打 `geoPyramidHint` 供商家「本月内容建议」展示。

### 9.6 分服务类型规则包（首批）

首批 `serviceTypeIds` 与 `02_相册模板与节点规则.md` 对齐，至少覆盖：

1. 小保养 / 大保养  
2. 刹车片/盘  
3. 电瓶  
4. 轮胎  
5. 空调  
6. 钣喷  
7. **底盘异响 / 悬挂胶套类**（高频 GEO）  
8. 通用 `*` 兜底  

以下为**产品示例**（实现时可落配置文件；表格可扩展，不必改代码发版）。

#### 9.6.1 底盘异响 / 胶套类

| 阶段 | 要拍 | 别拍（叠加通用） | 备注怎么写 |
| --- | --- | --- | --- |
| 检测 | 举升后胶套近景、撬动对比、球头检查 | 车牌全景 | 过减速带异响部位；已排除球头旷量等 |
| 方案 | —（图留档） | 报价单金额入镜 | 为何不换总成；轻量化压套方案 |
| 施工 | 专用压套工具、扭矩打卡、防松标记 | 钥匙、人脸 | 原厂标准力矩；定位是否复测 |
| 交付 | 旧件、局部复查 | — | 旧件已交还；路试结论 |

`geo_angle`：优先 `avoid_pitfall` + `standard_5s`。

#### 9.6.2 常规保养

| 阶段 | 要拍 | 备注怎么写 |
| --- | --- | --- |
| 施工 | 翼子板防护、工具车摆放、机油防伪/液位、滤芯新旧 | 项目清单；质检项数量化表述（如「机舱清洁复查」） |
| 交付 | 机舱清洁、旧件 | 旧件确认 |

`geo_angle`：优先 `standard_5s`（塔身信任）。

#### 9.6.3 刹车

| 阶段 | 要拍 | 备注怎么写 |
| --- | --- | --- |
| 检测 | 刹车片厚度实测、盘面状态 | 实测厚度；何种情况建议更换、何种不必 |
| 施工 | 安装扭矩、刹车管路可见规范操作 | 排气/磨合建议（无绝对化承诺） |

`geo_angle`：`avoid_pitfall` + `standard_5s`。

#### 9.6.4 钣喷

| 阶段 | 要拍 | 备注怎么写 |
| --- | --- | --- |
| 施工 | 无尘/防护、色差比对过程（无车牌） | 工序与质检；避免「100% 无色差」绝对化 |
| 交付 | 修复局部完工 | 交车检查项 |

#### 9.6.5 自带配件 / 责任类备注（跨类型 note 插件）

当相册标记「车主自带件」或备注含自带关键词时，注入 `liability` 提示：

- 要拍/要写：防伪查验、认证标号核对、施工前确认说明  
- 文案角度：责任划分与核验过程（维度 3）

| 缓存 | 按 `rulePack.version` + album 指纹缓存输出，规则更新后失效 |

**提示强度**：Phase1 默认 **不因教练缺口阻断完工**；`strength=strong` 仅强化展示。商业套餐 L2+ 若未来要阻断，复用 `02` 的 `required_level`，与教练清单打通。

### 9.8 规则引擎验收（文档级）

1. 切换服务类型，检测/施工阶段「要拍 / 别拍 / 备注」文案明显不同。  
2. 通用避忌在所有类型可见。  
3. 完工 LLM Prompt 含 `geo_angle`，且**无图、无金额**。  
4. 运营改配置后无需发版即可生效（或热加载）。  
5. 教练提示**不**触发外部 Vision API。

## 13. 授权文案颗粒度（须同步法律专页）

> **产品策略**：把「同意发布」包装成 **车主权益与社交货币**——荣誉参与感 + 绝对控制权；合规边界写进副文案/协议全文，主确认语用鼓励式、有情绪价值的语言。  
> **称号**：可用「透明维修体验官」「避坑体验官」等（随机轮换、无等级差异）。

### 13.1 主确认语（鼓励式 · 推荐）

**策略 1 · 荣誉与参与感（避坑体验官）**

面向高端车型或疑难维修等场景，赋予车主「专家/利他」身份，示例：

> 恭喜您的爱车满血复活！我们为您整理了一份《{车型}{项目}避坑指南》。  
> 是否愿意作为「透明维修体验官」，将这份**脱敏报告**分享给同城车友，帮助更多人避开修车陷阱？

效果取向：激发利他与社交分享欲；车主可自行转发朋友圈/车友群（私人分享路径，不等于上公开网站）。

**策略 2 · 绝对控制权（与主确认语同屏必现）**

> 我们承诺：案例仅用于技术科普，绝不泄露您的任何隐私。  
> 您可在小程序「我的服务相册」随时点击**下架**，公开站将尽快删除相关内容（第三方缓存清除可能有延迟）。

主按钮/确认区须让车主感到：**分享是权益，下架是随时可用的权利**——而非「把全部维修记录交出去」。

### 13.2 合规边界（副文案 / 法律专页须写清）

在协议全文或折叠「了解将展示什么」中保留窄授权，避免主文案过宽：

- 将展示：门店制作的**精选维修节点脱敏照片**、**脱敏后的案例说明（商家服务内容）**，以及你的**评价**（若你填写）。  
- **不包含**：个人信息、完整车牌/VIN、人脸、**具体金额**、完整工单原件。  
- **不包含**：将全部原始照片提供给第三方用于模型训练。

废止：「授权公开所有维修记录」、公域展示成交价、压迫式「授权给我们用于全网营销」等表述。

### 13.3 产品文案禁忌

- 主按钮避免生硬「授权公示」短句；优先邀请式长句（与 Phase1 §4.5.2a 荣誉鼓励一致）。  
- 禁止分享领钱、好评返现等物质诱导。
