## 2. 模板定位

相册模板用于指导商家在 **服务相册** 中按服务类型与 **六阶段** 记录维修过程，并为车主查看、配件确认、公开案例和 GEO 内容提供结构化基础。

模板作用：

1. 给商家提供 **分阶段拍摄指引**（降低操作负担）；
2. 支撑 **配件确认卡** 与合规检查清单；
3. 让用户通过时间线 **看得懂** 维修内容（配合翻译层）；
4. 提升公开案例与证据包质量；
5. 为标题、摘要、FAQ、SEO 提供字段来源；
6. 为案例质量分与 GEO 推荐权重提供依据。

## 2.1 六阶段与模板的对应关系

| 阶段 | 模板层节点类型 | 说明 |
|---|---|---|
| 一、接车 | before / intake | 到店、里程、外观 |
| 二、检测 | before / diagnose | 故障点、检测图 |
| 三、方案 | plan | 项目清单、**固定参考报价**（`planAmount`） |
| 四、配件 | parts + **配件确认卡** | 标准配件类型、非原厂确认 |
| 五、施工 | process / parts | 新旧对比、包装、编码 |
| 六、交付 | after / check | 完工、验收、质保 |

服务模板（§5–11）中的节点应 **映射** 到上述阶段；商家端按阶段 Tab 展示，而非仅按「维修前/中/后」三分法。

## 3. 设计原则

### 3.1 风险分级：建议 vs 阻断

| required_level | 含义 | L1 | L2 | L3/L4 |
|---|---|---|---|---|
| optional | 可选 | 可跳过 | 可跳过 | 建议上传 |
| recommended | 建议 | 建议 | 强烈建议 | 建议 |
| required | 必须 | — | **未满足则不可 completed** | **未满足则不可 completed** |

**V2.0 变更**：L2+ 下 `required` 节点（含配件类型、非原厂确认、新旧件对比）未完成时，系统 **阻断** 相册完工，不再仅影响评分。

L1 服务仍保持「建议不强制」，以符合主 PRD 中「商家极简」取向。

### 3.5 方案报价表与 `planParts`（定稿 2026-06-24 · B-PLAN）

> **目标**：从阶段三 **详细报价表/说明图** 结构化出 **方案零配件目录**，供阶段四逐件上传与车主 **配件验真** 对照。  
> **原则**：OCR **辅助录入**，须商家 **确认后锁定**；平台 **不鉴定** 配件真伪。

#### 3.5.1 阶段三 · 必传内容（L2+ 建议，L3/L4 强制）

| 内容 | 形式 | 存储 |
| --- | --- | --- |
| 维修方案说明 | 文本 + 可选过程图 | `stage_3.note` / 节点图 |
| **详细报价表** | **1～N 张图**（系统截图/纸质拍照） | `planQuoteImageIds[]` |
| 方案参考报价总额 | 单行金额 | `planAmount`（已有） |

**交叉校验（P1）**：OCR 行合计与 `planAmount` 不一致时，商家端 **黄条提醒**，不自动改价。

#### 3.5.2 `planParts[]` 字段（方案侧 · 锁定后只读给车主）

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `planPartId` | 是 | 稳定 ID |
| `name` | 是 | 如「左前大灯总成」 |
| `partType` | 是 | 标准类型（§3.3 / `10_配件告知` §3） |
| `partBrand` | 建议 | 品牌 |
| `partCode` | 建议 | 配件编码（OCR 或手工） |
| `qty` | 是 | 数量，默认 1 |
| `unitPrice` / `lineTotal` | 可选 | 来自报价表 OCR |
| `sourceImageIds[]` | 建议 | 对应报价表图 |
| `ocrRevision` | — | OCR 版本 |
| `status` | 是 | `draft` → `confirmed` |

**锁定**：商家在 **方案 OCR 确认台** 点击「确认方案配件目录」后写入 `planPartsLockedAt`；锁定后阶段四 **按目录引导**，验真页 **方案侧** 以此为准。

#### 3.5.3 方案报价表解析（`plan_quote_ocr` · 2026-06-24 增强）

| 项 | 规则 |
| --- | --- |
| 触发 | 阶段三上传/更新报价表图 → 商家点击「识别报价表」 |
| 策略 | **多模态 LLM 直连**；按 `PLAN_QUOTE_LLM_ENGINES` 顺序调用，前一模型失败自动切换下一模型 |
| 输出 | `planPartsDraft[]`（`status: draft`）+ `parseMethod: llm` + `llmEngine` + 原始文本预览（审计） |
| LLM | 百炼/豆包/Kimi 等 OpenAI 兼容多模态；**优先读本机 media 落盘（base64）**，无本地文件时用公网 URL |
| 失败 | 允许 **纯手工** 维护 `planParts`；L3+ 可配置必须有报价表图才允许锁定 |

环境变量（backend）：`PLAN_QUOTE_LLM_ENGINES=plan_quote,geo_vision,qwen_vl,doubao,kimi`；`PLAN_QUOTE_LLM_ENABLED`；`DASHSCOPE_API_KEY` 或 `PLAN_QUOTE_LLM_API_KEY`。

### 3.6 配件节点 · 按 `planParts` 引导上传（定稿 2026-06-24 · B-PART）

阶段四（`stage_4` / `partsJson`）由 **清单式向导** 驱动，而非自由增删：

| 规则 | 说明 |
| --- | --- |
| 1 planPart → 1 albumPart | `linkKey = planPartId` |
| 每行至少 1 张凭证图 | 包装 / 编码标签 / 实物 |
| 配件 OCR（`part_label_ocr`） | 从凭证图识别 `partCode`、`partBrand` 候选；**商家确认** 后写入 |
| 标准类型 | 商家必选；可与 OCR 建议不一致，以商家选择为准 |
| **增项** | Phase 1 **不** 做独立「+ 方案外增项」向导；增项须 **阶段三补传经客户确认的报价/增项单** → 更新 `planParts` → 阶段四按目录登记（见 **`11_施工增项留痕SOP.md`**）。数据层仍支持 `source: extra`，验真页单独分组「方案外增项」 |

#### 3.6.1 `partsJson[]` 扩展字段（相册侧）

在 §3.3 基础上增加：

| 字段 | 说明 |
| --- | --- |
| `planPartId` / `linkKey` | 绑定方案行 |
| `partBrand` / `partCode` | 品牌、编码 |
| `source` | `plan_linked` \| `extra` |
| `ocrRevision` | 配件图 OCR 版本 |
| `confirmedAt` | 商家确认时间 |

#### 3.6.2 配对状态（验真 API 计算）

| `linkStatus` | 含义 |
| --- | --- |
| `linked` | 方案行与相册行 1:1 绑定 |
| `plan_only` | 方案有、相册未登记 |
| `album_only` | 相册有、方案无（多为 extra） |
| `field_diff` | 已 link 但 type/brand/code 等字段不一致（**视觉提示**，非平台判定） |

#### 3.6.3 登记字段与「配件分类」产品定稿（2026-06-25）

**不做** 商家可见的「主要更换件 / 油液 / 辅料」分组或分级必填；清单 **一行一件、字段统一**。

| 字段 / 规则 | Phase 1 商家端 |
| --- | --- |
| 配件名称 | 来自方案识别或手工；标题展示，展开区不重复输入 |
| 标准类型（原厂/品牌/副厂等） | **保存本项必填** |
| 凭证图 | **保存本项必填**（≥1 张） |
| 品牌、编码 | **选填**；机油/制动液/冷却液与刹车片等 **同一口径**（油液亦有原厂/品牌/副厂差异） |
| 报价识别 | LLM 过滤纯工时行（`itemKind=labor` 等）；**不**向商家展示 `partCategory` |
| 验真 | 不因油液单独弱化编码对照；商家未填字段不参与 diff |

曾评估的向导分组、油液简化登记、验真编码弱化（开发计划 B-PART-04～08）**搁置**。

#### 3.6.4 旧件留痕（定稿 · 2026-07-05 · stage_5）

| 规则 | 说明 |
| --- | --- |
| 位置 | **阶段五**，独立上传区；**不在** 阶段四配件向导 |
| 形态 | 批量图 + **可选** `planPartId` 关联（下拉来自阶段四清单） |
| 必填 | Phase 1 **不强制**；L3+ 可在检查/AI 中加重「主要更换件建议留痕」 |
| 存储 | 与施工工单、通用过程图 **分桶**（见 `03/04_维修相册` §15 · `evidenceItems`） |
| 与新件 | 新件凭证仍在 `partsJson[].photos`；旧件 **不要求** 逐行镜像新件表单 |

### 3.4 公开案例优先使用高质量节点

公开案例优先使用以下类型图片：

1. 维修前状态；
2. 故障点；
3. 新旧配件对比；
4. 维修过程；
5. 完工结果；
6. 前后对比图；
7. 质检或检测结果。

不优先使用：

1. 模糊图片；
2. 重复图片；
3. 只拍门店环境但无维修信息的图片；
4. 隐私风险高的单据图片；
5. 车牌、人脸、VIN 难以有效脱敏的图片。

```
Q1：小保养一般多久做一次？
A：通常建议根据车辆保养手册、机油类型和实际用车环境决定，一般为 5000-10000 公里或 6-12 个月。

Q3：只换机油不换机滤可以吗？
A：通常建议机油和机滤一起更换，以保证过滤效果和保养质量。

```

```
Q2：大保养必须一次性全部做完吗？
A：不一定，应根据车辆保养手册、检测结果和实际车况决定。

Q3：旧件为什么建议拍照？
A：旧件照片可以帮助车主了解磨损和更换原因，也能提升维修透明度。

```

## **14. P0 范围**

P0 需要支持：

1. 7 个首批模板；
2. 每个模板具备建议节点；
3. 商家上传图片时可选择节点；
4. 系统根据服务项目自动匹配模板；
5. 商家可手动切换模板；
6. 节点完整度可计算；
7. 图片可按节点展示给用户；
8. 公开案例可读取节点结构；
9. FAQ 模板可用于案例生成；
10. 平台审核页可查看节点完整度。

## **15. 验收标准**

1. 每个首批服务项目可匹配到正确模板，并映射六阶段；
2. 商家可按阶段/节点上传图片；
3. L2+ required 节点未完成时不可将相册标为 completed；
4. 配件更换节点须可填写标准 part_type；
5. 用户端相册按六阶段 + 节点时间线展示；
6. 公开案例可使用节点生成正文；
7. 模板停用后不影响历史相册展示。
