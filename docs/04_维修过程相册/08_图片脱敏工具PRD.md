## 2. 产品定位

### 2.1 独立工具，而非「某业务流程里顺带调一下接口」

图片脱敏工具是透明维修平台的 **隐私基础设施**，与「维修相册」「案例审核」解耦：

- **能力层**：上传原图 → 一键自动脱敏 → 预览 → 可选手动修补 → **确认脱敏结果** → 输出仅含脱敏图的数据包。
- **业务层**：维修相册、历史案例、用户授权公开、运营审核等 **只消费** 已确认的脱敏结果；不在各业务里各写一套脱敏 UI。

未来可单独发布为：

- 官网「隐私打码工具」；
- 对外 H5 / 小程序插件；
- 其他业务线（保险、二手车等）通过同一 API 接入。

### 2.2 商家端与车主端：同一工具、同一流程

| 维度 | 要求 |
|---|---|
| 交互流程 | **完全一致**（见 §4） |
| 组件与页面 | 共用 `DesensitizeWorkbench`（工作台式脱敏台），仅 **文案、责任确认、后续动作** 按场景配置 |
| 脱敏算法与规则 | 同一套自动脱敏服务 + 同一套规则配置（见 `06_自动脱敏规则.md`） |
| 输出数据 | 统一字段：`rawAssets[]` / `maskedAssets[]` / `maskingStatus` / `confirmedAt` |

差异 **仅** 在于：

| 场景 | 责任主体 | 确认后流向 |
|---|---|---|
| 商家历史案例 / 非平台相册 | **商家** 对上传合法性与隐私负责 | 提交案例审核 → 平台审核 → 公开案例库 |
| 平台订单授权公开 | **用户** 对是否公开负责 | 提交授权 → 平台审核 → 公开案例库 |
| 运营复核 | 平台 | 对比原图与脱敏图（权限隔离） |

### 2.3 公示硬闸门（全平台统一）

**未在脱敏工具内完成「脱敏结果确认」的内容，禁止进入任何公开展示链路**，包括但不限于：

- 用户端案例 Tab / 案例详情；
- 门店主页案例列表；
- H5 公开案例页；
- 分享卡片与外链落地页。

原图（`raw`）仅允许在 **权限隔离** 的上下文中访问（商家本人草稿、用户本人订单档案、运营审核后台），不得作为公开 `src` 下发给 C 端或 SEO 页。

## 3. 核心流程（统一）

```text
选择/上传图片（原图，业务上下文内私有）
  → 【一键脱敏】调用自动脱敏服务（批量）
  → 进入脱敏工作台（预览每张脱敏结果）
  → （可选）手动打码 / 裁剪 / 替换单张
  → （可选）重新一键脱敏（覆盖或增量）
  → 勾选《脱敏结果确认》（+ 场景化《隐私/授权责任》）
  → 确认完成（写入 maskingConfirmedAt）
  → 业务后续：提交审核 / 授权公开 / 发布
```

### 3.1 一键脱敏

| 项 | 说明 |
|---|---|
| 触发 | 工作台主按钮「一键脱敏」；支持对全部待处理原图批量处理 |
| 行为 | 调用平台脱敏服务（OCR + 车牌/人脸/VIN 等规则，见 `06`） |
| 反馈 | 逐张状态：排队中 / 处理中 / 成功 / 需人工 / 失败 |
| 失败 | 不允许进入「确认完成」；需重试或手动打码 |

### 3.2 脱敏工作台（DesensitizeWorkbench）

所有端共用同一工作台布局：

```
顶部：进度（已处理 N/M）+ 风险摘要（可选）
中部：原图 | 脱敏图 对比（或仅脱敏图网格）
底部：一键脱敏 | 手动打码 | 重新处理选中 | 确认脱敏结果
```

| 能力 | MVP 必选 | 后续 |
|---|---|---|
| 脱敏图预览 | ✅ | — |
| 原图 vs 脱敏对比 | ✅ | — |
| 一键脱敏 / 重试 | ✅ | — |
| 手动打码 | P1 | 框选、马赛克、模糊 |
| 裁剪 | P2 | — |
| 批量确认 | ✅ | 须逐张浏览至少一次（可记录 `previewed`） |

### 3.3 确认脱敏结果

确认前必须满足：

1. 每一张将用于公开的图片均有 **脱敏成功** 或 **人工处理后** 的 `maskedUrl`；
2. 用户/商家已勾选场景化责任文案（见 §5）；
3. 写入 `maskingConfirmed: true` 与 `maskingConfirmedAt`（及操作人 `operatorId`）。

确认后：

- 业务接口 **只提交 `maskedAssets`**；
- 前端公开组件 **禁止绑定 `rawUrl`**。

## 4. 状态机与数据模型

### 4.1 单张图片

| 状态 | 说明 |
|---|---|
| `raw_uploaded` | 原图已上传，未脱敏 |
| `masking` | 自动脱敏中 |
| `masked_ready` | 脱敏完成，待人工预览 |
| `mask_failed` | 自动脱敏失败，需手动或重试 |
| `manual_masked` | 人工打码后可用 |
| `confirmed` | 已纳入「确认脱敏结果」集合 |

### 4.2 脱敏任务（批次）

| 字段 | 说明 |
|---|---|
| `taskId` | 脱敏任务 ID |
| `bizType` | `merchant_history` / **`order_pre_mask`** / `order_authorize` / `standalone_tool` |
| `bizId` | 相册 ID、订单相册 ID 等 |
| `operatorRole` | `merchant` / `user` / `ops` / **`system`**（`order_pre_mask` 后台任务） |
| `rawAssets[]` | 原图引用（私有存储） |
| `maskedAssets[]` | 脱敏图引用（可公开） |
| `preMaskVersion` | 订单预脱敏版本（`order_pre_mask`） |
| `preMaskTaskId` | `order_authorize` 关联的预脱敏任务 ID |
| `fromPreMask` | 授权预览是否来自预脱敏 |
| `maskingConfirmed` | 是否已确认 |
| `maskingConfirmedAt` | 确认时间 |
| `liabilityType` | `merchant` / `user`（责任归属） |

### 4.2.1 订单相册预脱敏（`order_pre_mask`）

**实现形态**：与商家历史案例相同，写入统一脱敏任务表（`desensitize_tasks` / 服务端等价存储），`bizType = order_pre_mask`，**不单独使用平行 storage**。

| 字段 | 说明 |
|---|---|
| `taskId` | 固定建议 `task_premask_{albumId}` |
| `bizType` | **`order_pre_mask`** |
| `bizId` | 订单相册 ID |
| `operatorRole` | **`system`**（后台异步，无人工操作台） |
| `liabilityType` | `platform` |
| `fingerprint` | 原图节点摘要；变更则 `preMaskVersion++` 并重跑 |
| `preMaskStatus` | `idle` / `running` / `ready` / `partial_failed` / `failed` |
| `preMaskVersion` | 原图变更后递增 |
| `preMaskedAt` | 最近完成时间 |
| `rawAssets[].preMaskedUrl` | 预脱敏图（private；用户仅在授权预览页可见） |
| `maskingConfirmed` | **恒为 false**（预脱敏不可代替用户确认） |

**触发**：商家提交完工（`03_订单相册流程.md` §6.4.1）；上传增量 P1。

**消费**：

1. 用户「确认公开」→ 读取 `order_pre_mask` 任务；
2. 创建 **`order_authorize`** 任务（`preMaskTaskId` / `preMaskVersion` 关联）；
3. 用户预览工作台核对 → `order_authorize.maskingConfirmed = true` → 提交公开。

**任务关系**：

```text
order_pre_mask (system, 后台)
       ↓ 复制 preMaskedUrl
order_authorize (user, 预览工作台)
       ↓ maskingConfirmed
公开案例素材
```

### 4.3 与业务文档字段对齐

| 历史 PRD 字段 | 本工具统一为 |
|---|---|
| `masked_image_url` | `maskedAssets[].url` |
| `privacy_status` | 单张 `status` + 批次 `maskingConfirmed` |
| `privacy_confirmed` | `maskingConfirmed` + 场景责任勾选 |

## 5. 责任与文案（场景配置，工具不变）

工具 UI 相同；**责任说明** 由 `bizType` 注入：

### 5.1 商家历史案例（`merchant_history`）

```
你上传的内容用于商家历史案例公开展示。请确认已取得必要授权，
或确保不会暴露客户车牌、人脸、手机号、证件等信息。
平台将进行自动脱敏与审核，商家对上传内容的合法性与真实性承担责任。
```

- 责任主体：**商家**
- 确认按钮文案：**确认脱敏结果并提交审核**

### 5.2 用户授权公开（`order_authorize`）

```
以下为脱敏后的预览，公开后他人仅能看到脱敏版本。
请逐张核对效果，必要时可重新处理或手动打码。
```

- 责任主体：**用户**
- 确认按钮文案：**确认并公开**
- 预脱敏：若 `order_pre_mask` 已完成，进入预览页时**直接展示**脱敏图，无需等待

### 5.3 独立工具页（`standalone_tool`，未来）

```
本工具仅用于隐私打码预览。请勿上传违法或侵权内容。
脱敏结果仅供参考，重要场景请人工复核。
```

## 6. 接入方式（业务只编排，不重复造轮子）

| 业务场景 | 接入点 | 脱敏工具入口 |
|---|---|---|
| 商家新建历史案例 | `05_历史案例上传.md` | 上传节点图后 → **进入脱敏工作台** → 确认后才可「提交审核」 |
| 商家编辑草稿 | 案例管理 | 任意时刻可重新进入工作台 |
| 用户订单授权公开 | `03_订单相册流程.md` §6 | 用户点击「授权公开」→ **脱敏工作台** → 确认后提交 |
| 运营审核 | `06_自动脱敏规则.md` | 查看原图/脱敏对比（只读，高权限） |
| 官网独立工具 | 未来 | 仅 `standalone_tool`，无后续审核时可导出脱敏图 |

**禁止**：在 `album/create` 等业务页仅勾选「将自动脱敏」即直接提交审核，**绕过工作台确认**。

## 7. 技术形态（便于独立部署）

### 7.1 前端

| 产物 | 说明 |
|---|---|
| `DesensitizeWorkbench` | 业务组件：预览、一键脱敏、确认闸门 |
| `ImageMaskEditor` | 手动打码（P1） |
| `services/desensitize.js` | 统一 API：创建任务、批量脱敏、确认、查询状态 |

各端 **路由不同、组件相同**：

- 商家：`/packageMerchant/pages/desensitize/workbench?taskId=`
- 用户：`/pages/desensitize/workbench?taskId=`
- H5/官网：同源 Web 组件或独立子域（未来）

### 7.2 后端（契约原则）

- 脱敏服务 **独立模块**（可单独扩容、单独 SLA）；
- 输入：`rawUrl`；输出：`maskedUrl` + `riskTags[]`；
- 公开 API **永不返回** `rawUrl`（C 端 / H5 / 案例列表）。

### 7.3 与 V0.1 mock 的关系

V0.1 的 `utils/desensitize-mock.js` 仅为占位，**不符合本 PRD**。  
V0.1+ 须改为：创建脱敏任务 → 工作台 UI → `maskingConfirmed` 后才允许 `publishCaseFromAlbum`。

## 8.1 脱敏形态演进（产品取向，2026-05-21 确认）

### 目标态（后台能力）

理想情况下，脱敏是 **平台后台异步能力**，而非商家/车主的日常操作步骤：

```text
业务侧提交审核 → 后台自动脱敏 → 前端仅「预览 + 修补」→ 确认后公示
```

这样可降低操作负担，并与审核、存储 SLA 统一调度。

### 当前折中（脱敏 V1，P0）

因自动脱敏的 **时效性与准确率仍不稳定**，V1 采用 **脱敏工作台** 作为过渡，而非在业务页勾选即提交：

| 能力 | V1 | 说明 |
|---|---|---|
| 提交审核入口 | 商家新建案例页 | 先 **存草稿** 拿到 `albumId`，再进入工作台 |
| 一键 AI 脱敏 | ✅ | 工作台主路径 |
| 手工打码 | V2 | V1 入口占位，提示后续开放 |
| 预览对比 + 责任确认 | ✅ | 确认后才写入 `maskingConfirmed` 并真正提交审核/发布 |
| 用户端工作台路由 | **暂不注册** | 与用户端同组件；V2 `order_authorize` 再开 `pages/desensitize/workbench` |

后续可演进为：**提交后后台任务** → 消息/列表通知 → **仅打开预览确认页**（复用 `DesensitizeWorkbench` UI，弱化「操作台」心智）。

### 与「提交审核」按钮的关系（商家历史案例）

- **新建页「提交审核」**：保存草稿并 **跳转脱敏工作台**（非直接进案例库）。
- **工作台「确认脱敏结果并提交审核」**：完成闸门后，才触发 `saveAlbum(submitReview)` 与案例发布。

## 9. 验收标准

1. 商家端与用户端打开同一 `taskId`，工作台 UI 与操作步骤一致（仅文案/按钮后续动作不同）。
2. 未 `maskingConfirmed` 的案例，用户端案例 Tab **不可展示**。
3. C 端接口与页面 **无 raw 图 URL**。
4. 商家历史案例流程中，**不存在**「仅勾选隐私即提交审核」的路径。
5. 文档与 `00_开发优先级.md` 中脱敏项状态可追踪。
