## **2. 定位与边界（必读）**

### **2.1 辙见相册是什么**

服务相册是 **维修过程留痕工具**，用于：

- 记录六阶段过程与凭证；  
- 为车主提供 **可翻阅的服务档案**；  
- 在用户授权后，经脱敏与审核发布为 **公开案例**。

### **2.2 辙见相册不是什么**

| 不做 | 说明 |
| --- | --- |
| **经营审批系统** | 不代替门店对增项报价、收款、合同效力的内部审批 |
| **法务 / 电子签平台** | Phase 1 **不提供** 增项电子签名、不验证签字真伪、不因缺签字 **阻断完工** |
| **鉴定 / 仲裁** | 不判定增项是否合理、不调解门店与车主费用争议 |

> **增项是否有效、是否收费、是否已获客户同意** — 由 **门店线下流程** 负责；相册只要求商家把 **已发生的确认过程** 尽量 **拍照留痕**。

### **2.3 Phase 1 产品决策（2026-07-10）**

| 决策 | 说明 |
| --- | --- |
| **不开发** 阶段四「+ 添加方案外增项」独立表单向导 | 避免相册过度介入经营；减少与 `planParts` 双轨维护 |
| **统一路径** | 增项 = **阶段三补传经客户确认的报价/增项单** + **更新 `planParts`** + **阶段四按目录登记** |
| **数据层保留** | `partsJson[].source = extra` 与验真页「方案外增项」分组 **仍支持**（API/验真已就绪）；Phase 1 商家端 **不主推** 该入口 |

## **3. 术语**

| 术语 | 含义 |
| --- | --- |
| **方案内配件** | 首次锁定 `planParts[]` 时已在报价目录中的更换件 |
| **方案外增项** | 施工过程中 **新增** 的工项/配件，未出现在 **当前锁定前** 的 `planParts` 中 |
| **增项单 / 补充报价单** | 载明新增项目与金额的门店单据（DMS 打印、手写签字单、微信确认截图等） |
| **经客户确认** | 门店线下已取得客户 **明示同意** 的凭证（见 §4.2）；相册 **不审核** 真伪 |

## **4. 何时走本 SOP**

在以下场景，商家应在相册中 **补充留痕**（不阻断施工，建议在 **交车前** 完成）：

| 场景 | 典型例子 |
| --- | --- |
| 拆检后发现额外损坏 | 定损后才发现副车架变形，需加换件 |
| 施工中升级配件 | 原方案副厂件，客户同意改品牌件 |
| 追加保养项 | 施工时建议更换变速箱油，客户同意 |
| 事故车追加钣金/喷漆面积 | 扩损后与保险公司或客户确认的新增工项 |

**不必走本 SOP**：纯工时调整且 **无新增更换件**、且 **不变更** 已锁定 `planParts` 行 — 可在阶段三 **方案说明** 或阶段六 **结算单** 文字留痕即可。

## **5. 商家标准操作（SOP）**

### **5.1 总流程**

```text
线下取得客户对增项的确认（门店责任）
        ↓
阶段三 · 上传「最新报价 / 增项单」照片（须能体现客户确认）
        ↓
更新 planAmount（如总额变化）→ 重新识别或手工维护 planParts → 再次锁定目录
        ↓
阶段四 · 对新增/变更的 planPart 行逐件上传配件凭证（类型 + 凭证图）
        ↓
（建议）阶段五 · 施工工单留痕；（建议）阶段六 · 结算单与方案总额交叉核对
        ↓
标记完工 → 车主可在验真页按编码自愿核对
```

### **5.2 Step 0 · 线下确认（门店，相册外）**

| 项 | 要求 |
| --- | --- |
| 沟通 | 向客户说明增项原因、项目、单价/总价、工期影响 |
| 凭证形式（任选其一或组合） | ① 客户签字的书面增项单/报价单 ② DMS/系统 **已确认** 打印件 ③ 微信/短信 **客户明确回复同意** 的截图（须含项目与金额） |
| 平台 | **不要求** 指定模板；**不** 提供电子签入口 |

### **5.3 Step 1 · 阶段三「方案与报价」留痕（必做）**

| 步骤 | 操作 | 说明 |
| ---: | --- | --- |
| 1 | 进入相册 **阶段三** | 节点：`stage_3` |
| 2 | **补传报价/增项单图** | 1～N 张；须包含 **新增项目** 及 **客户确认痕迹**（签字、系统「已确认」、或附注沟通截图） |
| 3 | 更新 **`planAmount`** | 若增项导致方案总额变化，修改单行参考报价，与线下一致 |
| 4 | 更新 **`planParts`** | **优先**：对新增报价单点击「识别报价表」→ 在 **确认台** 核对后锁定；**或** 在确认台 **手工新增行**（OCR 失败时） |
| 5 | **再次锁定** 方案配件目录 | 若此前已锁定：须 **解锁** → 更新 → 锁定（`planPartsRevision++`） |

**合规提示（商家端已有文案）**：  
「识别结果仅供参考，请以你确认的报价表为准；确认后将锁定方案配件目录，供阶段四录入与车主验真参考。」

**交叉校验**：OCR 行合计 ≠ `planAmount` 时，系统 **黄条提醒**，不自动改价 — 商家须人工核对。

### **5.4 Step 2 · 阶段四「配件凭证」登记（有更换件时必做）**

| 步骤 | 操作 | 说明 |
| ---: | --- | --- |
| 1 | 打开 **配件清单向导** | 以 **最新锁定** 的 `planParts` 为清单 |
| 2 | 对 **新增行** 与其它未完成行 | 选择标准类型 + 上传 ≥1 张凭证图（包装/编码/实物） |
| 3 | （选填）识别编码 | 使用「识别编码」OCR；商家核对后「保存本项」 |
| 4 | 检查进度 | 目标：`已完成 / planParts.length` |

**类型变更**：若增项导致某行配件类型与 **原** 报价不一致，须已在阶段三上传 **最新报价单**；阶段四对 **OCR 锁定类型** 的行不可直接改类型（见 `part-verify-copy.js` 引导）。

### **5.5 Step 3 · 阶段五 / 六（建议）**

| 节点 | 建议留痕 | 作用 |
| --- | --- | --- |
| **阶段五 · 施工过程** | 上传 **施工工单**（含增项工项） | 与报价时间线交叉印证（见 `album-evidence-guide` · `work_order`） |
| **阶段六 · 完工交付** | 上传 **结算单** | 车主可核对结算与方案/工单是否一致 |

### **5.6 Step 4 · 完工前自检**

| 检查项 | 通过标准 |
| --- | --- |
| 增项单/最新报价单已上传阶段三 | 图清晰可读，含新增项 |
| `planParts` 含新增更换件 | 已锁定 |
| 阶段四凭证 | 新增件均有类型 + 凭证图 |
| `planAmount` | 与线下确认总额一致（允许黄条提醒后人工确认） |
| 车主手机号已关联 | 仍满足 Phase 1 完工前置 |

## **6. 车主侧可见性**

| 能力 | 增项相关表现 |
| --- | --- |
| 相册时间线 | 阶段三可见 **更新后的报价图** 与方案说明 |
| 配件验真 | 以 **最新 `planParts`** 为方案参考；相册登记在验真卡展示 |
| 「方案外增项」分组 | 仅当存在 `source: extra` 且 **未绑定** `planPartId` 的 `partsJson` 行时出现；**Phase 1 正常路径下应无或极少**（增项已并入 `planParts` 时为 `plan_linked`） |
| 平台承诺 | **不鉴定** 增项是否合理；验真仅 **编码查询 vs 登记类型** 自愿留痕 |

## **7. 与验真 `linkStatus` 的关系**

| 状态 | 常见原因 | 商家处理 |
| --- | --- | --- |
| `plan_only` | 方案目录有行，阶段四未登记 | 补传阶段四凭证 |
| `album_only` | 相册有件，方案无对应行（历史 extra 或未更新 planParts） | **按本 SOP** 回阶段三补报价单并更新 `planParts` |
| `field_diff` | 类型/品牌/编码与方案不一致 | 核对是否未上传最新报价单；与车主沟通后更新阶段三 |
| `linked` | 方案与相册 1:1 | 正常 |

## **9. 常见问题（FAQ）**

**Q1：客户只在微信口头同意，没有签字单怎么办？**  
A：将 **微信聊天记录截图**（含项目、金额、明确同意）作为阶段三 **报价/增项单** 附件图上传，并在方案说明中简述。平台不验证真伪，但留痕优于无留痕。

**Q2：增项发生在阶段四已经录了一半之后怎么办？**  
A：暂停阶段四 → 回阶段三 **解锁 planParts** → 上传增项单 → 更新并 **重新锁定** → 再继续阶段四向导。已保存的 `partsJson` 行与仍存在的 `planPartId` 会自动对齐。

**Q3：只有工时增项、没有新配件，还要改 planParts 吗？**  
A：若无新增 `planParts` 行，可不改动目录；在阶段三 **方案说明** 或阶段六 **结算单** 留痕即可。若报价单 OCR 会误识别工时行，确认台 **删除纯工时行** 后再锁定。

**Q4：为什么不做「添加方案外增项」按钮？**  
A：双轨维护（planParts + extra 表单）易与验真 **方案侧** 脱节，且相册 **不应** 扮演增项审批。统一走 **报价单 → planParts** 更符合「证据先于说服」。

**Q5：API 里的 `source: extra` 还会用吗？**  
A：数据模型与验真分组 **保留**，供历史数据与未来 Phase 2；Phase 1 商家端 **不推广** 该录入方式，运营/开发 **不** 为其实现独立 WXML 向导。
