# **《评价与奖励PRD》**

---

## **1. 文档信息**


| 项目     | 内容                                |
| ------ | --------------------------------- |
| 文档名称   | 评价与奖励PRD                          |
| 当前版本   | V1.0                              |
| 适用端    | 用户端小程序、商家工作台、平台运营工作台              |
| 核心对象   | 用户评价、六维评分、评价奖励、案例公开奖励             |
| 核心目标   | 鼓励用户提交真实评价，提升平台可信度，同时通过奖励机制促进内容沉淀 |
| MVP 原则 | 不做复杂返佣，不做浏览收益，不做多级分销              |


---

# **2. 模块定位**

评价与奖励模块服务于三个目标：

1. **提高门店可信度**
  - 用户真实评价；
  - 六维评分；
  - 文字评价；
  - 可选图片评价。
2. **提升服务透明度**
  - 用户评价维修过程是否透明；
  - 评价价格是否清晰；
  - 评价商家是否按约履约。
3. **促进内容资产沉淀**
  - 鼓励用户评价；
  - 鼓励用户授权公开脱敏案例；
  - 为平台案例库、门店主页、服务页提供可信内容。

---

# **3. 核心原则**

## **3.1 奖励不是返佣**

平台奖励定位为：

```
真实评价激励
透明维修内容贡献激励

```

不是：

1. 用户拉新返佣；
2. 浏览收益；
3. 点击收益；
4. 多级分销；
5. 商家私下返现；
6. 诱导好评。

---

## **3.2 不允许诱导好评**

商家不得要求用户：

1. 必须给 5 星；
2. 必须写好评才返钱；
3. 删除差评；
4. 使用指定文案；
5. 线下返现换好评；
6. 以奖励为由干扰真实评价。

平台评价奖励只与“是否提交真实有效评价”相关，不与好评或差评直接绑定。

---

## **3.3 奖励发放需风控**

以下情况奖励不发或冻结：

1. 订单未完成；
2. 订单退款中；
3. 订单售后中；
4. 评价内容明显无效；
5. 疑似刷单；
6. 用户或商家异常；
7. 重复评价；
8. 平台运营判定风险。

---

# **4. 评价触发时机**

## **4.1 用户评价入口**

评价入口出现在：

1. 订单完成页；
2. 订单详情页；
3. 微信服务通知；
4. 我的订单列表；
5. 我的维修档案页；
6. 评价奖励中心。

---

## **4.2 触发条件**

订单进入以下状态后可评价：

```
已完成

```

包括：

1. 用户主动确认完成；
2. 系统自动完成。

不可评价状态：

1. 待支付；
2. 待接单；
3. 维修中；
4. 待确认；
5. 已取消；
6. 已关闭；
7. 已退款；
8. 售后中，建议暂不可评价或评价后隐藏。

# **5. 评价内容设计**

## **5.1 六维评分**

用户评价采用六维评分。


| 维度     | 说明               |
| ------ | ---------------- |
| 服务态度   | 接待、沟通、响应是否良好     |
| 专业程度   | 检查、维修、解释是否专业     |
| 价格透明   | 报价、费用、增项是否清晰     |
| 维修过程透明 | 是否能看到维修过程、相册是否清楚 |
| 完工效果   | 修复效果、交付质量        |
| 是否愿意推荐 | 综合推荐意愿           |


评分方式：

```
1-5 星

```

---

## **5.2 综合评分计算**

建议综合评分计算方式：

```
综合评分 = 六维评分平均值

```

后续可加权：


| 维度     | 权重建议 |
| ------ | ---- |
| 服务态度   | 15%  |
| 专业程度   | 20%  |
| 价格透明   | 20%  |
| 维修过程透明 | 20%  |
| 完工效果   | 20%  |
| 是否愿意推荐 | 5%   |


MVP 可直接平均。

---

## **5.3 文字评价**

字段：


| 字段    | 说明                   |
| ----- | -------------------- |
| 评价内容  | 用户输入                 |
| 最少字数  | MVP 可不强制，建议提示 10 字以上 |
| 最大字数  | 建议 500 字             |
| 敏感词检测 | 必须                   |
| 隐私检测  | 必须                   |
| 是否匿名  | 可选，默认脱敏昵称            |


---

## **5.4 图片评价**

MVP 可选支持。

若支持：


| 项目   | 规则                 |
| ---- | ------------------ |
| 图片数量 | 最多 9 张             |
| 图片类型 | jpg/png/webp       |
| 图片隐私 | 必须脱敏或审核            |
| 图片用途 | 展示在评价区，可用于案例补充，需授权 |
| 视频   | MVP 不支持            |

# **6. 评价提交流程**

```
订单完成
→ 用户进入评价页
→ 选择六维评分
→ 填写文字评价
→ 上传图片，可选
→ 提交评价
→ 系统风控检测
→ 评价进入正常/待审核状态
→ 触发奖励计算
→ 引导用户授权公开案例

```

入口：

```

→ 用户评价
→ 评价详情
```

## **9.2 回复规则**

1. 每条评价商家可回复一次；
2. 回复内容需经过敏感词检测；
3. 不得泄露用户隐私；
4. 不得辱骂用户；
5. 不得诱导用户修改评价；
6. 争议评价可申请平台介入。

---

## **9.3 回复状态**

| 隐藏  | 平台隐藏 |
| 删除  | 平台删除 |

# **10. 评价奖励设计**

## **10.1 奖励类型**

MVP 支持两类奖励：

1. **评价奖励**
  - 用户完成真实有效评价后获得；
  - 与好评差评无关；
  - 与订单复杂度、金额、车辆价值等有关。
2. **公开授权奖励**
  - 用户同意将维修记录脱敏公开为案例后获得；
  - 需通过脱敏和平台审核；
  - 若用户后续申请下架，奖励是否追回由平台规则决定，MVP 可不追回但限制频繁下架。

---

## **10.2 不做的奖励**

MVP 明确不做：

1. 分享点击奖励；
2. 浏览量奖励；
3. 转发收益；
4. 邀请商家返佣；
5. 多级推广收益；
6. 商家私设红包返现；
7. 用户拉用户分销。

---

# **11. 奖励计算规则**

## **11.1 评价奖励基础公式**

建议公式：

```
评价奖励 = 基础奖励 × 服务复杂度系数 × 订单金额系数 × 风控系数

```

其中：

- 基础奖励：平台配置；
- 服务复杂度系数：L1-L4；
- 订单金额系数：按订单金额区间；
- 风控系数：正常为 1，异常为 0 或冻结。

---

## **11.2 服务复杂度系数**


| 服务等级 | 示例           | 系数建议 |
| ---- | ------------ | ---- |
| L1   | 简单检测、雨刮      | 0.5  |
| L2   | 保养、轮胎、电瓶、刹车  | 1.0  |
| L3   | 底盘、空调、复杂故障   | 1.5  |
| L4   | 事故车、大修、新能源三电 | 2.0  |


---

## **11.3 订单金额系数**

标准订单按实付金额计算。

复杂预约若无线上支付，则可由商家填写价格区间，平台按区间中位数或服务等级估算。


| 金额区间      | 系数建议 |
| --------- | ---- |
| 0-299     | 0.5  |
| 300-999   | 1.0  |
| 1000-2999 | 1.5  |
| 3000 以上   | 2.0  |


---

## **11.4 车辆价值系数，可选**

MVP 可不做或简单配置。

如做：


| 车辆类型      | 系数         |
| --------- | ---------- |
| 普通家用车     | 1.0        |
| 豪华品牌      | 1.2        |
| 新能源车型     | 1.2        |
| 高端豪华/稀有车型 | 1.5，但需更高风控 |


---

## **11.5 奖励封顶**

必须配置封顶。

建议：


| 奖励类型      | 单次封顶         |
| --------- | ------------ |
| 评价奖励      | 5-20 元，运营配置  |
| 公开授权奖励    | 10-50 元，运营配置 |
| 单用户每日奖励   | 配置           |
| 单用户每月奖励   | 配置           |
| 单门店每日奖励总额 | 配置           |


MVP 可将金额设为运营后台配置，不在前端写死。

# **12. 奖励状态**


| 状态   | 说明           |
| ---- | ------------ |
| 待计算  | 用户已评价，系统待计算  |
| 待风控  | 等待风控规则检查     |
| 待审核  | 需平台人工审核      |
| 待发放  | 已通过，等待发放     |
| 已发放  | 奖励到账         |
| 已冻结  | 售后/退款/风控原因冻结 |
| 已驳回  | 不符合奖励条件      |
| 已失效  | 超时未领取或订单关闭   |
| 发放失败 | 微信支付或账户异常    |

# **13. 奖励发放方式**

## **13.1 MVP 推荐方式**

MVP 可选择以下之一：

1. **平台余额/奖励金账户**
  - 用户可在小程序查看；
  - 后续可提现或抵扣；
  - 技术上更可控，但涉及资金账户合规，需要谨慎。
2. **微信零钱/商家转账到零钱**
  - 用户感知强；
  - 需要微信支付商家转账能力；
  - 风控和实名要求更高。
3. **优惠券**
  - 合规压力较小；
  - 可促进复购；
  - 但用户即时激励较弱。

建议 MVP 初期：

```
先以“奖励记录 + 运营人工发放/优惠券”方式验证，再接入自动转账。

```

---

## **13.2 发放流程**

```
评价提交
→ 生成奖励记录
→ 系统计算金额
→ 风控检测
→ 平台审核，必要时
→ 发放奖励
→ 用户收到通知
→ 奖励状态变为已发放

```

# **14. 风控规则**

## **14.1 用户维度**

命中以下情况进入待审核或驳回：

1. 同一用户短期大量评价；
2. 同一设备多个账号频繁评价；
3. 同一手机号关联异常订单；
4. 评价文字重复度高；
5. 评价与订单服务不匹配；
6. 评价后立即退款；
7. 多次公开后申请下架；
8. 用户与商家疑似关联。

---

## **14.2 商家维度**

命中以下情况限制奖励：

1. 同一门店短期订单评价异常集中；
2. 好评率异常过高且内容相似；
3. 多用户同 IP/设备；
4. 商家被投诉诱导好评；
5. 门店存在刷单风险；
6. 商家线下返现行为被举报。

---

## **14.3 订单维度**

以下订单奖励冻结：

1. 售后中；
2. 退款中；
3. 已退款；
4. 订单金额异常；
5. 商家拒单后异常完成；
6. 无履约证据；
7. 复杂维修无任何说明或相册。

# **17. P0 验收标准**

必须满足：

1. 订单完成后用户可评价；
2. 评价包含六维评分；
3. 用户可填写文字评价；
4. 用户可选择匿名评价；
5. 评价与订单、门店、服务项目关联；
6. 用户提交评价后生成奖励记录；
7. 奖励金额支持平台配置；
8. 奖励状态用户可见；
9. 售后/退款订单奖励可冻结；
10. 平台可审核评价和奖励；
11. 平台可驳回异常奖励；
12. 商家可查看用户评价；
13. 商家回复如上线，需支持审核和隐藏；
14. 不允许将奖励绑定为好评奖励；
15. 所有奖励操作需留痕。
