# **2. 核心业务实体域**

平台核心数据分为 10 个实体域：


| 实体域    | 说明                   |
| ------ | -------------------- |

# **3. 核心实体关系总览**

## **3.1 主链路关系**

```
用户 user
  └── 车辆 vehicle
        └── 订单 order
              ├── 支付单 payment
              ├── 退款单 refund
              ├── 预约 appointment
              ├── 服务相册 repair_album
              │     ├── 相册节点 album_node
              │     └── 媒体素材 media_asset
              ├── 评价 review
              │     └── 奖励 reward
              └── 案例 case_content
                    └── H5公开页面 content_page

```

---

## **3.2 商家侧关系**

```
商家 merchant
  └── 门店 store
        ├── 服务商品 service_item
        ├── 订单 order
        ├── 服务相册 repair_album
        ├── 历史案例 case_content
        ├── 用户评价 review
        └── 结算记录 settlement

```

---

## **3.3 审核与风控关系**

```
业务对象 business_object
  ├── 审核任务 audit_task
  │     └── 审核日志 audit_log
  ├── 风险事件 risk_event
  │     └── 风控处理日志 risk_audit_log
  └── 操作日志 operation_log

```

业务对象可以是：

```
merchant / store / service_item / repair_album / case_content / review / reward / refund

```

# **5. 关键实体关系说明**

## **5.1 用户、车辆、订单**

关系：

```
一个用户可以有多辆车；
一辆车可以产生多个订单；
一个订单必须归属于一个用户；
订单可以绑定车辆，也可以允许用户临时填写车辆信息。

```

建议：

- MVP 阶段下单时可以不强制先创建完整车辆档案；
- 但订单中必须沉淀车辆快照，避免后续用户修改车辆档案导致历史订单信息变化。

---

## **5.2 商家、门店、服务商品**

关系：

```
一个商家可以有多个门店；
一个门店可以上架多个服务商品；
一个服务商品必须归属于一个商家或门店；
服务商品需要经过平台审核后才能上架。

```

建议：

- 如果早期一个商家只有一个门店，也仍然保留 store 维度；
- 后续多门店连锁可以自然扩展。

---

## **5.3 订单、支付、退款**

关系：

```
一个订单可以对应一个或多个支付单；
一个支付单可以对应一个或多个退款单；
退款金额总和不得超过实付金额；
订单状态、支付状态、退款状态需要分离。

```

重要原则：

```
order_status 不等于 pay_status，不等于 refund_status。

```

示例：

```
订单状态：WAIT_ACCEPT / ACCEPTED / IN_SERVICE / COMPLETED
支付状态：UNPAID / PAID / REFUNDED / PART_REFUNDED
退款状态：NONE / REFUNDING / REFUND_SUCCESS / REFUND_FAILED

```

---

## **5.4 订单、相册、案例**

关系：

```
一个订单可以生成一个服务相册；
一个服务相册可以生成一个案例草稿；
一个案例草稿通过审核后成为公开案例；
公开案例可以生成 H5 页面。

```

独立历史案例：

```
商家可以不依赖平台订单创建历史案例相册；
历史案例相册也可以生成公开案例；
但必须经过脱敏和平台审核。

```

---

## **5.5 评价、奖励、分享收益**

关系：

```
一个订单最多有一个主评价；
一个评价可以触发一条奖励记录；
一个公开案例或分享链接可以产生分享记录；
分享转化成功后可以产生分佣或收益记录。

```

限制：

- 订单未完成不得评价；
- 退款中、售后中不得发放奖励；
- 分享收益需要有明确归因记录；
- 奖励和收益必须有风控状态。

# **8. 隐私与敏感字段原则**

以下字段需要加密、脱敏或限制访问：


| 字段        | 处理方式           |
| --------- | -------------- |
| 手机号       | 加密存储，展示脱敏      |
| 车牌号       | 加密存储，默认脱敏展示    |
| VIN       | 加密存储，默认不展示     |
| 身份证号      | 原则上不采集，如采集必须加密 |
| 行驶证图片     | 私有存储，不得公开      |
| 原始维修图片    | 私有存储，公开使用脱敏版本  |
| 微信 openid | 不对前端无关场景暴露     |
| 精确地址      | 后台权限控制         |

# **9. 数据一致性原则**

## **9.1 交易类数据**

交易类数据以业务表为准：

```
订单金额以 order / order_item 为准；
实付金额以 payment_order 为准；
退款金额以 refund_order 为准；
结算金额以 settlement 为准；
奖励金额以 reward_record 为准。

```

埋点数据只用于分析，不作为财务依据。

---

## **9.2 状态流转**

所有关键状态流转必须写入日志：


| 状态对象 | 日志表                              |
| ---- | -------------------------------- |
| 订单   | order_status_log                 |
| 支付   | payment_log，可并入 payment_order 扩展 |
| 退款   | refund_log，可并入 refund_order 扩展   |
| 审核   | audit_log                        |
| 风控   | risk_audit_log                   |
| 操作   | operation_log                    |


---

# **10. P0 验收标准**

必须满足：

1. 核心实体关系清晰；
2. 用户、车辆、商家、门店、服务、订单、相册、案例、评价、奖励都有对应数据域；
3. 订单、支付、退款状态分离；
4. 相册可以从订单生成，也可以从历史案例生成；
5. 案例必须支持公开状态、审核状态、脱敏状态；
6. 评价奖励必须可追溯到订单；
7. H5公开内容不得直接读取原始隐私数据；
8. 支付、退款、奖励、结算类数据必须可对账；
9. 审核、风控、后台操作必须留痕；
10. 后续具体表设计可以基于本 ER 总览继续拆分。
