## **2. 状态机设计原则**

所有状态机必须遵循：

1. 状态值使用大写英文枚举；
2. 状态只允许按合法路径流转；
3. 关键状态变更必须写日志；
4. 状态变更接口必须支持幂等；
5. 交易类状态必须支持对账；
6. 订单状态、支付状态、退款状态必须分离；
7. 审核类状态必须记录审核人、审核时间、审核备注；
8. 风控类状态不得被普通业务流程绕过；
9. 已完成、已关闭、已取消等终态原则上不可逆；
10. 状态机变更必须同步更新 API 文档和前端枚举。

> **V2.0 实施范围**：商家入驻（§3）、服务审核（§4）、相册（§7）、案例（§8）、**咨询线索**（见 `05_咨询线索状态机.md`）。**§5–6、§9–12 不实现**。

# **3. 商家入驻状态机**

## **3.1 适用对象**

```
merchant
merchant_certification
store

```

---

## **3.2 商家状态**


| 状态             | 说明         |
| -------------- | ---------- |
| DRAFT          | 草稿，商家资料未提交 |
| PENDING_AUDIT  | 待审核        |
| AUDIT_REJECTED | 审核驳回       |
| ACTIVE         | 正常营业/启用    |
| SUSPENDED      | 暂停合作/冻结    |
| CLOSED         | 关闭/注销      |


---

## **3.3 状态流转**

```
DRAFT
  → PENDING_AUDIT
  → ACTIVE
  → SUSPENDED
  → ACTIVE
  → CLOSED

PENDING_AUDIT
  → AUDIT_REJECTED
  → DRAFT

AUDIT_REJECTED
  → DRAFT
  → PENDING_AUDIT

```

---

## **3.4 触发动作**


| 动作   | 当前状态                   | 目标状态           | 操作方 |
| ---- | ---------------------- | -------------- | --- |
| 保存资料 | -                      | DRAFT          | 商家  |
| 提交审核 | DRAFT / AUDIT_REJECTED | PENDING_AUDIT  | 商家  |
| 审核通过 | PENDING_AUDIT          | ACTIVE         | 平台  |
| 审核驳回 | PENDING_AUDIT          | AUDIT_REJECTED | 平台  |
| 暂停商家 | ACTIVE                 | SUSPENDED      | 平台  |
| 恢复商家 | SUSPENDED              | ACTIVE         | 平台  |
| 关闭商家 | ACTIVE / SUSPENDED     | CLOSED         | 平台  |


---

## **3.5 日志要求**

必须写入：

```
audit_task
audit_log
operation_log

```

```

---

## **4.2 服务商品状态**

### **审核状态 audit_status**


| 状态            | 说明   |
| ------------- | ---- |
| DRAFT         | 草稿   |
| PENDING_AUDIT | 待审核  |
| APPROVED      | 审核通过 |
| REJECTED      | 审核驳回 |


### **上架状态 sale_status**


| 状态        | 说明     |
| --------- | ------ |
| OFFLINE   | 下架     |
| ONLINE    | 上架     |
| SUSPENDED | 平台强制下架 |


---

## **4.3 状态流转**

```
DRAFT
  → PENDING_AUDIT
  → APPROVED
  → ONLINE

PENDING_AUDIT
  → REJECTED
  → DRAFT

ONLINE
  → OFFLINE
  → ONLINE

ONLINE / OFFLINE
  → SUSPENDED

```

---

## **4.4 触发动作**


| 动作     | 当前状态               | 目标状态          | 操作方 |
| ------ | ------------------ | ------------- | --- |
| 创建服务   | -                  | DRAFT         | 商家  |
| 提交审核   | DRAFT / REJECTED   | PENDING_AUDIT | 商家  |
| 审核通过   | PENDING_AUDIT      | APPROVED      | 平台  |
| 审核驳回   | PENDING_AUDIT      | REJECTED      | 平台  |
| 上架     | APPROVED / OFFLINE | ONLINE        | 商家  |
| 下架     | ONLINE             | OFFLINE       | 商家  |
| 平台强制下架 | ONLINE / OFFLINE   | SUSPENDED     | 平台  |
| 恢复     | SUSPENDED          | OFFLINE       | 平台  |


---

## **4.5 约束**

1. 未审核通过不得上架；
2. 商家被冻结时服务商品不可上架；
3. 服务价格变更后可根据规则重新审核；
4. 强制下架必须记录原因。

# **5. 订单状态机**（V1.0 遗留，V2.0 不实现）

## **5.1 适用对象**

```
order
order_status_log
fulfillment_record
appointment

```

---

## **5.2 订单主状态 order_status**


| 状态           | 说明    |
| ------------ | ----- |
| CREATED      | 已创建   |
| WAIT_PAY     | 待支付   |
| WAIT_ACCEPT  | 待商家接单 |
| ACCEPTED     | 商家已接单 |
| APPOINTED    | 已预约   |
| IN_SERVICE   | 服务中   |
| WAIT_CONFIRM | 待确认完成 |
| COMPLETED    | 已完成   |
| CANCELLED    | 已取消   |
| CLOSED       | 已关闭   |


---

## **5.3 状态流转**

```
CREATED
  → WAIT_PAY
  → WAIT_ACCEPT
  → ACCEPTED
  → APPOINTED
  → IN_SERVICE
  → WAIT_CONFIRM
  → COMPLETED

WAIT_PAY
  → CANCELLED
  → CLOSED

WAIT_ACCEPT
  → CANCELLED
  → CLOSED

ACCEPTED / APPOINTED
  → CANCELLED

IN_SERVICE
  → WAIT_CONFIRM

WAIT_CONFIRM
  → COMPLETED

```

---

## **5.4 触发动作**


| 动作     | 当前状态                              | 目标状态         | 操作方   |
| ------ | --------------------------------- | ------------ | ----- |
| 创建订单   | -                                 | CREATED      | 用户    |
| 确认下单   | CREATED                           | WAIT_PAY     | 用户    |
| 支付成功   | WAIT_PAY                          | WAIT_ACCEPT  | 系统    |
| 商家接单   | WAIT_ACCEPT                       | ACCEPTED     | 商家    |
| 确认预约   | ACCEPTED                          | APPOINTED    | 商家/用户 |
| 开始服务   | ACCEPTED / APPOINTED              | IN_SERVICE   | 商家    |
| 提交完工   | IN_SERVICE                        | WAIT_CONFIRM | 商家    |
| 用户确认完成 | WAIT_CONFIRM                      | COMPLETED    | 用户    |
| 系统自动完成 | WAIT_CONFIRM                      | COMPLETED    | 系统    |
| 用户取消   | WAIT_PAY / WAIT_ACCEPT / ACCEPTED | CANCELLED    | 用户    |
| 商家取消   | WAIT_ACCEPT / ACCEPTED            | CANCELLED    | 商家    |
| 超时关闭   | WAIT_PAY                          | CLOSED       | 系统    |


---

## **5.5 订单状态与支付退款关系**

订单状态不等于支付状态，也不等于退款状态。

```
order_status 独立维护交易履约状态；
pay_status 独立维护支付状态；
refund_status 独立维护退款状态。

```

---

## **5.6 日志要求**

每次订单状态变化必须写：

```
order_status_log
operation_log，可选

```

日志字段至少包括：


| 字段            | 说明     |
| ------------- | ------ |
| order_id      | 订单 ID  |
| from_status   | 原状态    |
| to_status     | 新状态    |
| action        | 触发动作   |
| operator_type | 操作人类型  |
| operator_id   | 操作人 ID |
| remark        | 备注     |
| created_at    | 创建时间   |

# **6. 支付退款状态机**

---

## **6.1 支付状态机**

### **适用对象**

```
payment_order

```

### **支付状态 pay_status**


| 状态         | 说明   |
| ---------- | ---- |
| UNPAID     | 未支付  |
| PAYING     | 支付中  |
| PAID       | 已支付  |
| PAY_FAILED | 支付失败 |
| CLOSED     | 已关闭  |


### **状态流转**

```
UNPAID
  → PAYING
  → PAID

PAYING
  → PAY_FAILED
  → PAYING

UNPAID / PAYING
  → CLOSED

```

### **触发动作**


| 动作     | 当前状态            | 目标状态       | 操作方    |
| ------ | --------------- | ---------- | ------ |
| 创建支付单  | -               | UNPAID     | 系统     |
| 调起支付   | UNPAID          | PAYING     | 用户     |
| 支付成功回调 | PAYING / UNPAID | PAID       | 第三方/系统 |
| 支付失败   | PAYING          | PAY_FAILED | 第三方/系统 |
| 重新支付   | PAY_FAILED      | PAYING     | 用户     |
| 超时关闭   | UNPAID / PAYING | CLOSED     | 系统     |


### **约束**

1. 支付成功回调必须验签；
2. 支付成功回调必须幂等；
3. 支付金额必须与订单应付金额一致；
4. 支付成功后才能推动订单进入 `WAIT_ACCEPT`；
5. 已支付支付单不能再次支付。

---

## **6.2 退款状态机**

### **适用对象**

```
refund_order

```

### **退款状态 refund_status**


| 状态              | 说明    |
| --------------- | ----- |
| NONE            | 无退款   |
| REFUND_APPLYING | 退款申请中 |
| REFUNDING       | 退款中   |
| REFUND_SUCCESS  | 退款成功  |
| REFUND_FAILED   | 退款失败  |
| REFUND_CLOSED   | 退款关闭  |


### **状态流转**

```
NONE
  → REFUND_APPLYING
  → REFUNDING
  → REFUND_SUCCESS

REFUNDING
  → REFUND_FAILED
  → REFUNDING

REFUND_APPLYING / REFUND_FAILED
  → REFUND_CLOSED

```

### **触发动作**


| 动作        | 当前状态                            | 目标状态            | 操作方    |
| --------- | ------------------------------- | --------------- | ------ |
| 发起退款      | NONE                            | REFUND_APPLYING | 用户/平台  |
| 审核通过并提交三方 | REFUND_APPLYING                 | REFUNDING       | 系统     |
| 退款成功回调    | REFUNDING                       | REFUND_SUCCESS  | 第三方/系统 |
| 退款失败回调    | REFUNDING                       | REFUND_FAILED   | 第三方/系统 |
| 重新提交退款    | REFUND_FAILED                   | REFUNDING       | 系统     |
| 关闭退款      | REFUND_APPLYING / REFUND_FAILED | REFUND_CLOSED   | 平台/系统  |


### **约束**

1. 退款总金额不得超过实付金额；
2. 退款回调必须验签；
3. 退款回调必须幂等；
4. 退款成功后需要更新订单的 refund_status；
5. 全额退款和部分退款需要区分。

| 上传完成 | UPLOADING            | WAIT_DESENSITIZE | 商家/系统 |

| 归档   | APPROVED / PUBLISHED | ARCHIVED         | 系统/平台 |


---

## **7.5 约束**

1. 未脱敏素材不得公开；
2. 审核未通过不得生成公开案例；
3. 历史案例相册也必须走脱敏和审核；
4. 原始素材必须私有存储；
5. 相册节点必填校验通过后才能提交审核。

5. SEO 信息修改可触发重新审核。

## **10.2 奖励状态 reward_status**


| 状态            | 说明    |
| ------------- | ----- |
| CREATED       | 已创建   |
| RISK_CHECKING | 风控检查中 |
| FROZEN        | 已冻结   |
| WAIT_GRANT    | 待发放   |
| GRANTING      | 发放中   |
| GRANTED       | 已发放   |
| GRANT_FAILED  | 发放失败  |

---

## **10.3 状态流转**

```
CREATED
  → RISK_CHECKING
  → FROZEN
  → WAIT_GRANT
  → GRANTING

  → CANCELLED

  → GRANT_FAILED

## **10.4 触发动作**


| 动作    | 当前状态          | 目标状态          | 操作方 |
| ----- | ------------- | ------------- | --- |
| 创建奖励  | -             | CREATED       | 系统  |
| 发起风控  | CREATED       | RISK_CHECKING | 系统  |
| 风控通过  | RISK_CHECKING | FROZEN        | 系统  |
| 风控拒绝  | RISK_CHECKING | CANCELLED     | 系统  |
| 解冻待发放 | FROZEN        | WAIT_GRANT    | 系统  |
| 开始发放  | WAIT_GRANT    | GRANTING      | 系统  |
| 发放成功  | GRANTING      | GRANTED       | 系统  |
| 发放失败  | GRANTING      | GRANT_FAILED  | 系统  |
| 重新发放  | GRANT_FAILED  | GRANTING      | 系统  |


---

## **10.5 约束**

1. 退款中、售后中不得发放奖励；
2. 风控未通过不得发放；
3. 奖励金额必须可对账；
4. 发放接口必须幂等；
5. 奖励记录不得物理删除。

| INVALID       | 归因无效 |
| RISK_REJECTED | 风控拒绝 |


---

## **11.4 分佣状态 commission_status**


| 状态            | 说明   |
| ------------- | ---- |
| CREATED       | 已创建  |
| FROZEN        | 冻结中  |
| WAIT_SETTLE   | 待结算  |
| SETTLING      | 结算中  |
| SETTLED       | 已结算  |
| SETTLE_FAILED | 结算失败 |
| CANCELLED     | 已取消  |


---

## **11.5 分佣状态流转**

```
CREATED
  → FROZEN
  → WAIT_SETTLE
  → SETTLING
  → SETTLED

SETTLING
  → SETTLE_FAILED
  → SETTLING

FROZEN / WAIT_SETTLE
  → CANCELLED

```

---

## **11.6 触发动作**


| 动作       | 当前状态                 | 目标状态          | 操作方 |
| -------- | -------------------- | ------------- | --- |
| 创建分享     | -                    | CREATED       | 用户  |
| 激活分享     | CREATED              | ACTIVE        | 系统  |
| 访问/下单归因  | PENDING              | ATTRIBUTED    | 系统  |
| 归因无效     | PENDING              | INVALID       | 系统  |
| 创建分佣     | -                    | CREATED       | 系统  |
| 冻结收益     | CREATED              | FROZEN        | 系统  |
| 订单完成且无售后 | FROZEN               | WAIT_SETTLE   | 系统  |
| 开始结算     | WAIT_SETTLE          | SETTLING      | 系统  |
| 结算成功     | SETTLING             | SETTLED       | 系统  |
| 结算失败     | SETTLING             | SETTLE_FAILED | 系统  |
| 取消分佣     | FROZEN / WAIT_SETTLE | CANCELLED     | 系统  |


---

## **11.7 约束**

1. 订单未完成不得结算分佣；
2. 退款或售后中不得结算；
3. 归因必须有明确规则；
4. 同一订单不得重复产生有效分佣；
5. 分佣必须经过风控；
6. 分佣金额必须可对账。

# **12. 售后状态机**

## **12.1 适用对象**

```
aftersale_order
dispute_record
refund_order

```

---

## **12.2 售后状态 aftersale_status**


| 状态                   | 说明    |
| -------------------- | ----- |
| CREATED              | 已创建   |
| WAIT_MERCHANT_HANDLE | 待商家处理 |
| WAIT_PLATFORM_HANDLE | 待平台介入 |
| PROCESSING           | 处理中   |
| WAIT_USER_CONFIRM    | 待用户确认 |
| RESOLVED             | 已解决   |
| REJECTED             | 已拒绝   |
| CANCELLED            | 已取消   |
| CLOSED               | 已关闭   |


---

## **12.3 状态流转**

```
CREATED
  → WAIT_MERCHANT_HANDLE
  → PROCESSING
  → WAIT_USER_CONFIRM
  → RESOLVED
  → CLOSED

WAIT_MERCHANT_HANDLE
  → WAIT_PLATFORM_HANDLE
  → PROCESSING

WAIT_MERCHANT_HANDLE / PROCESSING
  → REJECTED
  → CLOSED

CREATED / WAIT_MERCHANT_HANDLE
  → CANCELLED

```

---

## **12.4 触发动作**


| 动作     | 当前状态                              | 目标状态                 | 操作方   |
| ------ | --------------------------------- | -------------------- | ----- |
| 发起售后   | -                                 | CREATED              | 用户    |
| 分配商家处理 | CREATED                           | WAIT_MERCHANT_HANDLE | 系统    |
| 商家受理   | WAIT_MERCHANT_HANDLE              | PROCESSING           | 商家    |
| 申请平台介入 | WAIT_MERCHANT_HANDLE / PROCESSING | WAIT_PLATFORM_HANDLE | 用户/商家 |
| 平台处理   | WAIT_PLATFORM_HANDLE              | PROCESSING           | 平台    |
| 提交处理结果 | PROCESSING                        | WAIT_USER_CONFIRM    | 商家/平台 |
| 用户确认解决 | WAIT_USER_CONFIRM                 | RESOLVED             | 用户    |
| 关闭售后   | RESOLVED / REJECTED / CANCELLED   | CLOSED               | 系统    |
| 拒绝售后   | WAIT_MERCHANT_HANDLE / PROCESSING | REJECTED             | 商家/平台 |
| 取消售后   | CREATED / WAIT_MERCHANT_HANDLE    | CANCELLED            | 用户    |


---

## **12.5 约束**

1. 售后处理中，奖励和分佣应冻结；
2. 涉及退款时必须创建退款单；
3. 售后处理过程必须留痕；
4. 平台介入必须记录处理人和结论；
5. 售后关闭后原则上不可重新打开，需新建售后单。

# **14. 状态变更接口要求**

所有状态变更接口必须满足：

1. 校验当前状态；
2. 校验目标状态；
3. 校验操作人权限；
4. 幂等处理；
5. 并发控制；
6. 写状态日志；
7. 写操作日志，关键后台操作必须写；
8. 返回最新状态；
9. 异常时不得出现半状态；
10. 涉及金额时必须事务处理。

---

# **15. P0 验收标准**

P0 阶段必须满足：

1. 商家入驻状态可闭环；
2. 服务商品审核、上下架状态可闭环；
3. 订单状态、支付状态、退款状态分离；
4. 支付成功可驱动订单进入待接单；
5. 退款成功可更新订单退款状态；
6. 相册支持上传、脱敏、审核、生成案例；
7. 案例支持草稿、审核、发布、下线；
8. 评价支持提交、审核、展示；
9. 奖励支持创建、风控、冻结、发放；
10. 分享收益支持归因、冻结、结算；
11. 售后处理中能冻结奖励和分佣；
12. 所有关键状态流转必须留痕。
