# **3. 用户相关对象**

---

## **3.1 User**

### **对象说明**

用户主对象，表示平台 C 端用户。

### **对应表**

```
user

```

### **核心职责**

1. 记录用户账号身份；
2. 绑定微信 openid / unionid；
3. 关联车辆、订单、评价、奖励；
4. 作为用户侧权限边界。

### **关键字段**


| 字段            | 类型       | 说明                 |
| ------------- | -------- | ------------------ |
| id            | string   | 用户 ID，例如 `usr_xxx` |
| openid        | string   | 微信 openid，加密或受控存储  |
| unionid       | string   | 微信 unionid，可选      |
| mobile        | string   | 手机号，加密存储           |
| nickname      | string   | 昵称                 |
| avatar_url    | string   | 头像                 |
| status        | string   | 用户状态               |
| registered_at | datetime | 注册时间               |
| last_login_at | datetime | 最近登录时间             |


### **主要关系**

```
User 1:N UserVehicle
User 1:N Order
User 1:N Review
User 1:N RewardRecord
User 1:N ShareRecord

```

如果后续需要平台标准服务和商家服务分离，可拆分为：

# **9. 对象边界原则**

## **9.1 订单对象边界**

订单只负责交易主状态，不直接承载支付、退款、评价、相册详情。

```
Order 管交易；
Payment 管支付；
Refund 管退款；
RepairAlbum 管维修过程；
Review 管用户评价。

```

---

## **9.2 相册与案例边界**

```
RepairAlbum 是过程资料；
CaseContent 是公开内容；
ContentPage 是 H5 展示页面。

```

公开页面不得直接读取原始媒体和隐私字段。

---

## **9.3 奖励与分佣边界**

```
RewardRecord 负责评价奖励；
CommissionRecord 负责分享收益；
AttributionRecord 负责转化归因；
WithdrawalRecord 负责提现。

```

---

# **10. P0 验收标准**

P0 阶段核心数据对象必须满足：

1. 用户、车辆、商家、门店、服务、订单、支付、退款对象清晰；
2. 相册、案例、公开页面边界清晰；
3. 评价、奖励、分享、归因、分佣链路可追踪；
4. 支付、退款、奖励、分佣金额均使用分作为单位；
5. 涉及隐私的数据对象必须支持脱敏或权限控制；
6. 所有关键对象必须有状态字段；
7. 所有关键状态变化必须有日志记录；
8. 订单必须沉淀车辆快照和服务快照；
9. 公开案例只能使用脱敏后的媒体素材；
10. 后续表设计可基于本文对象继续拆分。
