# **2. 订单状态设计原则**

订单状态机遵循以下原则：

1. 订单状态、支付状态、退款状态分离；
2. 所有状态变化必须有触发方；
3. 所有状态变化必须写入状态日志；
4. 支付成功不能只依赖前端返回，必须以支付回调或主动查询为准；
5. 已完成订单不能直接回退，只能进入售后或退款流程；
6. 已取消订单原则上不可恢复；
7. 退款不一定改变订单主状态，需要结合退款状态处理；
8. 奖励、结算、案例公开依赖订单最终状态。

# **3. 状态字段拆分**

| 状态           | 说明      |
| ------------ | ------- |
| CREATED      | 已创建     |
| WAIT_PAY     | 待支付     |
| WAIT_ACCEPT  | 待商家接单   |
| ACCEPTED     | 商家已接单   |
| WAIT_SERVICE | 待到店/待服务 |
| IN_SERVICE   | 服务中     |
| WAIT_CONFIRM | 待用户确认   |
| COMPLETED    | 已完成     |
| CANCELLED    | 已取消     |
| CLOSED       | 已关闭     |

# **4. 标准订单流程**

## **4.1 普通标准服务订单**

适用于：

- 小保养；
- 电瓶更换；
- 轮胎更换；
- 空调清洗；
- 刹车片/刹车盘等相对标准化服务。

流程：

```
用户提交订单
→ CREATED
→ WAIT_PAY
→ 用户支付成功
→ WAIT_ACCEPT
→ 商家接单
→ ACCEPTED
→ 用户到店/商家开始服务
→ IN_SERVICE
→ 商家提交完工
→ WAIT_CONFIRM
→ 用户确认完成/系统自动确认
→ COMPLETED

```

---

## **4.2 复杂维修预约订单**

适用于：

- 钣喷；
- 事故维修；
- 疑难故障检测；
- 需要到店报价的服务。

流程：

```
用户提交预约
→ CREATED
→ WAIT_ACCEPT
→ 商家确认预约
→ ACCEPTED / WAIT_SERVICE
→ 用户到店检测
→ IN_SERVICE
→ 商家确认方案和费用
→ 用户确认方案
→ 支付或线下处理，视业务规则
→ 商家维修
→ WAIT_CONFIRM
→ COMPLETED

```

说明：

- 复杂维修不建议在用户提交时强制线上最终价；
- 可以使用预约单、检测单、报价确认单扩展；
- MVP 阶段可以先使用订单备注和履约节点承接。

# **5. 状态流转表**

## **5.1 正常流转**


| 当前状态         | 触发动作    | 触发方   | 下一状态         |
| ------------ | ------- | ----- | ------------ |
| CREATED      | 生成待支付订单 | 系统    | WAIT_PAY     |
| WAIT_PAY     | 支付成功    | 支付回调  | WAIT_ACCEPT  |
| WAIT_PAY     | 用户取消    | 用户    | CANCELLED    |
| WAIT_PAY     | 超时未支付   | 系统    | CLOSED       |
| WAIT_ACCEPT  | 商家接单    | 商家    | ACCEPTED     |
| WAIT_ACCEPT  | 商家拒单    | 商家    | CANCELLED    |
| WAIT_ACCEPT  | 接单超时    | 系统/平台 | CANCELLED    |
| ACCEPTED     | 到达服务时间  | 系统    | WAIT_SERVICE |
| WAIT_SERVICE | 开始服务    | 商家    | IN_SERVICE   |
| ACCEPTED     | 开始服务    | 商家    | IN_SERVICE   |
| IN_SERVICE   | 提交完工    | 商家    | WAIT_CONFIRM |
| WAIT_CONFIRM | 用户确认完成  | 用户    | COMPLETED    |
| WAIT_CONFIRM | 自动确认超时  | 系统    | COMPLETED    |


---

## **5.2 取消流转**


| 当前状态         | 取消方 | 是否允许    | 结果                |
| ------------ | --- | ------- | ----------------- |
| WAIT_PAY     | 用户  | 允许      | CANCELLED         |
| WAIT_ACCEPT  | 用户  | 允许      | CANCELLED，已支付则走退款 |
| WAIT_ACCEPT  | 商家  | 允许      | CANCELLED，已支付则走退款 |
| ACCEPTED     | 用户  | 视规则     | 可能需商家确认或平台介入      |
| ACCEPTED     | 商家  | 视规则     | 可能进入售后/平台处理       |
| IN_SERVICE   | 用户  | 原则不直接允许 | 进入售后或纠纷           |
| WAIT_CONFIRM | 用户  | 原则不直接取消 | 进入售后或纠纷           |
| COMPLETED    | 任意方 | 不允许取消   | 可申请售后             |


---

## **5.3 退款流转**

退款不完全等同于订单取消。


| 场景      | order_status          | refund_status  | pay_status             |
| ------- | --------------------- | -------------- | ---------------------- |
| 待接单用户取消 | CANCELLED             | REFUNDING      | PAID                   |
| 商家拒单    | CANCELLED             | REFUNDING      | PAID                   |
| 退款成功    | CANCELLED 或 COMPLETED | REFUND_SUCCESS | REFUNDED/PART_REFUNDED |
| 部分退款    | 原订单状态不一定变             | REFUND_SUCCESS | PART_REFUNDED          |
| 售后退款    | COMPLETED             | REFUNDING      | PAID                   |
| 退款失败    | 原状态保持                 | REFUND_FAILED  | PAID                   |

# **6. 关键动作规则**

## **6.1 创建订单**

触发方：用户。

创建时需要校验：

1. 服务是否上架；
2. 门店是否可接单；
3. 城市是否支持；
4. 预约时间是否合法；
5. 用户是否被限制下单；
6. 服务价格是否有效；
7. 车辆信息是否满足最低要求。

创建结果：

```
标准需支付订单：WAIT_PAY
预约类订单：WAIT_ACCEPT 或 WAIT_PAY，视业务配置

```

---

## **6.2 支付成功**

触发方：微信支付回调。

支付成功后：

1. 更新支付单为 PAID；
2. 校验支付金额；
3. 更新订单 pay_status 为 PAID；
4. 更新订单主状态为 WAIT_ACCEPT；
5. 通知商家有新订单；
6. 通知用户支付成功；
7. 写入订单状态日志。

注意：

```
前端支付成功页面只能作为展示，不能作为最终支付依据。

```

---

## **6.3 商家接单**

触发方：商家。

接单前校验：

1. 订单是否属于该商家/门店；
2. 当前状态是否为 WAIT_ACCEPT；
3. 商家是否处于可接单状态；
4. 服务是否仍可履约；
5. 是否命中风控限制。

接单后：

```
WAIT_ACCEPT → ACCEPTED

```

系统动作：

1. 通知用户商家已接单；
2. 生成或激活维修相册任务；
3. 生成履约节点；
4. 写入状态日志。

---

## **6.4 商家拒单**

触发方：商家。

拒单后：

```
WAIT_ACCEPT → CANCELLED

```

如果已支付：

```
refund_status → REFUND_APPLIED / REFUNDING

```

系统动作：

1. 通知用户；
2. 自动或人工发起退款；
3. 记录拒单原因；
4. 影响商家接单率指标。

---

## **6.5 开始服务**

触发方：商家。

允许状态：

```
ACCEPTED / WAIT_SERVICE

```

流转：

```
ACCEPTED / WAIT_SERVICE → IN_SERVICE

```

系统动作：

1. 激活维修相册节点；
2. 用户端可查看维修进度；
3. 记录履约开始时间。

---

## **6.6 提交完工**

触发方：商家。

允许状态：

```
IN_SERVICE

```

提交前校验：

1. 必填相册节点是否完成；
2. 是否有未处理风险；
3. 是否有必填说明；
4. 是否存在未上传交付图片。

流转：

```
IN_SERVICE → WAIT_CONFIRM

```

系统动作：

1. 生成交付报告；
2. 通知用户确认完成；
3. 可生成案例草稿，但不公开；
4. 写入履约完成时间。

---

## **6.7 用户确认完成**

触发方：用户。

允许状态：

```
WAIT_CONFIRM

```

流转：

```
WAIT_CONFIRM → COMPLETED

```

系统动作：

1. 开放评价入口；
2. 触发评价奖励资格判断；
3. 生成案例草稿；
4. 进入脱敏和审核流程；
5. 进入待结算周期。

---

## **6.8 系统自动完成**

触发方：系统。

条件：

```
订单处于 WAIT_CONFIRM 超过 N 小时/天；
无售后、无退款、无投诉；
平台配置允许自动确认。

```

流转：

```
WAIT_CONFIRM → COMPLETED

```

注意：

- 自动确认时间应可配置；
- 自动确认前可发送提醒通知；
- 如用户发起售后，则暂停自动确认。

---

# **7. 异常状态规则**

## **7.1 超时未支付**

条件：

```
WAIT_PAY 超过支付有效期

```

流转：

```
WAIT_PAY → CLOSED

```

系统动作：

1. 关闭订单；
2. 释放预约资源；
3. 支付单标记关闭；
4. 用户可重新下单。

---

## **7.2 接单超时**

条件：

```
WAIT_ACCEPT 超过商家接单时限

```

处理方式：

1. 提醒商家；
2. 超过二次提醒仍未处理，可自动取消或进入平台处理；
3. 已支付订单需要退款；
4. 影响商家履约指标。

---

## **7.3 支付成功但订单未更新**

场景：

```
微信支付成功，订单仍处于 WAIT_PAY

```

处理方式：

1. 支付回调重试；
2. 主动查询微信支付结果；
3. 对账任务修正；
4. 进入支付异常列表；
5. 不允许用户重复支付。

---

## **7.4 订单完成后退款**

订单完成后不能回退到服务中，只能：

```
COMPLETED + refund_status = REFUND_APPLIED / REFUNDING / REFUND_SUCCESS

```

并触发：

1. 奖励冻结；
2. 结算暂缓或冲正；
3. 售后记录；
4. 平台人工审核。

# **8. 订单状态与相册关系**


| 订单状态         | 相册状态      |
| ------------ | --------- |
| WAIT_ACCEPT  | 相册任务可预生成  |
| ACCEPTED     | 相册任务激活    |
| IN_SERVICE   | 商家上传维修过程  |
| WAIT_CONFIRM | 生成交付报告    |
| COMPLETED    | 生成案例草稿    |
| CANCELLED    | 相册任务取消或归档 |
| CLOSED       | 相册任务关闭    |

# **9. 订单状态与评价奖励关系**


| 订单状态         | 评价      | 奖励      |
| ------------ | ------- | ------- |
| WAIT_PAY     | 不可评价    | 不发放     |
| WAIT_ACCEPT  | 不可评价    | 不发放     |
| IN_SERVICE   | 不可评价    | 不发放     |
| WAIT_CONFIRM | 可提示即将评价 | 不发放     |
| COMPLETED    | 可评价     | 可判断奖励资格 |
| CANCELLED    | 不可评价    | 不发放     |
| CLOSED       | 不可评价    | 不发放     |
| REFUNDING    | 暂停评价奖励  | 冻结      |
| REFUNDED     | 不发放或追回  | 驳回/冲正   |

# **11. 订单状态日志**

每次状态变化必须写入：

```
order_status_log

```

建议字段：


| 字段            | 说明                                     |
| ------------- | -------------------------------------- |
| id            | 日志 ID                                  |
| order_id      | 订单 ID                                  |
| from_status   | 原订单状态                                  |
| to_status     | 新订单状态                                  |
| action        | 触发动作                                   |
| operator_type | user/merchant/platform/system/callback |
| operator_id   | 操作人 ID                                 |
| reason        | 原因                                     |
| remark        | 备注                                     |
| created_at    | 创建时间                                   |

# **12. P0 验收标准**

必须满足：

1. 订单主状态、支付状态、退款状态分离；
2. 支持用户提交订单；
3. 支持待支付、支付成功、支付失败；
4. 支持商家接单、拒单；
5. 支持开始服务、提交完工、用户确认；
6. 支持超时未支付关闭；
7. 支持接单超时处理；
8. 支持取消订单后退款；
9. 支持完成后售后退款；
10. 所有订单状态变化必须写日志；
11. 支付成功必须以支付回调或主动查询为准；
12. 已取消、已关闭订单不可继续履约；
13. 已完成订单不可回退，只能进入售后；
14. 订单完成后才能评价和判断奖励；
15. 订单完成后才能生成正式案例草稿。
