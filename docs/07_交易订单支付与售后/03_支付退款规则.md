## **2. 支付退款设计原则**

支付退款规则遵循以下原则：

1. 支付结果以微信支付回调或主动查询为准；
2. 前端支付成功只作为展示，不作为最终入账依据；
3. 订单状态、支付状态、退款状态必须分离；
4. 支付、退款、奖励、结算必须可追溯；
5. 所有支付和退款操作必须支持幂等；
6. 退款不得超过实付金额；
7. 已产生奖励、分佣、结算的订单退款时，需要冻结或冲正；
8. 支付异常必须进入对账或人工处理；
9. 所有支付和退款操作必须记录日志。

# **6. 支付流程**

## **6.1 标准支付流程**

```
用户提交订单
→ 系统生成订单
→ 系统生成支付单
→ 调用微信支付创建预支付单
→ 用户确认支付
→ 微信支付回调
→ 平台校验回调
→ 更新支付单状态
→ 更新订单支付状态
→ 更新订单主状态
→ 通知用户和商家

```

---

## **6.2 创建支付单规则**

创建支付单前必须校验：

1. 订单是否存在；
2. 订单是否属于当前用户；
3. 订单状态是否允许支付；
4. 订单金额是否有效；
5. 订单是否已关闭；
6. 是否存在未完成支付单；
7. 用户是否命中风控限制；
8. 服务商品是否仍可购买；
9. 门店是否仍可接单。

---

## **6.3 支付成功处理**

支付成功回调后，系统应执行：

1. 校验微信支付签名；
2. 校验订单号；
3. 校验支付金额；
4. 校验支付渠道；
5. 校验是否重复回调；
6. 更新支付单状态为 `PAID`；
7. 更新订单 `pay_status = PAID`；
8. 将订单主状态推进到 `WAIT_ACCEPT`；
9. 写入支付日志；
10. 通知用户支付成功；
11. 通知商家有新订单；
12. 触发接单倒计时。

---

## **6.4 支付失败处理**

支付失败可能来自：

- 用户主动取消；
- 微信支付失败；
- 网络异常；
- 支付超时；
- 支付金额异常；
- 回调验签失败。

处理规则：


| 场景      | 处理方式             |
| ------- | ---------------- |
| 用户取消支付  | 订单保持 WAIT_PAY    |
| 支付失败    | 支付单标记 PAY_FAILED |
| 支付超时    | 支付单关闭，订单可重新发起支付  |
| 回调金额不一致 | 标记支付异常，进入人工处理    |
| 重复支付    | 原则不允许，异常进入人工处理   |

# **7. 订单取消与退款关系**

## **7.1 未支付订单取消**

```
order_status = WAIT_PAY
pay_status = UNPAID

```

用户取消后：

```
order_status = CANCELLED
pay_status = CLOSED
refund_status = NONE

```

无需退款。

---

## **7.2 已支付未接单取消**

适用于：

- 用户取消；
- 商家拒单；
- 商家超时未接单；
- 平台取消。

处理：

```
order_status = CANCELLED
pay_status = PAID
refund_status = REFUND_APPLIED / REFUNDING

```

退款成功后：

```
pay_status = REFUNDED
refund_status = REFUND_SUCCESS

```

---

## **7.3 已接单未服务取消**

该场景需要区分责任方。


| 取消方 | 是否自动退款 | 说明            |
| --- | ------ | ------------- |
| 用户  | 不一定    | 可能产生预约占用或检测费用 |
| 商家  | 通常应退款  | 商家无法履约        |
| 平台  | 视原因    | 可能全额或部分退款     |


建议 MVP 阶段：

- 优先走平台审核；
- 不建议完全自动退款；
- 后台保留人工处理入口。

---

## **7.4 服务中退款**

订单进入：

```
IN_SERVICE

```

后，用户原则上不能直接取消，只能进入：

```
售后 / 纠纷 / 平台介入

```

退款方式包括：

- 不退款；
- 部分退款；
- 全额退款；
- 补偿券，后期可选；
- 线下协商，需平台记录。

---

## **7.5 已完成订单退款**

已完成订单不能回退到服务中。

处理方式：

```
order_status = COMPLETED
refund_status = REFUND_APPLIED / REFUNDING / REFUND_SUCCESS

```

同时需要：

1. 冻结评价奖励；
2. 冻结商家结算；
3. 如已结算，则生成冲正记录；
4. 如已公开案例，可视情况隐藏或标记争议；
5. 记录售后处理结果。

---

# **8. 退款类型**


| 退款类型   | 说明             |
| ------ | -------------- |
| 全额退款   | 退还用户全部实付金额     |
| 部分退款   | 退还用户部分金额       |
| 自动退款   | 符合规则时系统自动发起    |
| 人工退款   | 平台审核后发起        |
| 售后退款   | 完成后因质量、纠纷产生    |
| 商家责任退款 | 商家拒单、无法履约、服务问题 |
| 用户责任退款 | 用户爽约、临时取消等     |


---

# **9. 退款申请规则**

## **9.1 用户可申请退款的条件**

用户可申请退款的状态：


| 订单状态         | 是否可申请        |
| ------------ | ------------ |
| WAIT_PAY     | 不需要退款，可取消    |
| WAIT_ACCEPT  | 可申请          |
| ACCEPTED     | 可申请，但可能需审核   |
| WAIT_SERVICE | 可申请，但可能需审核   |
| IN_SERVICE   | 不直接退款，进入售后   |
| WAIT_CONFIRM | 不直接退款，进入售后   |
| COMPLETED    | 售后退款         |
| CANCELLED    | 如已支付，可查看退款进度 |
| CLOSED       | 不可申请         |


---

## **9.2 商家可发起退款的条件**

商家可在以下场景发起退款建议：

1. 无法接单；
2. 无法按时履约；
3. 用户车辆情况与服务不匹配；
4. 配件缺货；
5. 到店后发现服务不适用；
6. 双方协商退款。

商家发起后，一般进入平台审核。

---

## **9.3 平台可发起退款的条件**

平台可在以下情况下发起退款：

1. 商家拒绝履约；
2. 商家超时未接单；
3. 投诉成立；
4. 服务质量问题；
5. 支付异常；
6. 订单争议处理结果要求退款；
7. 监管或合规要求。

---

# **10. 退款审核规则**

## **10.1 自动通过场景**

以下场景可自动退款：


| 场景           | 条件        |
| ------------ | --------- |
| 商家拒单         | 订单已支付且未服务 |
| 接单超时         | 超过平台配置时限  |
| 支付后系统异常无法派单  | 平台确认      |
| 服务商品下架导致无法履约 | 未接单前      |


---

## **10.2 需要人工审核场景**

以下场景需要人工审核：

1. 商家已接单；
2. 用户已到店；
3. 已开始服务；
4. 商家上传了维修过程；
5. 订单已完成；
6. 用户投诉质量问题；
7. 涉及部分退款；
8. 涉及奖励、分佣、结算冲正；
9. 命中风控规则。

---

# **11. 退款金额规则**

## **11.1 基础规则**

```
可退金额 <= 用户实付金额 - 已退款金额

```

不允许超过实付金额退款。

---

## **11.2 部分退款规则**

部分退款需要记录：

1. 原订单金额；
2. 用户实付金额；
3. 本次退款金额；
4. 累计退款金额；
5. 不退款金额；
6. 退款原因；
7. 审核人；
8. 商家责任比例，后期可选。

---

## **11.3 优惠与奖励的处理**

MVP 如果暂不支持优惠券，可先不处理复杂优惠分摊。

但评价奖励、分享收益必须处理：


| 场景     | 处理          |
| ------ | ----------- |
| 退款申请中  | 冻结奖励        |
| 全额退款成功 | 取消奖励        |
| 部分退款成功 | 视规则部分保留或冻结  |
| 退款失败   | 恢复奖励资格      |
| 已提现奖励  | 生成冲正或限制后续提现 |


---

# **12. 退款回调处理**

退款回调必须执行：

1. 校验微信签名；
2. 校验退款单号；
3. 校验订单号；
4. 校验退款金额；
5. 防重复回调；
6. 更新退款单状态；
7. 更新支付单状态；
8. 必要时更新订单状态；
9. 冻结或撤销奖励；
10. 调整结算状态；
11. 通知用户退款结果；
12. 写入退款日志。

---

# **13. 幂等规则**

以下操作必须幂等：


| 操作     | 幂等依据                         |
| ------ | ---------------------------- |
| 创建支付单  | order_id + pay_scene         |
| 微信支付回调 | transaction_id               |
| 关闭支付单  | payment_id                   |
| 创建退款单  | order_id + refund_request_id |
| 发起微信退款 | refund_id                    |
| 微信退款回调 | wx_refund_id                 |
| 自动退款任务 | order_id + refund_type       |
| 奖励冻结   | reward_id + order_id         |


---

# **14. 对账规则**

每日需要进行对账，重点校验：

1. 平台支付单金额与微信支付金额是否一致；
2. 平台退款单金额与微信退款金额是否一致；
3. 支付成功但订单未更新的异常；
4. 微信已退款但平台未更新的异常；
5. 重复回调但平台多次处理的异常；
6. 支付金额与订单金额不一致的异常；
7. 已退款订单是否还进入结算；
8. 已退款订单是否仍发放奖励。

---

# **15. 风控规则**

支付退款风控重点关注：

1. 高频下单后取消；
2. 高频退款；
3. 同一用户多订单退款；
4. 同一商家高退款率；
5. 用户与商家疑似串通；
6. 退款后仍领取奖励；
7. 退款后案例仍公开展示；
8. 异常金额退款；
9. 多次重复回调；
10. 绕过平台线下退款。

命中风控后：

```
自动退款暂停 → 进入人工审核 → 记录风险事件

```

# **17. P0 验收标准**

支付退款模块上线前必须满足：

1. 支持微信支付创建订单；
2. 支持支付成功回调；
3. 支持重复回调幂等；
4. 支持支付失败和用户取消；
5. 支持未支付订单关闭；
6. 支持商家拒单后退款；
7. 支持用户取消后退款；
8. 支持人工退款审核；
9. 支持全额退款；
10. 支持至少一种部分退款处理方式；
11. 支持退款回调；
12. 支持支付和退款日志；
13. 支持订单、支付、退款状态分离；
14. 支持退款后冻结奖励和结算；
15. 支持支付异常对账。
