# **2. 模块定位**

订单与支付模块是平台交易闭环的核心模块。

它连接：

1. 用户端下单/预约；
2. 商家端接单/履约；
3. 微信支付；
4. 维修相册；
5. 用户评价；
6. 奖励发放；
7. 公开案例生成；
8. 平台运营审核；
9. 佣金与结算，MVP 暂简化。

---

# **3. 订单类型**

MVP 阶段订单分为两类：

---

## **3.1 标准服务订单**

适用于价格明确的服务。

例如：

1. 电瓶更换；
2. 轮胎服务；
3. 小保养；
4. 标准检测；
5. 部分标准化刹车片/刹车盘服务。

特点：

- 用户可在线确认价格；
- 用户可微信支付；
- 商家接单后履约；
- 可退款；
- 可生成维修相册；
- 可评价；
- 可生成公开案例。

---

## **3.2 复杂维修预约单**

适用于无法线上最终报价的服务。

例如：

1. 事故车维修；
2. 钣喷修复；
3. 底盘维修；
4. 空调维修；
5. 疑难故障；
6. 新能源专项维修。

特点：

- 用户提交预约；
- 不在线支付最终维修款；
- 可上传问题描述和车辆照片；
- 商家接单后用户到店；
- 到店检测后线下报价；
- 商家可在平台记录最终维修项目和价格区间；
- 可生成维修相册；
- 可评价；
- 可生成公开案例。

---

# **4. 订单核心流程**

## **4.1 标准服务订单流程**

```
用户选择服务方案
→ 选择门店
→ 选择车辆
→ 选择预约时间
→ 确认订单
→ 微信支付
→ 支付成功
→ 订单进入待商家接单
→ 商家接单
→ 用户到店
→ 商家标记到店
→ 商家开始维修
→ 商家上传维修相册，可选
→ 商家提交完工
→ 用户确认完成
→ 用户评价
→ 奖励计算
→ 用户授权公开，可选
→ 生成公开案例，可选

```

---

## **4.2 复杂维修预约流程**

```
用户选择复杂维修服务
→ 查看价格说明和案例
→ 选择门店
→ 选择车辆
→ 填写故障/损伤描述
→ 上传车辆照片，可选
→ 选择预约时间
→ 提交预约单
→ 商家接单/拒单/建议改期
→ 用户到店检测
→ 商家记录维修方案和价格区间
→ 用户确认维修，线下确认
→ 商家开始维修
→ 商家上传维修相册，可选
→ 商家提交完工
→ 用户确认完成
→ 用户评价
→ 奖励计算
→ 用户授权公开，可选

```

# **6. 状态流转规则**

## **6.1 标准订单状态流转**

```
待支付
→ 支付中
→ 待商家接单
→ 已接单
→ 待到店
→ 已到店
→ 维修中
→ 待用户确认
→ 已完成

```

异常流转：

```
待支付 → 已关闭
待商家接单 → 已退款/已关闭
已接单 → 已取消/退款中
维修中 → 售后中，特殊情况
待用户确认 → 售后中

```

---

## **6.2 复杂预约状态流转**

```
待商家接单
→ 已接单
→ 待到店
→ 已到店
→ 检测/报价中
→ 用户确认维修
→ 维修中
→ 待用户确认
→ 已完成

```

异常流转：

```
待商家接单 → 已关闭
已接单 → 建议改期
建议改期 → 用户已改期/已取消
已到店 → 已关闭，用户不维修
检测/报价中 → 已关闭，用户不维修

```

# **7. 用户下单规则**

## **7.1 下单前置条件**

用户下单或预约前必须满足：

1. 已微信登录；
2. 已绑定手机号；
3. 已选择服务；
4. 已选择门店；
5. 已选择预约时间；
6. 已选择或填写车辆信息；
7. 已同意服务规则和隐私规则。

---

页面字段：


| 字段    | 说明         |

| 服务类型  | 标准服务       |
| 门店信息  | 名称、地址、电话   |
| 预约时间  | 用户选择       |
| 车辆信息  | 品牌、车系、车牌可选 |
| 服务价格  | 一口价/套餐价    |
| 服务包含  | 配件、工时、检测等  |

| 取消规则  | 退款规则       |
| 用户手机号 | 用于联系       |
| 备注    | 用户可填写      |


---

## **7.3 复杂预约确认页**

页面字段：


| 字段     | 说明         |
| ------ | ---------- |
| 服务类型   | 事故车/钣喷/底盘等 |
| 门店信息   | 名称、地址、电话   |
| 预约时间   | 用户选择       |
| 车辆信息   | 品牌、车系、年款等  |
| 问题描述   | 用户填写       |
| 上传图片   | 可选         |

必须展示确认勾选：

# **8. 微信支付 PRD**

## **8.1 支付适用范围**

MVP 支持微信支付的场景：

1. 一口价服务；
2. 套餐价服务；
3. 标准检测费，若配置；
4. 后续可扩展预约订金。

MVP 不支持微信支付的场景：

1. 事故车最终维修款；
2. 到店后临时增项全额支付；
3. 保险理赔款；
4. 线下商家自定义收款。

---

## **8.2 支付流程**

```
用户提交标准订单
→ 后端创建订单
→ 后端调用微信支付统一下单
→ 返回支付参数
→ 小程序拉起微信支付
→ 用户完成支付
→ 微信支付回调平台
→ 平台校验签名
→ 更新支付单状态
→ 更新订单状态为待商家接单
→ 创建/激活维修相册
→ 通知商家接单

```

---

## **8.3 支付单状态**


| 状态   | 说明        |
| ---- | --------- |
| 待支付  | 已创建支付单    |
| 支付中  | 用户拉起支付    |
| 支付成功 | 微信确认成功    |
| 支付失败 | 微信返回失败    |
| 已关闭  | 超时未支付     |
| 已退款  | 退款完成      |
| 部分退款 | 后续扩展      |
| 异常   | 回调或金额校验异常 |


---

## **8.4 支付超时规则**

建议：

- 未支付订单有效期：30 分钟；
- 超时后订单自动关闭；
- 支付单关闭；
- 用户可重新下单。

---

## **8.5 支付回调规则**

支付回调必须校验：

1. 微信签名；
2. 商户号；
3. 应用 ID；
4. 平台订单号；
5. 微信支付单号；
6. 支付金额；
7. 支付状态；
8. 回调幂等性。

如果回调重复：

- 不重复更新余额；
- 不重复触发商家通知；
- 记录回调日志。

---

## **8.6 主动查询支付状态**

若出现以下情况，需要主动查询微信支付状态：

1. 用户支付后页面未返回；
2. 平台未收到回调；
3. 回调解析失败；
4. 订单长时间处于支付中；
5. 用户反馈已扣款但订单未更新。

流程：

```
前端查询订单状态
→ 后端发现支付中
→ 调用微信支付查询接口
→ 确认成功则更新订单
→ 确认未支付则保持待支付或关闭

```

# **9. 退款 PRD**

## **9.1 退款适用场景**

标准服务订单支持退款。

退款场景：

1. 用户支付后商家拒单；
2. 用户支付后商家超时未接单；
3. 用户主动取消且符合规则；
4. 门店无法履约；
5. 平台判定需退款；
6. 售后协商退款。

---

## **9.2 自动退款场景**

以下场景建议自动退款：

1. 商家拒单；
2. 商家超时未接单；
3. 支付成功但订单创建异常，人工兜底；
4. 平台取消订单。

---

## **9.3 用户取消退款规则**


| 阶段      | 规则               |
| ------- | ---------------- |
| 待支付     | 直接关闭，无退款         |
| 已支付未接单  | 可取消并全额退款         |
| 已接单未到店  | 可申请取消，原则全额退款，可配置 |
| 已到店未维修  | 视是否发生检测/工时，需协商   |
| 维修中     | 不支持直接退款，走售后      |
| 待确认/已完成 | 走售后              |


---

## **9.4 退款流程**

```
用户/系统/平台发起退款
→ 校验订单状态
→ 生成退款单
→ 调用微信退款接口
→ 微信退款回调
→ 更新退款单状态
→ 更新订单状态
→ 通知用户
→ 通知商家

```

---

## **9.5 退款单状态**


| 状态   | 说明      |
| ---- | ------- |
| 待退款  | 已创建退款单  |
| 退款中  | 已调用微信退款 |
| 退款成功 | 微信确认成功  |
| 退款失败 | 微信返回失败  |
| 人工处理 | 需要运营处理  |
| 已关闭  | 退款申请关闭  |


---

# **10. 商家接单规则**

## **10.1 接单入口**

```
商家工作台
→ 待接单
→ 订单详情
→ 接单/拒单

```

---

## **10.2 接单前商家可查看**

标准订单：

1. 服务项目；
2. 服务方案；
3. 预约时间；
4. 用户车辆；
5. 用户备注；
6. 用户手机号脱敏展示，可拨打虚拟号后续；
7. 支付状态。

复杂预约：

1. 服务类型；
2. 问题描述；
3. 用户上传图片；
4. 预约时间；
5. 车辆信息；
6. 联系方式。

---

## **10.3 商家拒单**

商家拒单必须填写原因。

常见原因：

1. 预约时间无法接待；
2. 服务项目暂不可做；
3. 配件无库存；
4. 车型不适配；
5. 门店临时停业；
6. 其他原因。

标准支付订单拒单后：

```
订单关闭
→ 自动退款
→ 通知用户选择其他门店

```

复杂预约拒单后：

```
预约关闭
→ 通知用户选择其他门店或改期

```

---

## **10.4 商家超时未接单**

建议规则：

- 标准支付订单：商家 2 小时内未接单，提醒商家；
- 超过 4 小时仍未接单，平台可自动关闭并退款；
- 复杂预约：2 小时提醒，24 小时自动关闭或转平台跟进。

MVP 可先配置化，人工运营兜底。

---

# **11. 到店与维修履约**

## **11.1 标记到店**

用户实际到店后，商家点击：

```
标记到店

```

订单状态进入：

```
已到店

```

---

## **11.2 开始维修**

商家确认开始服务后点击：

```
开始维修

```

订单状态进入：

```
维修中

```

系统提示：

```
建议上传维修过程图片，帮助用户了解维修进度，也有助于生成门店案例。

```

---

## **11.3 复杂维修报价记录**

复杂预约到店检测后，商家应填写：

1. 检测结果；
2. 维修建议；
3. 用户是否确认维修；
4. 维修项目；
5. 价格区间；
6. 是否涉及保险；
7. 预计完工时间。

MVP 可不强制填写具体成交金额，但建议至少填写价格区间。

---

# **12. 完工确认**

## **12.1 商家提交完工**

商家提交完工前可填写：

1. 维修说明；
2. 更换配件；
3. 完工建议；
4. 质保说明；
5. 是否建议用户公开案例；
6. 维修相册图片，可选。

提交后：

```
订单进入待用户确认
→ 用户收到完工提醒
→ 用户可查看相册

```

---

## **12.2 用户确认完成**

用户可操作：

1. 确认完成；
2. 申请售后；
3. 联系商家；
4. 查看相册。

确认后：

```
订单进入已完成
→ 进入评价流程

```

---

## **12.3 自动完成**

建议规则：

- 商家提交完工后 72 小时；
- 用户未确认；
- 用户未发起售后；
- 系统自动完成。

自动完成后：

- 用户仍可评价；
- 用户仍可申请售后；
- 订单可进入奖励计算，但若售后中则冻结奖励。

---

# **13. 订单与相册关系**

## **13.1 相册创建**

订单创建时自动创建相册。

标准服务订单：

- 下单时创建空相册；
- 支付成功后正式生效；
- 订单关闭则相册不可公开。

复杂预约单：

- 预约提交后创建空相册；
- 商家接单后可上传；
- 若用户未维修，可不生成公开案例。

---

## **13.2 相册权限**


| 角色   | 权限             |
| ---- | -------------- |
| 用户   | 查看自己订单相册       |
| 商家   | 上传和查看本门店订单相册   |
| 平台   | 查看原图、脱敏图、审核状态  |
| 其他用户 | 仅可查看授权公开后的脱敏案例 |


---

# **14. 订单与评价奖励关系**

订单完成后触发评价。

```
订单完成
→ 用户评价
→ 系统计算奖励
→ 风控检测
→ 平台审核
→ 发放/冻结/驳回

```

以下情况奖励冻结：

1. 订单售后中；
2. 订单退款中；
3. 用户评价异常；
4. 商家疑似刷单；
5. 高金额奖励需人工审核。

---

# **15. 订单与公开案例关系**

公开案例生成前置条件：

1. 订单已完成；
2. 相册至少有 1 张有效图片；
3. 用户明确授权公开；
4. 图片完成脱敏；
5. 平台审核通过。

未满足条件时：

- 订单仍正常完成；
- 相册仅作为用户维修档案；
- 不生成公开案例页。

# **19. 异常与边界**

## **19.1 用户已支付但订单未更新**

处理：

1. 查询微信支付状态；
2. 若成功，补偿更新订单；
3. 若失败，保持待支付；
4. 记录异常日志。

---

## **19.2 商家接单后无法履约**

处理：

1. 商家联系用户；
2. 用户同意取消，则退款；
3. 用户不同意，平台介入；
4. 记录商家履约异常。

---

## **19.3 用户到店后增加项目**

MVP 处理：

- 线下沟通确认；
- 商家可在订单中记录维修说明和价格区间；
- 平台不处理增项支付；
- 后续扩展补差价支付。

---

## **19.4 复杂预约用户未到店**

处理：

- 商家可标记未到店；
- 订单关闭；
- 记录用户爽约；
- 不生成评价奖励。

---

## **19.5 订单完成后退款**

处理：

- 进入售后流程；
- 奖励冻结；
- 若退款成功，公开案例可隐藏或标注争议；
- 高风险订单不进入公开案例库。

---

# **20. P0 验收标准**

必须满足：

1. 用户可创建标准服务订单；
2. 用户可创建复杂维修预约单；
3. 标准订单可调用微信支付；
4. 支付成功后订单状态正确更新；
5. 支持支付回调幂等；
6. 支持主动查询支付状态；
7. 商家可接单/拒单；
8. 商家拒单可触发退款；
9. 用户可取消符合条件的订单；
10. 商家可标记到店、开始维修、提交完工；
11. 用户可确认完成；
12. 系统可自动完成订单；
13. 订单创建后自动创建维修相册；
14. 订单完成后可评价；
15. 评价后可进入奖励流程；
16. 用户授权后可进入公开案例流程；
17. 平台可查看订单、支付、退款和状态日志。
