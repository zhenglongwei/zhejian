# **2. 售后模块定位**

售后与纠纷模块用于处理订单履约过程中或完成后的异常问题。

它连接：

1. 用户投诉与售后申请；
2. 商家响应与协商；
3. 平台运营介入；
4. 退款审核与执行；
5. 订单状态；
6. 支付退款状态；
7. 评价奖励冻结；
8. 结算冻结或冲正；
9. 公开案例隐藏或争议标记；
10. 商家履约质量评价。

---

# **3. 售后适用范围**

售后适用于以下订单：

1. 标准服务订单；
2. 复杂维修预约单；
3. 已支付但未服务订单；
4. 服务中订单；
5. 待用户确认订单；
6. 已完成订单；
7. 已评价订单；
8. 已生成公开案例的订单。

---

# **4. 售后类型**

MVP 阶段售后类型包括：


| 售后类型    | 说明                   | 是否可能退款 |
| ------- | -------------------- | ------ |
| 未服务退款   | 已支付但未接单、未到店或未开始服务    | 是      |
| 服务不满意   | 用户认为服务态度、交付效果不符合预期   | 视情况    |
| 维修质量问题  | 维修后问题未解决、复发或造成新问题    | 是      |
| 配件争议    | 配件品牌、型号、新旧、真伪、质保产生争议 | 是      |
| 价格争议    | 到店报价、增项费用、最终金额存在争议   | 是      |
| 相册与实际不符 | 相册记录与实际维修内容不一致       | 视情况    |
| 质保申请    | 服务完成后在质保期内申请返修或处理    | 视情况    |

# **5. 售后状态设计**

## **5.1 aftersale_status 售后状态**


| 状态      | 状态码建议                  | 说明             |
| ------- | ---------------------- | -------------- |
| 无售后     | NONE                   | 当前订单无售后        |
| 已申请售后   | AFTERSALE_APPLIED      | 用户已提交售后申请      |
| 商家处理中   | MERCHANT_PROCESSING    | 商家正在处理         |
| 待用户补充材料 | WAIT_USER_SUPPLEMENT   | 需要用户补充图片、说明等材料 |
| 待商家响应   | WAIT_MERCHANT_RESPONSE | 等待商家回复方案       |
| 平台介入中   | PLATFORM_INTERVENING   | 用户或商家申请平台介入    |
| 协商中     | NEGOTIATING            | 用户、商家、平台协商处理方案 |
| 待退款     | WAIT_REFUND            | 售后结论为退款，等待退款处理 |
| 退款中     | REFUNDING              | 已发起退款          |
| 售后已解决   | AFTERSALE_RESOLVED     | 售后处理完成         |
| 售后已关闭   | AFTERSALE_CLOSED       | 用户撤销、超时关闭或平台关闭 |
| 售后驳回    | AFTERSALE_REJECTED     | 平台判定不支持售后诉求    |


---

## **5.2 售后结果类型**


| 处理结果   | 说明                |
| ------ | ----------------- |
| 无责关闭   | 未发现商家或平台责任，售后关闭   |
| 商家解释说明 | 商家给出说明，用户接受       |
| 重新服务   | 商家重新检测、返修或补充服务    |
| 部分退款   | 退还用户部分金额          |
| 全额退款   | 退还用户全部实付金额        |
| 平台补偿   | 平台给予补偿券或其他补偿，后续扩展 |
| 商家赔付   | 商家承担赔付或退款责任       |
| 质保处理   | 按质保规则进行返修或处理      |
| 驳回申请   | 不符合售后条件，驳回        |
| 转人工纠纷  | 售后升级为纠纷处理         |

# **6. 售后发起规则**

## **6.1 用户可发起售后的订单状态**


| 订单状态         | 是否可发起售后 | 说明                |
| ------------ | ------- | ----------------- |
| WAIT_PAY     | 不可      | 未支付，无需售后，可取消      |
| WAIT_ACCEPT  | 可申请退款   | 已支付未接单，可取消退款      |
| ACCEPTED     | 可申请     | 已接单未服务，可能需审核      |
| WAIT_SERVICE | 可申请     | 待到店或待服务阶段         |
| IN_SERVICE   | 可申请售后   | 不支持直接取消，进入售后      |
| WAIT_CONFIRM | 可申请售后   | 用户未确认完成，可申请       |
| COMPLETED    | 可申请售后   | 已完成后进入售后流程        |
| CANCELLED    | 可查看退款进度 | 一般不可重复申请          |
| CLOSED       | 不可      | 已关闭订单不可售后，特殊情况找平台 |

用户端入口：

```
我的

→ 申请售后
```

可展示入口的状态：

1. 已支付未接单；
2. 商家已接单；
3. 服务中；
4. 待确认；
5. 已完成；
6. 已评价但仍在售后期内。

# **7. 售后时效规则**

## **7.1 申请时效**


| 售后类型    | 建议申请时效               |
| ------- | -------------------- |
| 未服务退款   | 服务开始前均可申请            |
| 服务不满意   | 订单完成后 7 天内           |
| 维修质量问题  | 订单完成后 15 天内，质保服务按质保期 |
| 配件争议    | 订单完成后 15 天内          |
| 价格争议    | 订单完成后 7 天内           |
| 相册与实际不符 | 订单完成后 15 天内          |
| 质保申请    | 质保期内                 |


说明：

- MVP 可先统一配置为订单完成后 7 天内可申请普通售后；
- 涉及质保的服务可按服务方案配置质保期；
- 超过时效后，用户仍可联系平台，但不保证受理。

---

## **7.2 商家响应时效**


| 场景   | 响应时效     |
| ---- | -------- |
| 普通售后 | 24 小时内响应 |
| 退款申请 | 24 小时内响应 |
| 质量问题 | 24 小时内响应 |
| 质保申请 | 24 小时内响应 |
| 紧急纠纷 | 平台可立即介入  |


商家超时未响应：

```
售后自动升级为平台介入

```

---

## **7.3 用户补充材料时效**

当平台或商家要求用户补充材料时：

- 用户需在 48 小时内补充；
- 超时未补充，平台可关闭售后；
- 用户可重新申请，若仍在售后期内。

# **8. 售后流程**

## **8.1 普通售后流程**

```
用户提交售后申请
→ 系统创建售后单
→ 售后状态：已申请售后
→ 通知商家
→ 商家响应
→ 商家给出处理方案
→ 用户确认方案
→ 执行处理方案
→ 售后完成

```

---

## **8.2 平台介入流程**

```
用户提交售后
→ 商家拒绝或双方协商失败
→ 用户申请平台介入
→ 平台查看订单、相册、评价、聊天/备注、支付退款记录
→ 平台给出处理结论
→ 执行退款/返修/驳回/关闭
→ 售后完成

```

---

## **8.3 退款型售后流程**

```
用户申请退款
→ 创建售后单
→ 创建或关联退款申请
→ 判断是否自动通过
→ 需要审核则进入平台审核
→ 审核通过
→ 创建退款单
→ 调用微信退款
→ 微信退款回调
→ 更新退款状态
→ 更新售后状态
→ 通知用户和商家

```

---

## **8.4 质保申请流程**

```
用户提交质保申请
→ 选择对应订单
→ 填写问题描述和上传凭证
→ 商家确认是否在质保范围内
→ 用户到店检测
→ 商家返修或说明原因
→ 用户确认处理完成
→ 售后关闭

```

# **9. 售后处理规则**

## **9.1 未服务退款**

适用场景：

1. 已支付但商家未接单；
2. 商家拒单；
3. 商家超时未接单；
4. 商家接单后无法履约；
5. 用户已预约但未到店且符合取消规则。

处理规则：


| 阶段               | 处理方式          |
| ---------------- | ------------- |
| WAIT_ACCEPT      | 可自动全额退款       |
| ACCEPTED 未到店     | 原则可申请退款，可能需审核 |
| WAIT_SERVICE 未服务 | 根据责任方判断是否全额退款 |
| 商家原因无法履约         | 通常全额退款        |
| 用户临时取消           | 可视规则审核        |


---

## **9.2 服务不满意**

适用场景：

1. 服务态度差；
2. 等待时间过长；
3. 服务体验不符合预期；
4. 完工说明不清晰；
5. 用户认为交付结果不满意。

处理方式：

1. 商家解释说明；
2. 商家补充服务；
3. 平台记录服务质量；
4. 情节严重可部分退款；
5. 不一定直接退款。

---

## **9.3 维修质量问题**

适用场景：

1. 原问题未解决；
2. 维修后短期复发；
3. 维修导致新问题；
4. 配件安装异常；
5. 车辆交付后出现与维修相关的问题。

处理方式：

1. 商家重新检测；
2. 商家返修；
3. 平台介入判断责任；
4. 可部分退款或全额退款；
5. 涉及安全问题需优先处理；
6. 影响商家履约质量评分。

---

## **9.4 配件争议**

适用场景：

1. 用户质疑配件品牌；
2. 配件型号与订单不一致；
3. 配件新旧存在争议；
4. 配件质保说明不清；
5. 配件价格明显争议。

处理方式：

1. 商家提供配件照片、包装、型号或采购说明；
2. 平台查看维修相册；
3. 如商家无法证明，平台可判定退款或返修；
4. 情节严重可标记商家异常。

---

## **9.5 价格争议**

适用场景：

1. 复杂维修到店报价与案例价差异较大；
2. 用户认为增项未充分告知；
3. 线下费用与平台展示说明不一致；
4. 商家未记录价格区间；
5. 用户确认维修前后价格不一致。

处理规则：

1. 标准订单以平台订单金额为准；
2. 复杂维修最终价格需到店检测后确认；
3. 平台展示案例价仅供参考；
4. 商家应记录维修方案和价格区间；
5. 未经用户确认的增项不应强制收费；
6. MVP 阶段线下付款争议由平台记录并介入协调。

---

## **9.6 相册与实际不符**

适用场景：

1. 相册图片非用户车辆；
2. 上传图片与实际维修项目不符；
3. 完工照片缺失关键内容；
4. 图片存在明显造假；
5. 用户不同意公开但被公开。

处理方式：

1. 平台核查相册原图、上传时间、订单信息；
2. 要求商家说明；
3. 必要时下架公开案例；
4. 冻结该订单评价奖励；
5. 记录商家违规行为；
6. 严重情况限制商家案例发布能力。

---

## **9.7 质保申请**

适用场景：

1. 服务方案承诺质保；
2. 配件存在质保；
3. 商家完工说明中承诺质保；
4. 平台服务规则包含质保说明。

处理规则：

1. 用户需在质保期内申请；
2. 质保范围以服务方案、商家说明、订单记录为准；
3. 用户人为损坏、二次事故、非维修相关问题可不纳入质保；
4. 商家需提供检测结论；
5. 平台可介入判断是否属于质保范围。

# **10. 平台介入规则**

## **10.1 平台介入触发条件**

以下情况可触发平台介入：

1. 商家超时未响应；
2. 商家拒绝处理；
3. 用户不认可商家方案；
4. 双方协商失败；
5. 涉及退款争议；
6. 涉及质量或安全问题；
7. 涉及配件真伪或型号争议；
8. 涉及相册造假；
9. 命中风控规则；
10. 涉及高金额订单。

---

## **10.2 平台可查看资料**

平台运营可查看：

1. 订单详情；
2. 支付状态；
3. 退款状态；
4. 订单状态日志；
5. 商家接单与履约记录；
6. 维修相册原图和脱敏图；
7. 完工说明；
8. 用户评价；
9. 售后申请材料；
10. 商家处理说明；
11. 历史售后记录；
12. 风控标记。

---

## **10.3 平台处理动作**

MVP 支持平台执行：

1. 查看售后详情；
2. 要求用户补充材料；
3. 要求商家补充说明；
4. 驳回售后申请；
5. 关闭售后；
6. 判定重新服务；
7. 判定部分退款；
8. 判定全额退款；
9. 发起退款；
10. 标记商家责任；
11. 标记用户异常；
12. 隐藏或争议标记公开案例；
13. 冻结奖励；
14. 冻结结算；
15. 写入处理备注。

# **11. 售后与订单状态关系**

售后不一定改变订单主状态。


| 订单状态         | 售后处理方式          |
| ------------ | --------------- |
| WAIT_ACCEPT  | 可取消订单并退款        |
| ACCEPTED     | 可申请取消或平台介入      |
| WAIT_SERVICE | 可申请售后或取消        |
| IN_SERVICE   | 不直接取消，进入售后/纠纷   |
| WAIT_CONFIRM | 可申请售后，暂停自动完成    |
| COMPLETED    | 订单不回退，叠加售后状态    |
| CANCELLED    | 可查看退款进度，一般不再售后  |
| CLOSED       | 原则不可售后，特殊情况平台处理 |


---

## **11.1 已完成订单售后**

已完成订单不可回退到服务中。

处理方式：

```
order_status = COMPLETED
aftersale_status = AFTERSALE_APPLIED / PLATFORM_INTERVENING / AFTERSALE_RESOLVED
refund_status = REFUND_APPLIED / REFUNDING / REFUND_SUCCESS

```

---

## **11.2 待确认订单售后**

当订单处于：

```
WAIT_CONFIRM

```

用户发起售后后：

1. 暂停自动完成；
2. 冻结评价入口或提示售后处理中；
3. 暂停奖励计算；
4. 平台或商家处理完成后再恢复。

---

# **12. 售后与退款关系**

售后可能触发退款，但售后不等于退款。

## **12.1 可自动退款场景**


| 场景          | 是否自动 |
| ----------- | ---- |
| 商家拒单        | 是    |
| 商家超时未接单     | 是    |
| 支付后订单异常无法派单 | 是    |
| 未服务且平台取消    | 是    |


---

## **12.2 需人工审核退款场景**

以下场景需人工审核：

1. 商家已接单；
2. 用户已到店；
3. 服务已开始；
4. 已提交完工；
5. 订单已完成；
6. 涉及部分退款；
7. 涉及质量争议；
8. 涉及配件争议；
9. 涉及线下增项；
10. 涉及奖励、结算冲正。

---

## **12.3 退款金额规则**

基础规则：

```
可退金额 <= 用户实付金额 - 已退款金额

```

处理方式：

1. 全额退款；
2. 部分退款；
3. 不退款；
4. 返修不退款；
5. 平台补偿，后续扩展。

---

# **13. 售后与评价奖励关系**

当订单进入售后后：


| 场景       | 评价          | 奖励      |
| -------- | ----------- | ------- |
| 售后申请中    | 可限制或提示      | 冻结      |
| 售后处理中    | 可限制         | 冻结      |
| 售后驳回     | 可恢复         | 恢复判断    |
| 售后解决且无退款 | 可恢复         | 恢复判断    |
| 部分退款成功   | 视规则处理       | 部分保留或冻结 |
| 全额退款成功   | 不发放         | 取消或追回   |
| 已提现奖励    | 生成冲正或限制后续提现 |         |


---

# **14. 售后与公开案例关系**

当订单存在售后或纠纷时，公开案例需谨慎处理。


| 场景        | 处理        |
| --------- | --------- |
| 售后申请中     | 暂停公开审核    |
| 已公开案例发生售后 | 可临时隐藏     |
| 质量争议成立    | 隐藏或标记争议   |
| 相册造假成立    | 下架案例并记录违规 |
| 售后驳回      | 可恢复公开流程   |
| 全额退款      | 不进入公开案例库  |


公开案例生成前仍需满足：

1. 订单已完成；
2. 用户授权公开；
3. 图片脱敏完成；
4. 平台审核通过；
5. 无高风险售后或纠纷。

---

# **15. 售后与商家风控关系**

以下行为会影响商家风控和履约指标：

1. 高退款率；
2. 高售后率；
3. 多次质量投诉；
4. 多次配件争议；
5. 多次价格争议；
6. 相册造假；
7. 拒绝处理售后；
8. 超时响应售后；
9. 平台判定商家责任；
10. 用户投诉成立。

平台可采取：

1. 运营提醒；
2. 降低排序权重；
3. 限制接单；
4. 暂停服务方案；
5. 暂停案例发布；
6. 冻结结算；
7. 清退商家，后续扩展。

---

# **16. 售后与用户风控关系**

以下行为会影响用户风控：

1. 高频申请退款；
2. 高频售后投诉；
3. 明显恶意差评；
4. 虚假问题描述；
5. 使用非本订单车辆照片；
6. 退款后继续要求服务；
7. 与商家疑似串通套取奖励；
8. 售后材料造假。

命中风控后：

```
售后自动通过暂停
→ 转平台人工审核
→ 记录风险事件

```

# **18. 售后通知规则**

## **18.1 用户通知**

触发场景：

1. 售后申请提交成功；
2. 商家已响应；
3. 商家要求补充材料；
4. 平台介入；
5. 平台处理结论；
6. 退款审核通过；
7. 退款成功；
8. 售后关闭；
9. 售后驳回；
10. 质保处理完成。

---

## **18.2 商家通知**

触发场景：

1. 用户提交售后；
2. 用户申请退款；
3. 用户补充材料；
4. 平台介入；
5. 平台要求商家说明；
6. 平台判定商家责任；
7. 退款成功；
8. 售后关闭；
9. 售后评价或投诉成立。

---

## **18.3 平台通知**

触发场景：

1. 用户申请平台介入；
2. 商家超时未响应；
3. 高金额退款；
4. 质量争议；
5. 配件争议；
6. 价格争议；
7. 相册造假；
8. 命中风控；
9. 退款失败；
10. 售后长期未处理。

# **19. 异常与边界情况**

## **19.1 用户重复提交售后**

处理：

1. 同一订单同一问题未关闭前，不允许重复提交；
2. 可允许用户补充材料；
3. 已关闭后仍在售后期内，可重新申请；
4. 高频重复申请进入风控。

---

## **19.2 商家不响应**

处理：

```
商家超时未响应
→ 自动提醒
→ 仍未响应
→ 平台介入
→ 影响商家售后响应指标

```

---

## **19.3 用户不补充材料**

处理：

```
平台/商家要求补充材料
→ 用户超时未补充
→ 售后可关闭

```

---

## **19.4 已结算后发生售后退款**

处理：

1. 生成结算冲正记录；
2. 冻结商家后续结算；
3. 记录退款责任；
4. 如涉及奖励，生成奖励冲正或追回。

---

## **19.5 已公开案例发生纠纷**

处理：

1. 暂时隐藏案例；
2. 平台核查相册和订单；
3. 纠纷成立则下架；
4. 纠纷不成立可恢复；
5. 相册造假则记录商家违规。

---

## **19.6 复杂维修线下付款争议**

MVP 处理：

1. 平台记录争议；
2. 商家提交维修方案、报价记录和用户确认情况；
3. 用户提交付款凭证或沟通记录；
4. 平台协调处理；
5. 暂不处理线下款原路退款；
6. 后续可扩展平台内补差价支付和退款。

# **21. P0 验收标准**

售后与纠纷模块上线前必须满足：

1. 用户可在符合条件的订单中发起售后；
2. 支持至少 7 类售后类型；
3. 售后单可记录问题描述和图片凭证；
4. 商家可查看并响应售后；
5. 平台可查看售后详情；
6. 平台可介入处理售后；
7. 支持售后状态流转；
8. 支持售后日志；
9. 支持售后触发退款申请；
10. 支持人工审核退款；
11. 支持售后中冻结评价奖励；
12. 支持售后中冻结或影响结算；
13. 支持已完成订单售后，不回退订单主状态；
14. 支持商家超时未响应升级平台介入；
15. 支持已公开案例在售后争议中隐藏或标记；
16. 支持售后通知用户、商家和平台；
17. 支持基础风控标记。
