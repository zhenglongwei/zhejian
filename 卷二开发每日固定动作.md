# 卷二开发 · 每日固定动作

> **这是什么**：卷二阶段的**每日开发 checklist**，保证商家工作台（入驻、服务方案、线索、门店预览）与卷一用户端、卷三运营台**契约一致**。  
> **不是**：要一次性读完的 PRD；**不用背**，按当天任务查对应小节即可。  
> **进度看哪里**：`docs/00_开发计划.md` **§7.2**（任务 ID 如 M-SVC-12、B-SVC-01）

## 怎么用（3 分钟读懂）

## 1. 开做前（每天第一件事）

1. **Read 开发计划**：`docs/00_开发计划.md` **§7.2** + **§7.2.10 进度汇总**；优先 P0 阻塞项（见 §7.2.10 文末列表）。
2. **Read 对应 PRD**（按任务选读）：

   | 任务域 | 必读 PRD |
   | --- | --- |
   | M-WB-* | `03_商家端/00_商家工作台PRD V2.0.md` |
   | M-ONB-* / B-MERCH-03 | `01_商家入驻PRD.md` + 运营 `02_商家审核管理.md` |
   | M-SVC-* / B-SVC-* | `02_服务商品上架.md` + 用户端 `03_服务商品详情页.md` |
   | OPS-SVC-* | `06_平台运营后台/01_运营后台PRD V2.0.md` §6 + `02_服务商品上架.md` §9 |
   | M-LEAD-* | `03_咨询线索管理.md` + `11_数据结构与状态机/05_咨询线索状态机.md` |
   | M-STORE-* | 工作台 §8 + 入驻 §13 + 用户端 `04_门店详情页.md` |
   | M-DASH-* / B-STATS-* | `06_商家数据看板.md`（P1） |

3. **Read 现有实现**（改代码前必做）：

   | 域 | 关键文件 |
   | --- | --- |
   | 工作台 / 入驻 | `packageMerchant/pages/workbench/`、`onboarding/` · `services/merchant.js` · `backend/.../merchant-onboarding.service.js` |
   | 服务方案 mock | `services/service.js`（`loadMerchantServices` / `saveServicePlan`）· `constants/service.js` |
   | 线索（已接 API） | `services/merchant-lead.js` · `backend/src/routes/merchant-leads.js` |
   | 运营商家审核 | `admin-web/` 商家审核页 · `admin-merchant.service.js` |

4. **非 trivial 变更**：先输出「需求理解 → 任务 ID → 涉及文件 → 方案 → 是否改表/改契约 → 卷一/卷三是否联动」，再写代码。

5. **指定 skill**（见 [§7](#7-skill--rules-选用速查)）：
   - 商家页 / 商家 API → **`merchant-workbench-scaffold`**
   - 接 prod、替换 mock → **`api-integration`**
   - 运营 Web 服务审核 → **`ops-admin-scaffold`**
   - 新页面通用壳 → `page-scaffold`（业务规则仍以 merchant-workbench 为准）

## 2. 开发中（按任务类型）

### 2.1 工作台壳（M-WB-*）

- [ ] 状态路由覆盖：`none` / `pending` / `rejected` / `need_modify` / `approved`
- [ ] 待办数字来自 API（线索 stats、相册 stats），非写死
- [ ] 快捷操作不含 V1 订单；**M-WB-10** 清理 `packageMerchant/pages/order/*` 注册
- [ ] 审核通过后 `refreshMerchantSession` 刷新 JWT

### 2.2 商家入驻扩展（M-ONB-* / B-MERCH-03）

- [ ] 字段与 PRD §6–§8 一致：执照、信用代码、地图坐标、门头/工位图、维修资质
- [ ] 图片上传持久化（`media-upload`）；拒绝 temp 路径
- [ ] 擅长服务与资质 / L 级提示（暂不可提交时给明确原因）
- [ ] 营销文案 V2.0 口径（**M-ONB-01**：禁「每一单维修」「免佣」）
- [ ] 运营台商家审核详情能展示新增字段（与 **ops-admin-scaffold** 同步）

### 2.3 服务方案（M-SVC-* / B-SVC-*）

**开做必用 `merchant-workbench-scaffold` + `api-integration`。**

- [ ] **`merchant_service_plan` 落库**；用户端 **B-CONTENT-02 切 DB**（非 SEED 常量）
- [ ] 状态机：草稿 → 待审核 → 通过/驳回 → 上架/下架（**禁止** submit 即 `published`）
- [ ] 价格模式：一口价 / 区间 / 事故车；**事故车禁止一口价**（前后端双拦）
- [ ] 区间价 `max >= min`；参考价文案含「到店检测确认」
- [ ] 商家详情 **M-SVC-05 编辑** 与列表状态 Tag 正确
- [ ] mock 移除后 `ENV.mode=prod` 可完整走通

### 2.4 咨询线索（M-LEAD-*）

- [ ] Tab 与用户端状态机一致（待处理 / 已联系 / 已关闭）
- [ ] 详情自动 mark viewed；拨号 + 标记已联系 + 关闭（含原因）
- [ ] 页脚平台说明（`consultMerchant`）；**无** 订单/支付/履约按钮
- [ ] **M-LEAD-07**：生产环境冒烟（列表 → 详情 → 联系 → 关闭 → 用户端同步）

### 2.5 运营服务审核（OPS-SVC-*）

**开页必用 `ops-admin-scaffold`。**

- [ ] `admin-web` 服务审核列表 + 详情 + 通过/驳回/要求修改/限制预约/下架
- [ ] `service_review_log` 或等价留痕
- [ ] 资质与 L 级、事故车一口价等规则可对照操作
- [ ] 仅 admin 角色；loading / empty / error / 防重复提交

### 2.6 门店预览 / 员工 / 看板（P0/P1）

- [ ] **M-STORE-01**：工作台入口 → 预览用户端门店页
- [ ] **M-STAFF-***（P1）：`merchant_staff` 邀请 API + 权限隔离
- [ ] **M-DASH-***（P1）：依赖 stats API；无订单/评价指标

### 2.7 卷二边界（避免串卷）

- [ ] **不改** 相册六阶段 / 脱敏 / 冷启动，除非 `00_开发计划` 卷一 A-* 明确项
- [ ] **不建设** 独立历史案例（`05_历史案例上传.md` 已归档）
- [ ] **不恢复** V1 订单模块
- [ ] 用户端服务/门店展示随 B-SVC-05 变更时，回归 **U-SVC-*** / **U-STORE-***

## 3. 写完一块（当天或 PR 前）

按改动类型触发检查：

| 改动类型 | 必跑 skill |
| --- | --- |
| 商家工作台新页 / 表单 / 商家 API | **`merchant-workbench-scaffold`** 自查 → **`api-integration`** |
| 替换 mock / prod 联调 | **`api-integration`** |
| 服务方案价格 / 详情展示 | **`price-compliance-check`** |
| 运营 Web 服务审核 | **`ops-admin-scaffold`** → `design-system-check`（Tag/文案） |
| 入驻扩展 + 运营详情 | **`merchant-workbench-scaffold`** + **`ops-admin-scaffold`** |
| 新 UI 组件组合 | `component-usage-review` + `design-system-check` |
| 误触相册/公开链 | `privacy-desensitization-check` |
| 图片 URL 异常 | `miniprogram-image-debug` |

**卷二额外自查**：

1. 未入驻能否绕过 `ensureMerchant()` 访问业务页？
2. 服务方案待审核态是否在用户端不可见？
3. 线索关闭后用户端是否同步「已关闭」？
4. 是否仍有 localStorage / SEED 静默 fallback？
5. 是否误加订单/评价/奖励入口或文案？

## 4. 合并前

1. **`merchant-workbench-scaffold` 自查**（若改 packageMerchant / 商家 API）
2. **`ops-admin-scaffold` 自查**（若改 admin-web 服务审核）
3. `component-usage-review` + `design-system-check`（有新 UI 时）
4. 涉及价格：`price-compliance-check`
5. 更新 **`docs/00_开发计划.md` §7.2**：`[ ]` → `[x]`/`[~]`，§8 版本记录一行
6. PR 描述：任务 ID、验收口径、迁移/SQL、是否依赖 env

## 5. 发体验版 / 上预发前

1. **`release-checklist`**（含商家入驻 → 工作台 → 服务方案 → 线索全链路）
2. **`price-compliance-check`**（服务 Tab / 详情 / 商家创建页）
3. 确认 `MERCHANT_AUTO_APPROVE=false` 与生产 env 一致
4. 商家 JWT + 用户 JWT 同小程序双角色仍正常
5. 卷三回归：案例审核、商家审核、脱敏 smoke 仍通过
